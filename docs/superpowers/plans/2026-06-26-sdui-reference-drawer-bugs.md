# Спека: баги SDUI справочных полей и боковой панели (drawer)

- **Продукт:** fin-web (фронтенд)
- **Дата:** 2026-06-26
- **Где воспроизводится:** документ «Заявка на регистрацию ГП-сделки», экран SDUI-формы и боковые панели «Показать всё» / «Добавить».
- **Примечание:** баги находятся в общих SDUI-компонентах, поэтому затрагивают все документы и справочные поля, не только ГП-сделку.

---

## Баг 1. Дропдаун справочного поля показывает записи прошлого фильтра

**Описание:**
При смене значения, от которого зависит справочное поле, выпадающий список не перезагружается и показывает устаревшие варианты.

**Шаги воспроизведения:**
1. Открыть «Заявка на регистрацию ГП-сделки».
2. Выбрать организацию A.
3. Открыть дропдаун «Банковский счёт организации» — отображаются счета организации A. *(корректно)*
4. Не выбирая ничего, сменить организацию на B.
5. Снова открыть дропдаун «Банковский счёт организации».

**Ожидаемо:** в списке счета организации B.

**Фактически:** в списке те же счета организации A. Запрос за новыми данными не отправляется. Список обновляется на счета B только если начать вводить текст в поле.

**Влияние:** пользователь может выбрать банковский счёт, не принадлежащий выбранной организации.

### Технический анализ

**Затронутый файл:** `src/features/sdui/ui/nodes/fields/reference-field-node.tsx`

**Что делает бэкенд (корректно):** при смене организации сервер присылает SDUI-патчи, обновляющие проп поля `field.schetOrg`:
```json
{ "op": "setProp", "nodeId": "field.schetOrg", "key": "optionsSource",
  "value": { "url": "/api/dictionary-entries/BankovskieScheta/entries",
             "params": { "Organizatsiya": "30277", "skipDependsOn": "true" } } }
```
То есть новый фильтр доходит до клиента — серверная часть работает правильно. Баг чисто фронтовый.

**Механизм бага:**
- Список вариантов хранится в локальном стейте компонента: `const [options, setOptions] = useState<SelectOption[]>([])`.
- Наполняется функцией `fetchOptions`, которая вызывается в двух местах:
  - `onInputChange` при `reason === 'input'` — когда пользователь **печатает**;
  - `onOpen` — но **только при `options.length === 0`**:
    ```tsx
    onOpen={() => { if (options.length === 0) { void fetchOptions() } }}
    ```
- При смене организации патч обновляет `node.props.optionsSource`, компонент перерисовывается с новыми параметрами, **но локальный `options` не сбрасывается** — остаётся старый список.
- При повторном открытии `onOpen` видит непустой (устаревший) `options` → условие `length === 0` ложно → перезапрос не выполняется → показывается кеш прошлой организации.
- Ввод текста идёт по другой ветке (`onInputChange → fetchOptions(val)`) и использует уже актуальные `optionsSource.params`, поэтому список «чинится» только при печати.

**Суть:** данные закешированы в стейте компонента, инвалидация кеша при смене фильтра отсутствует.

**Направление фикса:** инвалидировать `options` при изменении значения параметров источника (сравнение по содержимому, а не по идентичности объекта-пропа), например через `useEffect`, зависящий от сериализованных `optionsSource.params` / `filter`, со сбросом `setOptions([])`. Тогда следующее открытие снова попадёт в ветку `options.length === 0` и перезапросит данные. Опционально — если дропдаун открыт в момент смены, перезапросить сразу, не дожидаясь переоткрытия. Фикс общий для всех ссылочных полей (компонент один).

> Статус: фикс уже вчерне реализован в `reference-field-node.tsx`, требует сборки (`npm run build`) и проверки в приложении.

## Баг 2. В панели «Показать всё» строки есть, но данные полей не отображаются

**Описание:**
При открытии боковой панели «Показать всё» список записей загружается, но визуально ячейки пустые — данные не рендерятся.

**Шаги воспроизведения:**
1. На справочном поле (например, «Банковский счёт организации» или «Банковский счёт контрагента») нажать «Показать всё».
2. Откроется боковая панель со списком записей.

**Ожидаемо:** в колонках списка видны значения записей (код, наименование, банк, счёт и т.д.), их можно выбрать.

**Фактически:** записи в списке присутствуют (список не пустой, счётчик показывает количество), но все колонки пустые — содержимое строк не отображается.

**Влияние:** выбрать запись из списка практически невозможно — не видно, что выбираешь.

### Технический анализ

**Затронутые файлы:**
- фронт: `src/features/sdui/ui/nodes/composite/list-node.tsx`
- бэкенд (источник контракта): `webbuh-api/.../view/composer/ChoicePanelComposer.java`

**Механизм бага — рассинхрон контракта колонок.**
Панель «Показать всё» строится на бэкенде как `LIST` с колонками `TABLE_COLUMN`. Бэкенд кладёт код атрибута колонки в проп **`attributeCode`** (`NodeProps.ATTRIBUTE_CODE = "attributeCode"`):
```java
.props(Map.of(NodeProps.HEADER, header, NodeProps.ATTRIBUTE_CODE, binding))
```
А фронт `list-node.tsx` для извлечения значения ячейки читает проп **`binding`**:
```tsx
accessorFn: (row) => {
  const binding = col.props?.binding as string  // ← undefined: бэкенд прислал attributeCode
  if (!binding) return ''                        // ← для каждой ячейки возвращается ''
  ...
}
```
Поскольку `col.props.binding` всегда `undefined`, `accessorFn` для всех ячеек возвращает пустую строку. Строки при этом загружены (`rows.length > 0`, поэтому состояние «нет данных» не показывается) → визуально пустые колонки при наличии записей. Это в точности совпадает с симптомом «записи есть, но визуал не рендерится».

**Дополнительный нюанс — значения-объекты.**
Часть колонок ссылается на атрибуты-справочники (`Bank`, `ValyutaScheta` и т.п.); их значение в строке — объект `{ id, nameRu, displayName, presentation, ... }`. Текущий `accessorFn` достаёт из объекта только поле `name`:
```tsx
if (val && typeof val === 'object' && 'name' in val) { return val.name ?? '' }
```
У DTO справочника поля `name` нет (есть `nameRu` / `displayName` / `presentation`), поэтому даже после исправления ключа такие ячейки покажут пусто или `[object Object]`.

**Направление фикса (фронт):**
1. Читать код колонки как `attributeCode` с фолбэком на `binding`:
   `const binding = (col.props?.attributeCode ?? col.props?.binding)`.
2. Для значений-объектов резолвить презентацию по приоритету `presentation → displayName → nameRu → name → String(id)`.

Альтернатива — поправить на бэкенде (слать `binding`), но устойчивее принять оба ключа на фронте. Фикс общий для всех `LIST`-панелей.

## Баг 3. Боковая панель (drawer) не прокручивается — нижние поля недоступны

**Описание:**
Содержимое боковой панели не помещается по высоте и не скроллится, из-за чего нижняя часть контента недоступна.

**Шаги воспроизведения:**
1. Открыть боковую панель с длинным содержимым: «Добавить» (форма создания новой записи справочника) или «Показать всё» с большим списком.
2. Попытаться прокрутить панель вниз.

**Ожидаемо:** содержимое прокручивается, нижние поля/строки доступны для просмотра и заполнения.

**Фактически:** прокрутки нет, нижняя часть контента обрезается. Поля, оказавшиеся ниже видимой области, невозможно заполнить.

**Влияние:** длинные формы создания и длинные списки нельзя использовать полностью — часть данных недоступна.

### Технический анализ

**Затронутый файл:** `src/features/sdui/ui/dialog-host.tsx`

**Механизм бага:**
У `Drawer` в `slotProps.paper.sx` задан `overflow: 'hidden'`, а внутренний контейнер содержимого
```tsx
<div className="flex h-full flex-col p-7"> {/* крестик закрытия */} {content} </div>
```
не создаёт прокручиваемую область — нет `overflow-y-auto` и ограничения высоты на блоке с контентом. В результате контент, превышающий высоту панели, обрезается родителем с `overflow: hidden`, и прокрутить его невозможно.

**Направление фикса (фронт):**
Сделать тело drawer flex-колонкой с фиксированной шапкой (крестик) и **прокручиваемой** областью контента:
```tsx
<div className="flex h-full flex-col p-7">
  <div className="flex items-center justify-end shrink-0"> {/* крестик */} </div>
  <div className="min-h-0 flex-1 overflow-y-auto"> {content} </div>
</div>
```

**Важное взаимодействие с Багом 2 (список).**
У компонента `LIST` (`list-node.tsx`) уже есть собственный внутренний скролл (`scrollRef` + `overflow-auto`) и infinite-scroll через `IntersectionObserver` с `root: scrollContainer`. Чтобы не получить двойную прокрутку и не сломать дозагрузку страниц:
- для drawer вида **CHOICE_FORM** (список «Показать всё») высота должна доходить до `LIST` по flex-цепочке (`min-h-0 flex-1`), чтобы список скроллился **внутри себя**;
- для drawer вида **форма/«Добавить»** прокручивать должен внешний контейнер контента.

Различать можно по `panel.node.props.kind === 'CHOICE_FORM'` (его выставляет бэкенд), либо обеспечить корректную flex-цепочку (`min-h-0 flex-1`) от `PAGE`/`VSTACK`-рендереров вниз к `LIST`, чтобы список получал ограниченную высоту, а формы — внешний скролл.
