- **Дата:** 2026-06-22
- **Owner (предложил):** пользователь (постановка); проектирование — architect

## 1. Контекст

### 1.1 Что есть сейчас

SDUI-форма документа собирается `ViewComposer`'ом из layout-метаданных (`layout_definitions/nodes/node_props`) + `CoreAttribute` (ADR-0006, `sdui-architecture.md`). Командная панель сверху формы — узел `TOOLBAR` (`node_code='commandBar'`).

**Кнопки сегодня заводятся РУЧНЫМ per-type SQL-сидом.** Пилот «Заявка на регистрацию ГП-сделки»: `migration_2026-05-28_sdui_layout_zayavka_buttons.sql` вручную вставляет в `layout_nodes` четыре `BUTTON` (`btn.postClose/save/post/print`) + `btn.print` с четырьмя дочерними `MENU_ITEM` (печатные формы) и все их `props` — суммарно 8 узлов + ~17 строк props на ОДИН документ.

`ViewController.handleCommand` (`ViewController.java:482-528`) уже маршрутизирует команды generic-ветками:
- `save`/`saveAndClose`/`post`/`postAndClose` → `processSave` (общая для всех типов, через `SduiSaveService` + `DocumentService.saveEntry`);
- `unpost` → `processUnpost`;
- `ref.*` → `processRefCommand` (ADR-0009: showAll/select/create/open);
- `dict.saveAndSelect` → `processDictionarySaveAndSelect` (ADR-0009 Phase 2);
- иначе → `handler.handleCommand` (туда уходит `print:<id>`); если handler `null` или команду не распознал — **тихий no-op** (только `log.debug`/`log.warn`, фронту пустые `patches`/`effects`).

Динамика панели по состоянию уже доказана: `handler.handleOpen` перепатчивает `btn.post` на «Отменить проведение»/`unpost` у проведённого документа (`sdui-architecture.md §9` пример, и SDUI-handler заявки ГП).

Печатные формы — авторитетный источник: `ManagerHandler.addPrintCommands()` возвращает `List<Map>{id, name, order}`; для пилота он отдаёт **ровно 4** команды (`ZayavkaGpSdelki`, `ZayavkaGpSdelkiKaz`, `Forma77`, `Forma77Kaz` — RU+KZ), что **посимвольно совпадает** с 4 ручными `MENU_ITEM` сида (проверено: `ZayavkaNaRegistratsiyuGPSdelkiManagerHandler.java:74-77` vs `migration_2026-05-28...sql:122-130`).

`sdui-architecture.md §9` **уже декларирует** целевую модель «`TOOLBAR commandSource=AUTO` → `CommandComposer` собирает кнопки из `CommandRegistry` + флаги-возможности + кастомные команды `ManagerHandler`». Константа `NodeProps.COMMAND_SOURCE = "commandSource"` в контракте есть. **Но `CommandRegistry`, `CommandComposer`, флаги-возможности на `CoreType`/`DocumentType` — в коде НЕ существуют** (это TODO-чеклист `sdui-architecture.md §13`, не реализация).

### 1.2 Что умеет фронт (ФИКСИРОВАН — проверено по коду fin-web)

| Узел/механизм | Факт | Источник |
|---|---|---|
| `BUTTON` | читает `label`/`command`/`enabled`/`variant`; `variant=primary`→MUI contained, `variant=dropdown`→Menu с дочерними `MENU_ITEM` | `nodes/action/button-node.tsx` |
| `BUTTON` value | **НЕ передаёт `value`** в dispatch — шлёт только `{type:COMMAND, command}` | там же |
| `BUTTON` icon | **НЕ рендерит `props.icon`** — только текстовый `label` | там же |
| `MENU_ITEM` | читает `label`/`command` | `nodes/action/menu-item-node.tsx` |
| `TOOLBAR` | flex-контейнер дочерних; нет авто-overflow → «Ещё» моделируем **явной** dropdown-кнопкой (§3.4), не адаптивным сворачиванием | `nodes/layout/toolbar-node.tsx` |
| `EffectHandler` | `navigate`/`openDialog`/`closeDialog`/`notify`/`download` — все есть | `lib/effect-handler.ts` |
| клиентские команды | Help/Find/Sort/CustomizeForm/ShowInList-UI — **НЕ реализованы вообще** | grep по sdui |
| заголовок формы | видимый заголовок = инжект `LABEL`-узла, не `props.title` (только `document.title`) | `nodes/layout/page-node.tsx` + MEMORY |

### 1.3 Проблема

Ручной per-type-сид кнопок не масштабируется: ~85 документов × **гомогенный** набор (запись/проведение/печать) = сотни рукописных `INSERT` для команд, которые у всех типов **одинаковы**, плюс неизбежный дрейф (один документ получил кнопку, другой нет) и отсутствие единообразия. Нужен generic-механизм сборки верхней панели — но честный относительно того, что реально реализовано (а не «дописать §9»), **и честный относительно границ экономии**: см. §6 — composer устраняет рукописный сид **только гомогенного закреплённого набора**; полный состав «Ещё» (301 `Form.Command` + `Document.*.Command` per-type) **не экономится** — он остаётся per-type, но переезжает из оторванного SQL в типобезопасный Java рядом с веткой обработки.

### 1.4 Что НЕ переоткрывается

Stateful form-session, revision, patch-first, scratch-модель, конфликт-резолв 409 (ADR-0006); server-driven ссылочное поле и `ref.*`-команды (ADR-0009); инварианты SDUI (сервер — авторитет, клиент — рендерер, layout — метаданные). Этот ADR **достраивает командную панель** в рамках этих инвариантов.

## 2. Охват (scope)

**В охвате — ТОЛЬКО верхняя командная панель формы вида `OBJECT_FORM`:**
- **документы** — «Главная командная панель формы» карточки документа (`ФормаОбъекта`);
- **карточка справочника** (`ФормаЭлемента`) — «Главная командная панель формы» + «Панель команд: КоманднаяПанель» — **но условно, см. §2.1 (фаза 1b)**.

**Явно ВНЕ охвата** (будущие фазы, зависят от ещё-не-существующей инфраструктуры либо от правки фиксированного фронта):
- **Командные панели списков справочников** (`ФормаСписка`, `ФормаВыбора`) и все их команды: `Create`/`CreateFolder`/`Copy`-в-списке/`Delete`-в-списке/`ИзменитьВыделенные`/`Choose`/`Find`/`CancelSearch`/`Sort`/`Tree`/`MoveUp`/`MoveDown`. **Причина:** `LayoutKind.LIST_FORM`/`CHOICE_FORM` в `ViewController`/`ViewComposer` **не обрабатываются** (зафиксировано `ADR-0009 §1.1`), а на фронте нет SDUI-узла-экрана списка. Списочная панель невозможна без экрана списка — это отдельный «ADR LIST_FORM screen». Из 938 кнопок справочников в каталоге реально достижима только карточка (`ФормаЭлемента`).
- **Кнопки внутри табличных частей (ТЧ)** и боковых **Групп** (212 + 67 кнопок документов; 513 + 114 справочников) — отдельная фаза.
- **Контекстные меню** строк/полей.
- **AUTO-сборка через `CommandRegistry` + capability-метаданные** для нестандартных панелей — **Phase 2** (см. §3.6).
- **Чисто-иконочные кнопки** (фронт не рендерит `icon`, §1.2) — выводятся только как текстовые либо вне Phase 1.

### 2.1 Фаза 1a (документы) vs 1b (карточка справочника)

`OBJECT_FORM`-сессия сегодня реализована и доказана **только для документов** (`ViewController` завязан на `DocumentEntry`/`DocumentAttribute`; `requireEntryForMutation`, `processSave` работают с документом). Полноценная редактируемая `OBJECT_FORM`-сессия **справочника** (open + EVENT/COMMAND + save карточки справочника) — это **ADR-0009 Phase 2**, и на момент написания она реализована лишь частично (дочерняя Dictionary-сессия для `ref.create`/`ref.open`, без полноценной самостоятельной формы-экрана справочника).

**Решение (challenger R2-A):**
- **Phase 1a — документы** (`ФормаОбъекта`): полностью в охвате, прувпойнт — пилот ГП.
- **Phase 1b — карточка справочника** (`ФормаЭлемента`): в охвате **ТОЛЬКО после** того, как `OBJECT_FORM`-сессия справочника реально заработает (handler + OPEN + setValue-патчи на карточке справочника как самостоятельном экране). До этого карточка справочника — наравне с LIST_FORM: инфра не готова. Не выводим панель карточки справочника, пока нет рабочей сессии. Это устраняет асимметрию: и список, и карточка справочника гейтятся одинаково — по наличию обрабатываемого `LayoutKind`/сессии.

Стандартный набор `OBJECT_FORM`-карточки справочника (когда 1b откроется): `ЗаписатьИЗакрыть`/`Записать` (+ `Печать`, если у справочника есть печатные формы). Проведения у справочника нет (§5).

## 3. Решение

**Принцип:** верхняя панель `OBJECT_FORM` собирается generic-композитором по фиксированному стандартному набору, а не рукописным сидом на каждый тип; динамика (проведён/новый/помечен на удаление) — рантайм-патчами от handler'а; пользовательские команды объявляет handler. Per-type SQL-сид остаётся как **override-исключение** для нестандартных панелей.

### 3.1 `CommandBarComposer` — фиксированный стандартный набор (Phase 1)

Узел `commandBar` (`TOOLBAR`) в layout несёт `props.commandSource = AUTO` (константа `NodeProps.COMMAND_SOURCE` уже в контракте) **вместо рукописного дерева кнопок**. При сборке `ViewComposer` для `TOOLBAR` с `commandSource=AUTO` зовёт новый `CommandBarComposer`, который инжектит **фиксированный** набор `BUTTON`/`MENU_ITEM`-узлов по **одному достоверному факту**, а не по «выведенным возможностям»:

| Факт (достоверный, проверяемый) | Что инжектит |
|---|---|
| тип — документ (`OBJECT_FORM`, domain=DOCUMENT) | `Провести и закрыть` (primary, `postAndClose`), `Записать` (`save`), `Провести` (`post`) — **всегда всем документам** (capability «проводится» в Phase 1 НЕ вычисляется, см. §3.2) |
| `ManagerHandler.addPrintCommands()` непуст | dropdown `Печать` (`variant=dropdown`) + `MENU_ITEM` на каждый элемент `addPrintCommands` (`label=name`, `command=print:<id>`) |
| handler зарегистрирован в `ViewFormHandlerRegistry` и объявил пользовательские команды (§3.4) | `BUTTON` на каждую (`command=<id>`, `label`) |

`CommandBarComposer` НЕ выводит «проводится ли документ» и НЕ читает несуществующих capability-флагов. Источник набора — единственный достоверный факт «это документная `OBJECT_FORM`» + непустой `addPrintCommands` + список пользовательских команд handler'а. Это снимает блокер «capability из `onPost` невыводим» (§8, Атака 1.1): `onPost` имеет `default {}` в интерфейсе → «наличие onPost» истинно у всех и ничего не различает; поэтому мы его и не используем как признак.

> **Почему фиксированный набор, а не `CommandRegistry`/AUTO-по-возможностям.** `CommandRegistry`, `CommandComposer` и флаги-возможности `CoreType` в коде не существуют (`sdui-architecture.md §13` — TODO). Phase 1 реализует **минимальную** проверенную дельту: фиксированный набор + рантайм-gating. Полноценный реестр команд с capability-метаданными — Phase 2 (§3.6). Это честная фазировка, а не «дописать §9 как малую дельту» (§8, Атака 5.1).

### 3.2 Динамика панели — рантайм-патчи от handler, НЕ автовывод

Доступность/состояние кнопок (`enabled`/`visible`/`label`/`command`) по состоянию записи (новый / проведён / помечен на удаление) патчатся `handler.handleOpen` (и `processSave`/`processUnpost` в ответе) через `setProp`. Это **уже работает** в пилоте: `btn.post` перепатчивается на «Отменить проведение»/`unpost` у проведённого документа.

Паритет с пилотом (`applyPostButtonState`):
- документ проведён → `setProp(btn.post, label, "Отменить проведение")` + `setProp(btn.post, command, "unpost")`; кнопка `Провести и закрыть` скрывается/гасится;
- документ новый (id=null) → `Провести`/`Записать` доступны, но запись создаётся первым `save` (см. `processSave`);
- помечен на удаление → handler решает (обычно гасит `Провести`).

### 3.3 Непроводимые документы — ОДИН механизм скрытия + явное поведение generic-ветки (challenger R2-B)

Фиксированный набор §3.1 даёт `Провести` **всем** документам. Для документов, которые не проводятся вообще:

**Решение (один механизм, не два):** скрытие `Провести`/`Провести и закрыть` для непроводимого типа делает **`handler.handleOpen` через `setProp(visible=false)`** — это тот же механизм, что и динамика §3.2 (а не альтернативный per-type-сид). Унификация: один путь, тот же, что доказан пилотом.

**Защита от «молчаливо несработавшей кнопки»:** если handler забыл скрыть `Провести` у непроводимого типа и пользователь кликнул `post` → generic `processSave(isPosting=true)` дойдёт до `onPost`, который у такого типа `default {}` (пустой) → раньше это был бы **тихий no-op** («кнопка сработала, движений нет»). **Требование Phase 1:** generic post-ветка обязана сделать поведение явным — если тип не объявлен проводимым (нет переопределённого `onPost`/нет регистрируемых движений по факту обработки), `processSave(isPosting)` возвращает `notify("error"/"warning", "Документ данного вида не проводится")` вместо тихого успеха. Это превращает забытый `setProp` из невидимого UX-бага в видимое уведомление. (Точную проверку «проводим ли тип» в Phase 1 берём консервативно: тип считается проводимым, если его `ObjectHandler` переопределяет `onPost` ИЛИ если по итогам обработки появились движения; при пустом результате — явный notify, не молчаливый commit `isPosted=true`.)

### 3.4 Полный охват: закреплённые кнопки + меню «Ещё ⋯»

**Верхняя командная панель 1С — это НЕ 5-7 видимых кнопок.** Это небольшой набор закреплённых видимых кнопок **плюс меню «Ещё ⋯» (троеточие), в котором лежит весь остальной массив команд формы.** Масштаб реален: у документа `ЭСФ` в главной панели **24** команды, `ЭлектронноеПисьмоВходящее` — **15**, `СчетКОплате` — **10**, `НачислениеЗарплатыСотрудникам` — **9**. По всему листу документов: `Form.Command` = 301, стандартные ~81, плюс `Document.*.Command.*`. Охватываем **все** команды верхней панели, не только видимые.

#### 3.4.1 Модель: закреплённые + одна dropdown-кнопка «Ещё»

`CommandBarComposer` (`commandSource=AUTO`) эмитит:
1. **Закреплённые `BUTTON`** — `Провести и закрыть` (primary), `Записать`, `Провести`, отдельная dropdown-кнопка `Печать` (если есть печатные формы), и доменные главные действия, помеченные handler'ом (§3.4.2);
2. **ОДНУ dropdown-кнопку «Ещё»** (`label="Ещё"`, `variant=dropdown`) — её дочерние узлы покрывают **весь остальной массив**: стандартные карточные (`Перечитать`/`Пометить на удаление`/`Показать в списке`/`Скопировать`), все непомеченные `Form.Command`, `Document.<Тип>.Command.*`, `CommonCommand.*`.

**Новый `NodeType` НЕ нужен:** «Ещё» — обычная `variant=dropdown` `BUTTON`, тот же механизм, что `Печать` (фронт `button-node.tsx` рендерит MUI `<Menu>` из дочерних узлов через `NodeRenderer` — проверено по коду).

#### 3.4.2 Правило «закреплено vs Ещё» (challenger Возр. 2 + ИТС v8std/716)

**По умолчанию закреплены:** запись (`save`/`saveAndClose`), проведение (`post`/`postAndClose`/`unpost`), `Печать`. **Остальное → «Ещё».**

**НО доменное главное действие закрепляется видимым, НЕ прячется в «Ещё».** ИТС v8std/716 «Командная панель документа» (через `analyst-1c`): *«командная панель должна позволять выполнить самые важные и частотные действия, **не открывая подменю Ещё**»*; доменные команды ключевого действия документа (`Document.ЭСФ.Command.Отправить`) — **всегда видимая кнопка**, не пункт «Ещё» (механика БСП «подключаемые команды»: ЭСФ-команды в отдельной видимой группе). Дефолт «всё кроме записи/проведения/печати → Ещё» **недостаточен** — он спрятал бы `Отправить`, главное действие ЭСФ.

**Решение:** handler **явно** помечает доменное главное действие как закреплённое через `pinned`-флаг (§3.4.3). Команды без пометки → «Ещё»; handler осознанно поднимает важные доменные команды в закреплённые. Это per-type ручная пометка — но честная (§6, challenger Возр. 4): ответственность «не спрятать главное действие» лежит на авторе handler'а каждого типа.

#### 3.4.3 Контракт объявления команд handler'ом — четыре категории

Категории команд верхней панели — **четыре** (по каталогу, не две):

| Категория | Что | Пример | Роутинг |
|---|---|---|---|
| 1. `Form.StandardCommand.*` | платформенные стандартные | `Write`/`Post`/`Reread`/`SetDeletionMark`/`ShowInList`/`Help`/`CustomizeForm` | generic-ветки `ViewController` (§3.5) |
| 2. `Form.Command` (301) | локальные процедуры формы | `Заполнить`/`Рассчитать`/`Перезаполнить…`/`СчётФактураСоздать` | `handler.handleCommand` |
| 3. `Document.<Тип>.Command.*` | команды объекта/расширения | ЭСФ: `Отправить`/`Отозвать`/`Отклонить`/`Проверить`/`Обновить`/`Выгрузить`; письма: `Ответить`/`ОтветитьВсем`/`Переслать` | `handler.handleCommand` |
| 4. `CommonCommand.*` / `DocumentJournal.*.Command.*` | общие/журнальные | `СохранитьПисьмоНаДиск` | `handler.handleCommand` |

Категории 2, 3, 4 на бэке роутятся **одинаково** (`COMMAND {command=id}` → `handler.handleCommand`). **Разделение ответственности (server-driven-принцип, §3.4.7):**
- **ЧТО за кнопки, где, как выглядят** (состав/вид/порядок/закрепление/секция) — **серверные метаданные** (реестр `command_bar_items`), НЕ код handler'а;
- **ЧТО делает клик** (мутации/эффекты/`download`) — `handler.handleCommand(session, entry, commandId, patches, effects)`;
- **динамика по состоянию записи** (видимость/доступность для проведён/нов/помечен) — `setProp`-патчи handler'а (§3.2) — это **поведение**, не статический UI.

`CommandBarComposer` читает реестр по `typeCode`: `pinned=true` → `BUTTON` в закреплённой зоне; `pinned=false` → `MENU_ITEM` внутри «Ещё»; `section` → `SEPARATOR` между группами. Handler **не объявляет**, какие кнопки существуют — только обрабатывает клик. Модель реестра — §3.4.7.

#### 3.4.4 Секции в «Ещё» — SEPARATOR, не вложенные подменю (challenger Возр. 1)

ИТС (через `analyst-1c`): меню «Ещё» в 1С **иерархично** — внутри вложенные подменю (`Создать на основании`▸, `Печать`▸, `Показать в списке`▸). Фронт `MENU_ITEM` **плоский, cascading-подменю не умеет** (проверено `menu-item-node.tsx`). Но `button-node.tsx` рендерит детей `<Menu>` через `NodeRenderer` → в Menu можно класть `SEPARATOR` (есть в `NodeType`) и `LABEL` как заголовок секции.

**Решение:** «Ещё» — **плоский список с секциями через `SEPARATOR`/`LABEL`-узлы**, не вложенные подменю. `CommandBarComposer` вставляет `SEPARATOR` между секциями (по `CommandBarItem.section`). **Это сознательное упрощение**, явно зафиксированное: полный паритет с cascading-подменю 1С — будущая фаза (требует доработки фронта). `Печать` остаётся **отдельной** dropdown-кнопкой рядом с «Ещё» (не вложена). `Создать на основании` — вне Phase 1 (документная связь — отдельный ADR).

#### 3.4.5 Примеры (строки реестра `command_bar_items` — §3.4.7; транслитерация — по таблице)

1. **`Заполнить`** (`Form.Command`) → строка реестра `{commandId:"Zapolnit", label:"Заполнить", kind:server, pinned:false}` (в «Ещё»). `handler.handleCommand("Zapolnit")` пересчитывает ТЧ → `replaceNode`/`setValue` + `notify`. (`Заполнить`→`Zapolnit`.)

(`Выгрузить`→`Vygruzit`, `Казначейство`→`Kaznacheystvo`; `СформироватьSWIFTФайл`→`SformirovatSWIFTFayl`, аббревиатура `SWIFT` как есть.)
3. **`Document.ЭСФ.Command.*`** (категория 3): `{commandId:"Otpravit", label:"Отправить", pinned:true}`, `{commandId:"Otozvat", label:"Отозвать", pinned:true}`, `{commandId:"Otklonit", label:"Отклонить"}`, `{commandId:"Proverit", label:"Проверить"}`, `{commandId:"Obnovit", label:"Обновить"}`, `{commandId:"Vygruzit", label:"Выгрузить", kind:server}`. `Отправить`/`Отозвать` закреплены (главные действия ЭСФ — ИТС), остальные в «Ещё». (Транслитерация: `Отправить`→`Otpravit`, `Отозвать`→`Otozvat`, `Отклонить`→`Otklonit`, `Проверить`→`Proverit`, `Обновить`→`Obnovit`, `Выгрузить`→`Vygruzit`.)

#### 3.4.6 Пер-типовая вариативность набора команд — два слоя

**Набор кнопок у каждого документа РАЗНЫЙ.** Это ключевой факт, который раздел делает явным. Командный интерфейс формы 1С — **два слоя** (`analyst-1c`, ИТС, раздел Разработка → Управляемые формы → Командный интерфейс):

> Командный интерфейс управляемой формы документа в 1С формируется в два слоя. **Первый слой** — стандартные команды расширения объекта (Записать, Провести, Провести и закрыть, Отменить проведение, Перечитать, Пометить на удаление, Показать в списке, Справка и др.) — платформа добавляет **автоматически** на основании типа основного реквизита формы; свойство `АвтоЗаполнение` командной панели (`Истина` по умолчанию) подхватывает их без объявления в редакторе. **Второй слой** — явно сконфигурированные разработчиком элементы: пользовательские команды формы (`Form.Commands`), объектные команды документа (`Document.<Тип>.Command`) и стандартные команды с переопределёнными свойствами. Выгрузка явно сконфигурированных элементов содержит **только второй слой**; отсутствие документа в такой выгрузке означает, что он использует **исключительно авто-набор платформы**, а не то, что у него нет командной панели.

**Проекция на webbuh:**

| Слой | Что | Откуда в webbuh | Вариативность |
|---|---|---|---|
| **Стандартный костяк** (1-й слой) | `save`/`saveAndClose`/`post`/`postAndClose`/`unpost`/`reread`/`setDeletionMark`/`showInList` + `Печать` | **`CommandBarComposer`** генерит по факту «документная OBJECT_FORM» (§3.1) — фиксированные правила, **не из реестра** | **ЕДИНЫЙ** для всех 91 типа |
| **Уникальная часть** (2-й слой) | `Form.Command` + `Document.<Тип>.Command` + `CommonCommand` | **строки серверного реестра `command_bar_items` на тип** (сид из каталога `docs_buttons.tsv`) → `CommandBarComposer` (§3.4.7) | **РАЗНАЯ** на тип = разные строки реестра |

**Распределение (проверено по каталогу):** из **91** типа документа кастомные команды в верхней панели есть лишь у **39** (35 с `Form.Command`); остальные **~52** — **только стандартный костяк**. Разброс кастома: `ЭСФ` 24 (17 кастомных, 4 категории, pinned-действия `Отправить`/`Отозвать`), `ЭлектронноеПисьмоВходящее` 15, `СчетКОплате` 10, `НачислениеЗарплатыСотрудникам` 9, …, хвост по 1-2 (`АктСверкиВзаиморасчетов`, `ВводНачальныхОстатковВНА`, и сам пилот `ЗаявкаНаРегистрациюГПСделки` — 1).

**«Разный набор на тип» = разные СТРОКИ реестра метаданных на тип (данные), а не разные Java-методы.** **Отсутствие типа в каталоге ≠ «нет кнопок»** — это «только авто-стандарт»: ~52 типа **не имеют строк** в `command_bar_items` (только костяк от `CommandBarComposer`). Каталог — **дельта поверх костяка** (2-й слой), а не полный набор.

#### 3.4.7 Механизм источника уникальной части — DATA-DRIVEN реестр команд (server-driven-принцип)

> **Разворот решения (PM-директива владельца, §8 раунд 5).** Прежняя редакция выбирала вариант B (Java `getCustomCommands()` per-handler) по критерию «инженерная экономия + compiler-safety». **Критерий пересмотрен владельцем**: цель проекта — **полностью server-driven UI**, где состав кнопок определяется **серверными метаданными**, а handler отвечает **только за обработку действий**. Вариант B лишь переносил декларацию UI из SQL в Java — UI оставался описан кодом конкретных handler'ов, что **не по-настоящему server-driven**. Compiler-safety — слабая выгода, заменяется CI-гейтом. **Выбран data-driven (бывший вариант A).**

**Модель: серверный реестр команд `command_bar_items`.** Состав/вид/порядок/закрепление/секция КАСТОМНЫХ команд (категории 2/3/4) — **данные**, не код. Реестр (новая таблица либо расширение layout-метаданных §1.1):

| Поле | Тип | Назначение |
|---|---|---|
| `targetTypeCode` | string | тип документа (+ `layoutKind=OBJECT_FORM`) — ключ выборки |
| `commandId` | string | id команды (транслитерация по таблице): `Otpravit`, `Zapolnit`, … — посимвольно = ветка в `handleCommand` |
| `label` / `labelKz` | string | подписи RU/KZ (двуязычность проекта) |
| `icon` | string? | метаданные на будущее (фронт не рендерит, §1.2) |
| `pinned` | boolean | `true` — закреплённый `BUTTON`; `false` — пункт «Ещё» (§3.4.2) |
| `section` | string? | логическая секция в «Ещё» → composer вставит `SEPARATOR` (§3.4.4) |
| `orderIndex` | int | порядок среди закреплённых / внутри «Ещё» |
| `kind` | enum `server\|client` | `server` — `COMMAND` на `/api/view`; `client` — клиентский эффект (в Phase 1 client-кастома нет) |

`CommandBarComposer` читает реестр по `targetTypeCode` → инжектит закреплённые `BUTTON` (`pinned=true`) + пункты «Ещё» (`pinned=false`) с `SEPARATOR` по `section`, всё **из данных**. **Стандартный костяк остаётся фиксированными правилами composer'а** (не строками реестра) — обоснование единообразия ниже.

**Handler = ТОЛЬКО обработка.** Убран контракт-декларация `getCustomCommands()`. Handler реализует:
- `handleCommand(session, entry, commandId, patches, effects)` — что делает клик (мутации/эффекты/`download`);
- `getHandledCommandIds()` — **контракт обработки для CI-гейта** (множество id, которые handler умеет обрабатывать). Это **НЕ декларация UI** (не несёт label/pinned/section/порядок) — только список «что я умею обработать», для проверки §ниже.

**Граница данные vs код — чёткий шов (challenger возр. б):**

| Аспект | Где | Почему |
|---|---|---|
| **Статическая композиция**: какие кнопки, где (pinned/Ещё), порядок, label, секция | **метаданные** (`command_bar_items`) | чистый UI-состав = данные |
| **Что делает клик** (мутации/эффекты/download) | **код** (`handleCommand`) | бизнес-логика, не UI |
| **Динамика по состоянию записи**: видимость/доступность для проведён/нов/помечен; смена `Провести`↔`Отменить проведение` | **код** (`setProp`-патчи handler'а, §3.2) | это **поведение** (реакция на состояние), не статический UI; доказано пилотом |

**`visibleWhen`/`enabledWhen` в метаданных в Phase 1 НЕ вводим (строгий вариант, challenger возр. б).** Любая зависимость видимости/доступности от состояния записи — **только** через `setProp`-патч handler'а (§3.2). Причина: формальной границы «простое условие по атрибуту vs сложное» не существует (`developer-1c`: в 1С условной видимости по реквизиту нет декларативно — всё в коде модуля формы), а перекрытие `visibleWhen` с §3.2-патчами на одном классе условий («видимость по `isPosted`») породило бы два механизма для одного и разнобой между типами. Шов однозначен: **статика → данные; любая зависимость-от-состояния → код-патч**. Декларативные условия — возможная будущая фаза с формально перечислимым набором предикатов, не Phase 1.

**Печать (`addPrintCommands()`) — явное обоснованное ИСКЛЮЧЕНИЕ из «UI=данные» (challenger возр. в):** остаётся Java per-handler, в реестр НЕ переезжает. Обоснование (не «она динамична» — в нашем коде `addPrintCommands()` статична, без параметра entry, и реестр её технически потянул бы):
1. **Платформенный прообраз 1С** — печать в 1С *есть* код (БСП `УправлениеПечатью`, `ДобавитьКомандыПечати` в модуле менеджера), не статическая таблица; Java-источник соответствует 1С, а не нарушает принцип;
2. **Печать несёт исполняемую ресурс-привязку** (`getTemplateName`, маппинг `id`→XLSX-шаблон, XLSX→PDF) — это **UI + ресурс**, не чистая декларация UI;
3. **Будущая рантайм-динамика** (права/состояние/подключаемые формы по ИТС БСП) выразима только кодом — реестр её не вместит.

Это записано как **исключение, а не замалчивается**: кастом — данные, печать — особый код-источник по перечисленным причинам.

**CI-гейт — замена compiler-safety (challenger возр. а, г) — Phase 1, не Phase 2:**
1. **Источник истины = прод-артефакт.** Гейт читает **тот же миграционный сид** `command_bar_items`, что накатывается в прод (не тест-фикстуру, не отдельную копию). Сид-миграция — иммутабельный источник; ручные правки строк в прод-БД запрещены (как и для layout-метаданных §1.1). Зелёный гейт ⇒ прод-реестр корректен, потому что гейт проверяет ровно накатываемый артефакт.
2. **Полнота связи id↔обработка:** каждый `commandId` из реестра (`kind=server`) обязан иметь обработчик — ветку в generic-роутере ИЛИ в `getHandledCommandIds()` handler'а данного типа. **Orphan server-ветка** (id в `getHandledCommandIds`, но нет строки в реестре) → **error** (не warn): это прямой индикатор рассинхрона данные↔код — забытая/выпиленная кнопка с живой обработкой.
3. **Валидация схемы строк** (компилятор раньше проверял всю структуру `CommandBarItem`): гейт валидирует типы полей (`pinned` boolean, `orderIndex` int) и enum-домены (`kind ∈ {server,client}`, `section` из допустимого набора) каждой строки — не только связь id↔handler.
4. **Транслитерация `commandId`** (challenger возр. г, CLAUDE.md-инвариант): `commandId` теперь живёт и в данных (сид), и в коде (`getHandledCommandIds`/`handleCommand`) — опечатка в сиде (`Otpravить` вместо `Otpravit`) = кнопка есть, клик → тихий no-op. Гейт проверяет транслитерацию `commandId` по таблице (как `TransliterationConsistencyTest` для кодов типов). Это закрывает расширенный фронт «молчаливо несработавшего хендлера».

**Принятый осознанно риск (честно, challenger возр. г):** разворот на data-driven **воссоздаёт** рассинхрон декларация↔обработка по границе данные↔код — ровно тот, который прежняя редакция (вариант B) избегала компилятором. Это **сознательная плата за server-driven-принцип**: UI-состав в данных важнее compiler-safety. Защита — CI-гейт (пп. 1-4) + транслит-проверка; она слабее компилятора по форме, но достаточна при условии «источник гейта = прод-артефакт» (п. 1) и «orphan = error» (п. 2).

**Путь наполнения (механический, задача data-engineer):** для каждого из 39 типов — фильтр каталога (`docs_buttons.tsv`, верхняя панель, не-StandardCommand) → строки реестра `{commandId (транслитерация), label, labelKz, pinned, section, orderIndex, kind}` → сид-миграция. Обработка (`handleCommand`-ветки + `getHandledCommandIds`) — backend per-type (бизнес-логика, неизбежно код). **Отдельная задача** (§7), не «реализовать раз».

**Вывод по (б):** клиентские команды **классифицированы**, каждой дано определённое поведение (`Help` — отложен с конкретной причиной «нет контента» и планом «статический help на вебе»; `CustomizeForm`/`SaveAs` — отложены с перечнем фронт-дельт). «Охватить всё» = **классифицировать и дать определённое решение**, а не обязательно вывести кнопку немедленно. Это водораздел между «отложено с записанной причиной» и «молчаливо выкинуто».

### 3.6 Миграция пилота ГП с ручного сида на `commandSource=AUTO`

1. Узлу `commandBar` пилота проставить `props.commandSource = AUTO`; **удалить** рукописные `btn.postClose/save/post/print` + 4 `MENU_ITEM` + их props (это и есть отказ от per-type-сида на эталоне).
2. `CommandBarComposer` сгенерит тот же набор: документ → `Провести и закрыть`/`Записать`/`Провести`; `addPrintCommands()` непуст (4 формы) → dropdown `Печать` с 4 пунктами. **Регресса печати нет** — `addPrintCommands` пилота возвращает посимвольно те же 4 пункта, что были в сиде (§1.1, проверено; §8 Атака 1.3 опровергнута фактом).
3. `handler.handleOpen` пилота уже перепатчивает `btn.post`→`unpost` — динамика §3.2 сохраняется.
4. Прувпойнт: пилот после миграции рендерит идентичную панель без единого рукописного `BUTTON`-узла → доказательство, что `CommandBarComposer` заменяет сид без потери поведения.

**Пилот ГП не репрезентативен по объёму кастома** — у него в верхней панели **1** кастомная команда (`ВыгрузитьВКазначейство`-класс), стандартный костяк + печать. Он доказывает **костяк + динамику + печать**, но не нагружает механизм уникальной части (§3.4.7). **Второй прувпойнт — `ЭСФ`** (24 команды в гл.панели, 17 кастомных, все 4 категории, pinned-действия `Отправить`/`Отозвать`, «Ещё» с секциями): это репрезентативный стресс-тест `getCustomCommands()` + «Ещё» + `pinned` + `Document.<Тип>.Command`-роутинга. Рекомендуется как следующий тип после пилота.

## 4. Различия документ vs справочник

| | Документ (`ФормаОбъекта`) — Phase 1a | Карточка справочника (`ФормаЭлемента`) — Phase 1b |
|---|---|---|
| Проведение | есть: `post`/`postAndClose`/`unpost` | **нет** (у справочника нет проведения) |
| Запись | `save`/`saveAndClose` | `save`/`saveAndClose` (часто `ЗаписатьИЗакрыть` primary) |
| Печать | dropdown из `addPrintCommands` | dropdown, если у справочника есть печатные формы |
| Готовность инфры | реализована (DocumentEntry-сессия) | **зависит от ADR-0009 Phase 2** OBJECT_FORM-сессии справочника (§2.1) |
| Создание/группы/иерархия/выбор | — | списочные → **LIST_FORM, вне scope** |

`Создать`/`СоздатьГруппу`/иерархия/`ИзменитьВыделенные`/`Выбрать` — атрибуты **списка** справочника, не карточки → целиком вне scope (§2). «Выбрать» из `ФормаВыбора` уже покрыта `ref.select`/`dict.saveAndSelect` (ADR-0009), как самостоятельная command-bar-команда не вводится.

## 5. Состояния кнопок (паритет с пилотом)

`enabled`/`visible`/`label`/`command` — патчи `setProp` от handler:
- **новый** (id=null): `Провести`/`Записать` активны;
- **проведён**: `btn.post` → label «Отменить проведение», command `unpost`; `Провести и закрыть` скрыт;
- **помечен на удаление**: `Провести` скрыт/погашен (handler);
- **непроводимый тип**: `Провести`/`Провести и закрыть` скрыты handler'ом (§3.3), generic post-ветка защищена явным notify.

Это ровно `applyPostButtonState` пилота, обобщённый: набор приходит из `CommandBarComposer`, состояние — из handler-патчей.

## 6. Последствия

**Выигрываем:**
- **По-настоящему server-driven UI:** состав/вид/порядок/закрепление кастомных кнопок — **серверные метаданные** (реестр `command_bar_items`), handler — **только обработка**. Изменить панель типа (label, порядок, закрепить/в «Ещё») = правка строк реестра, без перекомпиляции handler'ов. Это honors цель проекта (UI = данные).
- **Гомогенный костяк** (запись/проведение/печать) генерится композитором **1 раз** фиксированными правилами — не дублируется per-type.
- **Пер-типовая вариативность — данные, не код:** «разный набор на тип» = разные строки реестра (§3.4.6), а не 39 Java-методов.
- Печать — единый источник `ManagerHandler.addPrintCommands()` (явное обоснованное исключение, §3.4.7).
- Динамика панели — один доказанный механизм (handler-патчи `setProp`), чёткий шов «статика=данные / поведение=код» (§3.4.7).
- Контракт фронта не меняется; фронт получает готовое дерево, источник состава ему прозрачен.

**Платим (честно — challenger раунд 5):**
- **Принятый осознанно рассинхрон-риск:** `commandId` живёт и в данных (реестр-сид), и в коде (`handleCommand`/`getHandledCommandIds`) — связь декларация↔обработка разрезана по границе данные↔код. Компилятор её не ловит (в отличие от отвергнутого варианта B). Это **сознательная плата за server-driven-принцип**. Защита — **CI-гейт Phase 1** (§3.4.7): источник = прод-сид; `commandId`(server)⇒обработчик; orphan server-ветка = **error**; валидация схемы строк; транслит-проверка `commandId`. Гейт слабее компилятора по форме, достаточен при «источник=прод-артефакт».
- **`pinned` доменных действий и состав «Ещё»** — per-type **данные** (строки реестра), наполняются из каталога. Это не экономит объём (39 типов × N команд всё равно описываются), но переносит описание в данные (цель SDUI), а обработку оставляет минимальной в коде.
- **Наполнение для 39 типов — отдельная задача:** сид реестра (data-engineer) + `handleCommand`-ветки/`getHandledCommandIds` (backend). **Не «реализовать раз».** ~52 типа не требуют строк реестра (только костяк).
- Новая таблица `command_bar_items` + `CommandBarComposer` + 3 generic-ветки (`reread`/`setDeletionMark`/`showInList`) + `getHandledCommandIds` контракт + CI-гейт.
- Защита generic post-ветки от тихого no-op (§3.3).
- Карточка справочника (1b) откладывается до ADR-0009 Phase 2.

**Отложено (с обоснованием, не молча):**
- **Cascading-подменю «Ещё»** → в Phase 1 плоские секции через `SEPARATOR`/`LABEL` (§3.4.4); полный иерархический паритет с 1С — будущая фаза (фронт-доработка).
- **`Создать на основании`** (группа в «Ещё» 1С) — отдельный ADR документной связи.
- **`Help`** — нет контента (`HelpContent` пуст в БГУ КЗ, `developer-1c`); отложено до статического help-контента на вебе (§3.5б).
- **`CustomizeForm`/`SaveAs`** — фронт-дельты перечислены в [frontend-spec](../sdui/frontend-spec-command-bar.md).
- AUTO через `CommandRegistry` + capability-метаданные — Phase 2 (с обязательным CI-гейтом против кнопок-пустышек, §8 Атака 1.2).
- Командные панели списков (`LIST_FORM`/`CHOICE_FORM`), ТЧ-кнопки, группы, контекстные меню, иконочные кнопки.

## 8. Альтернативы и контраргументы (challenger round-trip)

Red-team review проведён `challenger`'ом (2 раунда). Формат: **возражение → ответ автора → статус**.

### Blocker

1. **Атака 1.1 — «возможности типа» из `onPost` невыводимы** [blocker]: `DocumentObjectHandler.onPost` имеет `default {}` → «есть onPost» истинно у всех документов и ничего не различает; capability-флагов в коде нет (TODO). — **Ответ:** принято. Автовывод снят. Phase 1 `CommandBarComposer` инжектит `Провести` **всем** документам по факту «это документная OBJECT_FORM» (не по onPost); состояние — рантайм-патч по фактическому `isPosted` (§3.1, §3.2). Capability-флаг — Phase 2, ручные метаданные. — **Статус: closed.**
2. **Атака 4.1 — командная панель списка обещана, но `LIST_FORM`/`CHOICE_FORM` не обрабатываются** [blocker]: `ADR-0009 §1.1` — эти LayoutKind не обрабатываются в `ViewController`/`ViewComposer`; на фронте нет узла-экрана списка. — **Ответ:** принято полностью. Scope сужен до `OBJECT_FORM` (§2): списки справочников, `ИзменитьВыделенные`, `Создать`/`Tree`/`Find`/`Sort` — вне охвата, зависят от будущего «ADR LIST_FORM screen». — **Статус: closed.**
3. **Атака 5.1 — AUTO стоит на 3 несуществующих компонентах (`CommandRegistry`/`CommandComposer`/capability-флаги), продан как «дописать §9»** [blocker]: единственное проверенное в проде — per-type-сид + generic save/post/unpost/ref. — **Ответ:** принято. Decision переписан на честную фазировку (§3.1, §3.6): Phase 1 = фиксированный набор + рантайм-gating патчами (предложенная challenger'ом «недооценённая альтернатива 5.1» принята как основа); `CommandRegistry`/capability/AUTO-по-возможностям → Phase 2. — **Статус: closed.**
4. **Атака 2.1 — `ИзменитьВыделенные` требует `value`, а `BUTTON` его не шлёт** [blocker]: `ADR-0009 §1.1` — на COMMAND dispatch шлёт `{type, command}` без value; `value` на COMMAND введён ADR-0009 только под `ref.select`. — **Ответ:** принято. `ИзменитьВыделенные` — списочная, ушла в out-of-scope LIST_FORM (§3.4). Весь `OBJECT_FORM`-набор безаргументен (или аргумент в суффиксе `print:<id>`), `BUTTON` без value достаточно — фронт не трогаем. — **Статус: closed.**

### Major

5. **Атака 1.2 — AUTO множит кнопки-пустышки без гейта** [major]: composer нагенерит кнопку для типа, у которого нет ветки в `handleCommand` → тихий no-op на 85 документах. — **Ответ:** принято; в Phase 1 не актуально (набор фиксирован, все ветки реализованы — пустышек нет by construction). Для Phase 2 зафиксирован **обязательный CI-гейт**: каждая инжектируемая composer'ом команда имеет ветку в generic-роутере ИЛИ в `handler.getSupportedCommands()` (§6, §7). — **Статус: closed (Phase 1) / условие Phase 2.**
6. **Атака 1.3 — миграция пилота на AUTO теряет печатные формы** [major]: сид руками развернул 4 пункта (RU+KZ); если `addPrintCommands` отдаёт 2 — регресс. — **Ответ:** **опровергнуто фактом.** `ZayavkaNaRegistratsiyuGPSdelkiManagerHandler.java:74-77` возвращает ровно 4 команды (`ZayavkaGpSdelki`/`...Kaz`/`Forma77`/`Forma77Kaz`), KZ-варианты — отдельные пункты, лейблы посимвольно совпадают с сидом. AUTO-dropdown = идентичные 4 пункта (§3.6). Challenger проверил код и снял. — **Статус: closed (опровергнуто).**
7. **Атака 3.1 — смешаны карточные и списочные команды** [major]: `ShowInList`/`SetDeletionMark`/`Copy` — списочные операции; в карточке семантика другая. — **Ответ:** принято. Введены два раздельных каталога (§3.5 — только `OBJECT_FORM`; LIST_FORM вне scope). Классификация client/server — внутри каждого. — **Статус: closed.**
8. **Атака R2-A — карточка справочника опирается на нереализованную ADR-0009 Phase 2** [major, раунд 2]: асимметрия — LIST_FORM вынесен за «не существует», а карточку справочника (497 форм) оставил при той же неопределённости OBJECT_FORM-сессии справочника. — **Ответ:** принято. Введено явное разделение Phase 1a (документы, инфра готова) / Phase 1b (карточка справочника — **только после** рабочей OBJECT_FORM-сессии справочника ADR-0009 Phase 2, §2.1). Гейт симметричен списку. — **Статус: closed.**
9. **Атака R2-B — gating непроводимых документов: два механизма (сид vs патч) без прувпойнта + тихий no-op в generic post-ветке** [major, раунд 2]: забыли `setProp` → кнопка `Провести` висит на непроводимом, клик → пустой `onPost` → молчаливый no-op. — **Ответ:** принято. Зафиксирован **один** механизм скрытия — `handler.handleOpen` `setProp(visible=false)` (тот же, что динамика §3.2, доказан пилотом), плюс generic post-ветка обязана делать поведение явным: при непроводимом типе/пустом результате `onPost` → `notify("error")`, не тихий commit (§3.3). — **Статус: closed.**

### Minor

10. **Атака 2.2 — `icon` не рендерится фронтом** [minor]: пилот кладёт `icon=print`, фронт игнорирует. — **Ответ:** принято; зафиксировано (§1.2, §2): команды Phase 1 — текстовый `label`; иконочные кнопки 1С вне Phase 1; `icon` допустим как метаданные на будущее, без визуала. `variant=primary`(contained) работает. — **Статус: closed.**
11. **Атака 3.2 — `ShowInList`/`Reread` — effect/read-only, не мутация** [minor]: классифицировать как «server-roundtrip мутация» неточно. — **Ответ:** принято; `showInList` → `effect navigate`, `reread` → reload записи (read-only), `setDeletionMark` остаётся мутацией (§3.5а). Отдельные generic-ветки. — **Статус: closed.**
12. **Атака 4.2 — «Выбрать» уже = `ref.select`** [minor]: покрыта ADR-0009. — **Ответ:** принято; не дублируется, и `ФормаВыбора` вне scope (§4). — **Статус: closed.**

### Раунд 3 — модель «Ещё ⋯» (полный охват) и client-команды

После расширения охвата на **весь** массив команд верхней панели (закреплённые + «Ещё», §3.4) проведён отдельный challenger round-trip. Возр. 2 и 3 закрыты первоисточниками 1С (`analyst-1c`, `developer-1c`).

13. **Возр. 1 — плоский «Ещё» теряет иерархическую группировку 1С** [major]: `analyst-1c` (ИТС v8std/716) — «Ещё» в 1С иерархично (вложенные подменю `Создать на основании`▸/`Печать`▸); фронт `MENU_ITEM` плоский, cascading не умеет. — **Ответ:** принято. Проверен `button-node.tsx`: `<Menu>` рендерит детей через `NodeRenderer` → секции достижимы через `SEPARATOR`/`LABEL`-узлы (не cascading). «Ещё» = плоский список с секциями-`SEPARATOR` (§3.4.4); полный иерархический паритет — будущая фаза, **явно** зафиксировано как сознательное упрощение. — **Статус: closed.**
14. **Возр. 2 — дефолт pinned прячет доменное главное действие (ЭСФ.Отправить)** [major]: ИТС v8std/716 «командная панель должна позволять важнейшие действия **не открывая Ещё**»; доменные команды — видимые кнопки, не в «Ещё». — **Ответ:** принято со ссылкой. Дефолт «всё кроме записи/проведения/печати → Ещё» признан недостаточным; handler **явно** помечает доменное главное действие `pinned=true` (§3.4.2). — **Статус: closed.**
15. **Возр. 3 — `Help` как `notify`/`openDialog` — пустышка** [major]: `developer-1c` — `Help` = контент `HelpContent`/«Справочная информация»; **в БГУ КЗ поле пустое** (типовая не наполняет). — **Ответ:** принято. `Help` в Phase 1 **НЕ выводится** (`notify(help)` был бы пустышкой), но **классифицирован с обоснованием**: источника контента нет, отложен до статического help-контента на вебе (рекомендация `developer-1c`: markdown/HTML по типу, не миграция из 1С) — §3.5б. Не молчаливый выброс. — **Статус: closed.**
16. **Возр. 4 — `getCustomCommands` на 301 команду = та же per-type ручная работа, экономия композитора иллюзорна** [major]: composer генерит только гомогенный набор; «Ещё» (301 команда) пишется per-type в Java. — **Ответ:** принято фактически, тезис «иллюзорна» опровергнут конкретикой. §1.3 и §6 переписаны честно: composer экономит **только** гомогенный закреплённый набор (генерится 1×, не 85×); «Ещё» **не экономится**, остаётся per-type. Переезд SQL→Java оправдан типобезопасностью (id рядом с веткой обработки — снимает риск рассинхрона), тестируемостью, командой+обработкой в одном классе. — **Статус: closed.**
17. **Возр. 4-bis — `section?` в DTO vs плоский `MENU_ITEM`** [minor]: зачем поле, если фронт не группирует. — **Ответ:** `section?` живой — composer транслирует в `SEPARATOR`-узлы внутри `<Menu>` (фронт `SEPARATOR` рендерит); не cascading, а плоские секции. — **Статус: closed.**

### Раунд 4 — механизм источника пер-типового набора (data-driven сид vs Java per-handler)

После добавления §3.4.6/§3.4.7 (пер-типовая вариативность: 39 типов с кастомом, ~52 только костяк) проведён challenger round-trip по выбору механизма источника уникальной части. Исходная склонность автора — **гибрид** (декларация-данные + обработка-Java); challenger развернул на **B (Java per-handler)** по критериям самого ADR.

18. **Возр. 1 — «B держит id в одном месте, гибрид в двух» неверно** [major]: в B `id` тоже в двух местах (`getCustomCommands` + `handleCommand`-ветка), различие — не «одно vs два», а **расстояние** (B: два метода одного класса; гибрид: SQL-сид и Java в разных артефактах). — **Ответ:** принято. По метрике §6 «id рядом с веткой = снимает рассинхрон» B строго лучше: оба места в одном файле, под компилятором и unit-тестом. — **Статус: closed.**
19. **Возр. 2 — прецедент `addPrintCommands()` противоречит выбору данных** [major]: печать — уже Java per-handler в проде; печать и кастом функционально однородны (§3.4.1 в одном ряду); выбирать данные для кастома при Java для печати = внутреннее противоречие. — **Ответ:** принято. Единый Java-механизм (B) для всех команд панели; прецедент `addPrintCommands` — аргумент за B. — **Статус: closed.**
20. **Возр. 3 — «экономия гибрида против B» = ноль** [blocker для рекомендации гибрида]: обработка (`handleCommand`-ветки) пишется руками в **любом** варианте (§6 сам признал «Ещё не экономится»); разница — только язык декларации (39 Java-методов vs 39 групп SQL-строк), и §6 утверждает Java лучше → логика ADR ведёт к B. «Донаполнить без перекомпиляции» — фантом (рабочую кнопку без деплоя обработки не добавить; только id-пустышку). — **Ответ:** принято полностью. Рекомендация изменена с гибрида на **B** (§3.4.7). Гибрид/A оправдан лишь при **незаявленной** потребности runtime-кастомизации панели — оставлен как layout-override escape hatch поверх B, не основной механизм. — **Статус: closed.**
21. **Возр. 4 — гибрид/A тащит CI-гейт из Phase 2 в Phase 1** [major]: рассинхрон `id` (сид↔ветка) в данных компилятором не ловится → нужен гейт «id ⇒ ветка» уже в Phase 1; в B класс ошибок снимается без отдельного гейта (один класс, unit-тест, типобезопасность). — **Ответ:** принято; выбор B устраняет необходимость гейта в Phase 1 (§3.4.7); гейт остаётся условием Phase 2 (AUTO/`CommandRegistry`). — **Статус: closed.**

**Вердикт challenger раунда 4 — ⚠ SUPERSEDED раундом 5 (PM-директива владельца).** Раунд 4 выбрал B по критерию «инженерная экономия / compiler-safety». **Владелец проекта пересмотрел сам критерий** (раунд 5): не «что дешевле/безопаснее в коде», а «что honors server-driven-принцип (UI = данные)». При новом критерии раунд-4-аргументы за B (compiler-safety, единый Java-механизм) **переоценены** — они защищают инженерную выгоду, а не архитектурную цель, и заменяются CI-гейтом. **Решение §3.4.7 развёрнуто B → data-driven.**

### Раунд 5 — РАЗВОРОТ B → data-driven (PM-директива владельца)

**Фиксация разворота честно:** критерий выбора §3.4.7 изменён владельцем с «инженерная экономия/compiler-safety» на «server-driven-принцип (UI=данные, handler=только обработка)». При этом критерии выбор = **data-driven реестр `command_bar_items`**, а не B. Compiler-safety (главная выгода B) заменён CI-гейтом Phase 1. Принцип владельца принят как требование, не оспаривается; challenger атаковал только реализуемость новой модели.

**Переоценка прежних раунд-4-возражений за B:**
- Возр. 18 (B: id рядом с веткой) — **больше не решающий**: близость декларации к обработке — инженерная выгода, проигрывающая архитектурному требованию «UI=данные». Рассинхрон закрывается CI-гейтом, не компилятором.
- Возр. 19 (прецедент `addPrintCommands` за единый Java) — **переоценён**: печать теперь — **явное обоснованное исключение** (UI+ресурс-шаблон+платформенный прообраз БСП, §3.4.7/возр.в ниже), а не аргумент за Java-кастом.
- Возр. 20/21 (экономия гибрида/гейт в Phase 1) — **сняты разворотом**: гейт в Phase 1 теперь принят как явная плата.

22. **Возр. а — CI-гейт как замена compiler-safety: источник истины** [blocker, процедурный]: гейт на сиде/тест-БД ≠ прод; ловит 1 связь vs компилятор всю схему; orphan = warn слабо. — **Ответ:** принято. Гейт читает **прод-сид-миграцию** (иммутабельный накатываемый артефакт, ручные правки строк запрещены — как layout §1.1) → зелёный гейт = прод корректен; гейт расширен до **валидации схемы строк** (типы/enum-домены) + **транслитерации `commandId`**; **orphan server-ветка = error**, не warn (§3.4.7). — **Статус: closed.**
23. **Возр. б — `visibleWhen`/`enabledWhen`: утечка UI обратно в код, серая зона** [major]: формальной границы «простое vs сложное» нет (`developer-1c`: в 1С условной видимости по реквизиту нет декларативно — всё код); перекрытие с §3.2-патчами на классе «видимость по `isPosted`». — **Ответ:** принято, **строгий вариант**: `visibleWhen`/`enabledWhen` в Phase 1 **НЕ вводим**; любая зависимость видимости/доступности от состояния — **только** `setProp`-патч handler'а (§3.2). Шов однозначен: статика→данные, зависимость-от-состояния→код-патч. Декларативные условия — Phase 2 с формально перечислимым набором предикатов. — **Статус: closed.**
24. **Возр. в — печать в Java при кастоме в данных = противоречие принципу** [major]: тезис «печать динамична» в коде ложен (`addPrintCommands()` без параметра entry — статична, реестр потянул бы). — **Ответ:** принято, тезис исправлен. Печать — **явное обоснованное исключение** (§3.4.7) по трём причинам: платформенный прообраз 1С (печать = код БСП), исполняемая ресурс-привязка (`getTemplateName`/XLSX→PDF — UI+ресурс, не чистая декларация), будущая рантайм-динамика (права/состояние/подключаемые формы) выразима только кодом. Записано как исключение, не замалчивается. — **Статус: closed.**
25. **Возр. г — разрез декларация↔обработка по границе данные↔код воссоздаёт рассинхрон-риск + ADR противоречит сам себе (выбирает B)** [blocker, процедурный]: §3.4.7/§6/§8-р4 утверждали B — конфликт с директивой; транслит `commandId` теперь в данных И коде = расширенный фронт «молчаливо несработавшего хендлера». — **Ответ:** принято. §3.4.7, §6, §8-р4 переписаны под директиву (B → data-driven, вердикт р4 помечен superseded); рассинхрон-риск **зафиксирован как осознанно принятая плата** за server-driven-принцип; CI-гейт усилен **транслит-проверкой `commandId`** (как `TransliterationConsistencyTest`). — **Статус: closed.**

**Вердикт раунда 5:** 4 возражения (2 blocker процедурных, 2 major) — все закрыты переписью §3.4.7/§6/§7/§8 и усилением CI-гейта. Оба blocker были **не против принципа владельца** (challenger подтвердил: принцип не нарушается), а против незакрытых реализационных дыр (источник гейта, внутреннее противоречие текста) — закрыты. Эскалация PM не требуется: разворот — директива владельца, реализационные возражения учтены в тексте.

**Итог диспута:** 25 возражений за 5 раундов (7 blocker, 13 major, 5 minor) — все закрыты: правкой scope/фазировки/охвата/механизма, 2 опровергнуты проверкой кода, 3 закрыты первоисточниками 1С (ИТС v8std/716, БСП, `developer-1c` по `HelpContent`/условной видимости, `analyst-1c` по двухслойному командному интерфейсу). Финальная архитектура: (1) **scope = верхняя панель `OBJECT_FORM`** (документы Phase 1a + карточка справочника Phase 1b при готовности ADR-0009 Phase 2); (2) **костяк = фикс. правила composer'а** + рантайм-gating патчами (`CommandRegistry`/capability — Phase 2); (3) **полный охват через «Ещё ⋯»** (dropdown + секции-`SEPARATOR`); (4) **пер-типовая вариативность = СТРОКИ реестра метаданных на тип** (§3.4.6); (5) **механизм уникальной части — DATA-DRIVEN реестр `command_bar_items`** (server-driven-принцип владельца, §3.4.7); handler = только обработка (`handleCommand` + `getHandledCommandIds`); compiler-safety заменён **CI-гейтом Phase 1** (источник=прод-сид, схема, транслитерация, orphan=error); печать — обоснованное Java-исключение; (6) **доменные действия `pinned`** (ИТС); (7) **client-команды классифицированы** (`Help` отложен). Граница код/данные: **статика→метаданные, поведение/реакции→handler-патчи** (`visibleWhen` не в Phase 1). Осознанно принятая плата: рассинхрон данные↔код, закрытый CI-гейтом. Scope-cuts: cascading-подменю, `Создать на основании`, `Help`/`CustomizeForm`/`SaveAs`, декларативные условия видимости — будущие фазы/ADR.
