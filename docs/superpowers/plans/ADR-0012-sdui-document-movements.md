## 1. Контекст

### 1.1 Что есть сейчас

**Срез «Дт/Кт» / «Движения документа»** в 1С — read-only форма, показывающая, что документ записал в регистры при проведении (`ОбработкаПроведения`). Это не журнал регистра, а **фильтр по регистратору** (текущему документу).

В SDUI это сегодня реализовано **только у пилота ГП** и **императивно в Java**, что и есть нарушитель, устраняемый этим ADR.

**Нарушитель — `ZayavkaNaRegistratsiyuGpSdelkiViewFormHandler` (`view/handler/zayavkanaregistratsiyugpsdelki/`):**
- `handleOpen` инжектит кнопку через `patches.insertNode("commandBar", 4, buildDtKtButton())` — `BUTTON{label:"Дт/Кт", command:"showDtKt", icon:"dtkt"}` (строки 286–288, 445–461) — per-type хардкод кнопки в коде хендлера;
- `handleCommand` ловит `showDtKt` → `handleShowDtKt` (381–427) — берёт **только** регистр бухгалтерии через `accountingRegisterService.findByRecorder(id)`, форматирует строки в синтетический binding `dtkt.rows`;
- `buildDtKtDialog()` (468–507) **императивно строит дерево** `PAGE → TABLE(binding="dtkt.rows") → 4×TABLE_COLUMN` (Период/Счёт/Сумма/Содержание) прямо в Java, отдаёт через `effects.openDialog(...)`.

**Это антипод server-driven:** разметка диалога, набор колонок, единственная вкладка (бухгалтерия), кнопка — всё захардкожено в Java одного типа. ~90 остальных типов ДтКт/движений **не получают вовсе**.

### 1.2 КЛЮЧЕВАЯ НАХОДКА — generic-сервис данных уже есть

`GET /api/document-entries/{id}/movements` → `DocumentMovementsByKindService.getMovements(id)` (`document/recorder/service/`) → `DocumentMovementsByKindDto`:
- `Long documentEntryId`;
- `AccountingMovementsGroupDto accounting` — **null**, если документ не писал в регистр бухгалтерии; иначе `{registerTypeCode, registerTypeNameRu, registerTypeNameKz, List<AccountingMovementEntryDto> entries}`. Строка проводки — **структурированный DTO** с резолвленными субконто (`subkontosDt/Kt: List<AccountingRegisterSubkontoDto>{position, kindRu, displayName, referenceId, code}`), `accountDtCode/accountKtCode`, `summa`, `kolichestvoDt/Kt` (+ схлопнутое `kolichestvo`), `valyutnayaSummaDt/Kt`, `soderzhanie`, `isActive`, `period`, `lineNo`, `id`;
- `List<AccumulationMovementsGroupDto> accumulation` — по одной группе на тип регистра накопления; строки — **EAV-формат** `List<Map<String,Object>>`: служебные ключи `_id/_period/_movementKind/_isActive` + `{attributeCode → value}` (значение сериализовано `valueFieldHelper.getObjectAsDto`, для ссылок — `{id, displayName}`);
- `List<InformationMovementsGroupDto> information` — аналогично, без `_movementKind`.

Сервис **уже** группирует движения по виду регистра, отбрасывает пустые виды (null/`List.of()`), резолвит метаданные типа регистра (`registerTypeNameRu/Kz`) и имена субконто (тот же `mapSubkontos` + `resolveSubkontoName`, что в журнале проводок). **Это фундамент данных** — composer строит дерево из НЕГО, а не из прямого `findByRecorder` per-type.

### 1.3 Метаданные «какие регистры пишет документ»

`document/handler/kbp/KbpRegisterBindings.java` — `Map<docTypeCode, List<RegBinding>>` (~162 биндинга после ADR-0008, partial). Это слой **объявленных** регистраторов (РН/РС; бухрегистр там НЕ перечислен — он отдельно, через `onPost`). **Истина состава вкладок — НЕ это**, а фактические непустые движения (§3.6). `KbpRegisterBindings` — вспомогательно (можно показать «объявлен, но пуст» как диагностику; в Phase 1 не используем для состава).

### 1.4 Что умеет фронт (проверено по fin-web)

| Узел | Факт | Источник |
|---|---|---|
| `TABS`/`TAB` | `tabs-node.tsx` рендерит вкладки, скрывает `props.visible===false`, заголовок из `props.title ?? props.label`; контент = `activeTab.children` через `NodeRenderer` | `nodes/layout/tabs-node.tsx` |
| `TABLE` | `table-node.tsx` — read-only грид: строки `getValue(node.binding)` (массив `{rowId, [binding]:val}`), колонки = `TABLE_COLUMN`(props `label`/`binding`/`flex`), ячейка = `String(row[binding] ?? '')` | `nodes/composite/table-node.tsx` |
| `TABLE_COLUMN` | `label`/`binding`/`flex` | `nodes/composite/table-column-node.tsx` |
| `openDialog`/`closeDialog`/`notify` | есть; корень поддерева несёт `props.presentation: 'drawer'\|'modal'` (ADR-0009, default modal) | `lib/effect-handler.ts`, `PanelHost` |
| ячейка-ссылка проваливание | **НЕ умеет** — ячейка = `String(...)`, не кликабельна | `table-node.tsx` |
| визуал неактивной строки | **НЕ умеет** — нет признака isActive у строки | `table-node.tsx` |

**Вывод (уточнён, challenger major-2):** read-only рендер **структуры** вкладок+таблиц фронт уже умеет (новых `NodeType` = 0). **НО** `table-node.tsx:113` рисует ячейку как `String(row[binding] ?? '')` — это ломает **ссылочные** ячейки (`{id,presentation,entityRef}` → `[object Object]`). Поэтому Phase 1 распадается так:
- **примитивные/датовые/булевы/денежные ячейки — форматирует СЕРВЕР** (composer отдаёт строку: `period`→`dd.MM.yyyy HH:mm`, `summa`→десятичная строка, `isActive`→«Да»/«—»); фронт `String(value)` показывает их корректно **без правок**;
- **ссылочные ячейки — малая фронт-дельта Phase 1** (~5 строк в `table-node.tsx`): если значение — объект с `presentation`, показать `value.presentation`. Без неё субконто/измерения отрисуются как `[object Object]` (регресс к пилоту).

Честно: **фронт-дельта Phase 1 НЕ ноль** — она минимальна (одна нормализация объектной ячейки), но необходима для ссылочных колонок. `entityRef` едет в данных Phase 1 (forward-compatible), но **интерактивится** (клик-навигация) только Phase 2. Визуал неактивных строк — Phase 2 (флаг `_isActive` в данных Phase 1). Перечень — frontend-spec §3, выбор зафиксирован здесь (не делегируется фронт-команде).

### 1.5 Проблема

Per-type императивный диалог ДтКт (§1.1) не масштабируется и нарушает server-driven-инвариант: разметка/колонки/состав вкладок описаны Java-кодом одного хендлера; бухрегистр — единственный показанный вид; ~90 типов не покрыты. Нужен **generic-композитор движений**, строящий дерево вкладок+таблиц из `DocumentMovementsByKindService` для **любого** документа, без знания конкретного типа.

### 1.6 Что НЕ переоткрывается

Stateful form-session, scratch, revision (ADR-0006); ссылочные значения `{id,presentation}` и `presentation=drawer|modal` (ADR-0009); командная панель и реестр `command_bar_items` (ADR-0010); read-only рендер TABLE (ADR-0011). Этот ADR **достраивает движения** в рамках этих инвариантов.

## 2. Охват (scope)

**В охвате:** read-only представление «Движения документа» (вкладки = непустые регистры, таблицы строк) + срез «Дт/Кт» (та же форма с дефолтной вкладкой бухгалтерии) для **проведённого** документа `OBJECT_FORM`, generic для **всех ~91 типов**.

**Вне охвата (будущее/отдельные ADR):**
- запись движений (`onPost`, проводки) — это бизнес-логика хендлеров, не этот ADR;
- редактирование движений — в 1С движения не правятся напрямую (только перепроведением); таблицы движений **read-only** (≠ ADR-0011);
- «Структура подчинённости», настройка списка движений, группировка, печать движений, отбор/сортировка пользователем — out-of-scope;
- проваливание (навигация в карточку EAV-записи субконто/счёта/измерения) и «Обновить» — фазируются (§5);
- журнал регистра целиком (не по регистратору) — это отдельный экран (как `LIST_FORM`, вне SDUI-сессии).

## 3. Решение

**Принцип (PM-директива владельца, тот же, что ADR-0010 §3.4.7 и ADR-0011 §2.5):** построение, разметка и все взаимодействия ДтКт/движений — **на бэке**. Состав вкладок = фактические непустые движения; колонки/заголовки = **данные движений + метаданные регистров**, не Java конкретного хендлера. Фронт = **чистый рендерер** (read-only).

### 3.1 `MovementsComposer` — generic-композитор движений

Новый `MovementsComposer` (пакет `view/composer/`) принимает `Long documentEntryId`, зовёт `DocumentMovementsByKindService.getMovements(id)` и строит дерево:

```
PAGE (props.presentation = modal|drawer, title = "Движения документа")
└── TABS  (id = "movements.tabs")
    ├── TAB (бухгалтерия — ПЕРВОЙ, если accounting != null)
    │   └── TABLE (binding = "movements.acc.<regCode>", колонки = каркас бухрегистра §4.1)
    ├── TAB (накопление — по одной на непустую AccumulationMovementsGroupDto)
    │   └── TABLE (binding = "movements.accum.<regCode>", колонки из метаданных типа §4.2)
    └── TAB (сведения — по одной на непустую InformationMovementsGroupDto)
        └── TABLE (binding = "movements.info.<regCode>", колонки из метаданных типа §4.2)
```

Строки каждой таблицы кладутся в view-state под `binding` таблицы (через `setValue(binding, rows)` на ответе команды — паттерн `dtkt.rows`, но **не синтетический per-type**, а единообразный для всех). Composer **не знает тип документа** — работает по `DocumentMovementsByKindDto`.

**Заголовок вкладки** = `registerTypeNameRu` (Kz по языку сессии) — серверные метаданные типа регистра, НЕ хардкод. Для бухгалтерии — `registerTypeNameRu` (обычно «Журнал проводок»).

**Сериализация ссылок** — `{id, presentation}` как в [[sdui-render-contract]]/ADR-0009 (НЕ DTO): ячейка-ссылка несёт `entityRef{domain, id, presentation}` для проваливания (§5). Аккумуляция/сведения уже отдают `{id, displayName}` из `getObjectAsDto` — composer переименует `displayName→presentation`, субконто резолвятся `AccountingRegisterMapper` (`kindRu`/`displayName`/`referenceId`).

> **`entityRef.typeCode` — Phase 1 НЕ кладёт (challenger major-1, решено).** Источник данных движений (`AccountingRegisterSubkontoDto`, `getObjectAsDto`) несёт `domain`(=`valueType`)/`referenceId`/`displayName`, **но НЕ конкретный typeCode** записи (DICTIONARY говорит «справочник», не какой именно — Kontragenty/Sklady). Резолв конкретного typeCode по `referenceId` = **правка слоя данных**, которая выходит за «composer = тонкий маппинг». Решение: **Phase 1 кладёт `entityRef{domain, id, presentation}` без `typeCode`** — этого достаточно для forward-compatibility (domain+id стабильны). `typeCode` добавляется **вместе с навигацией в Phase 2** (резолв типа по `referenceId` — отдельная задача слоя данных, §5/§7). Контракт Phase 1 НЕ обещает поля без источника.

### 3.2 Композиция = данные/метаданные + факт движений, НЕ Java хендлера

Граница (та же, что ADR-0010/0011 — сведена в §8):
- **состав вкладок** = фактические непустые группы из `DocumentMovementsByKindDto` (накопление/сведения уже сгруппированы и отфильтрованы сервисом; accounting=null отбрасывается);
- **порядок вкладок** = детерминированное правило composer'а: бухгалтерия → накопления → сведения; внутри вида — по `registerTypeNameRu` (сервис уже отдаёт в стабильном `LinkedHashMap`-порядке);
- **колонки накопления/сведений** = из **метаданных типа регистра** (его атрибуты: измерения + ресурсы), + `ВидДвижения` для накопления (§4.2);
- **каркас колонок бухрегистра** = **фиксированные детерминированные правила composer'а** (Период/СчётДт/СубконтоДт1..N/СчётКт/СубконтоКт1..N/Сумма/[Кол-во]/[Вал.сумма]/Содержание/Активность), с серверным резолвом заголовков субконто и условным показом количественных/валютных колонок (§4.1).

Хендлер конкретного типа в построении дерева движений **не участвует вообще** (в отличие от §1.1).

### 3.3 ДтКт-кнопка — GENERIC через реестр командной панели ADR-0010

**Кнопка НЕ инжектится в хендлере** (`buildDtKtButton`/`insertNode("commandBar"...)` сносится). Вместо этого:
- команда `showMovements` (открыть форму движений; срез ДтКт = та же форма с дефолтной вкладкой бухгалтерии) — **стандартная команда костяка проведённого документа** в `CommandBarComposer` (ADR-0010 §3.4.6, «костяк = фиксированные правила»). Composer добавляет её в командную панель **всем** документам, но `handler.handleOpen`/`CommandBarComposer` гасит/скрывает её, пока документ **не проведён** (паритет с динамикой `setProp`, ADR-0010 §3.2);
- обработка `showMovements` — **generic-ветка `ViewController.handleCommand`** (не per-handler): зовёт `MovementsComposer.compose(session.getDocumentEntryId())` → `effects.openDialog(tree)` (или `closeDialog`/`notify` при отсутствии движений, §3.6). Если движений нет → `notify("warning", "Документ не проведён / нет движений")` (паритет с текущим поведением).

**Совместимость:** legacy-команда `showDtKt` сохраняется как алиас `showMovements` с дефолтной активной вкладкой бухгалтерии (generic-ветка обрабатывает обе). Это снимает регресс пилота без правки фронта.

`showMovements`/`showDtKt` — безаргументны (`BUTTON` шлёт `{type:COMMAND, command}` без value — ADR-0010 §1.2), id документа берётся из `session` — фронт не трогаем.

### 3.4 Контракт ответа — узлы

Composer возвращает поддерево из **существующих** `NodeType` (новых не вводим):

| Узел | Назначение | props |
|---|---|---|
| `PAGE` | корень диалога | `title="Движения документа"`, `presentation=modal` (Phase 1; drawer — опционально) |
| `TABS` | контейнер вкладок | `id="movements.tabs"` |
| `TAB` | одна вкладка = один регистр | `title=<registerTypeNameRu>`, `regKind=ACCOUNTING\|ACCUMULATION\|INFORMATION`, `regCode` |
| `TABLE` | строки регистра (read-only) | `binding="movements.<kind>.<regCode>"`, `editable=false`, `readOnly=true` |
| `TABLE_COLUMN` | колонка | `label`, `binding`(=ключ в строке), `dataType?`, `align?`, `navigable?` (см. §5) |

Строки — read-only массив `{rowId, [binding]:value, _isActive}`:
- `rowId` = `String(_id)` записи регистра (стабилен; правок нет, identity только для рендера);
- ссылочные ячейки несут `entityRef{domain, id, presentation}` (Phase 1 — без `typeCode`, см. §3.1; фронт показывает `presentation`); навигация — Phase 2;
- `_isActive` (boolean) — флаг строки для визуального отличия неактивных (сторно) движений (Phase 2-визуал; Phase 1 показываем все строки без отличия, не скрываем);
- **форматирование значений ячеек — на сервере** (фронт = чистый рендерер, `String(value)`): `period` (LocalDateTime) сериализуется composer'ом как `dd.MM.yyyy HH:mm` (паритет с прежним `buildDtKtDialog`, чтобы не было регресса даты); `summa`/`kolichestvo`/`valyutnaya*` — строкой с десятичным форматом; `isActive` → значение колонки «Активность» — локализованный текст («Да»/«—» или галочка-символ), не сырой boolean. См. §8 (форматирование — серверные правила).

### 3.5 Взаимодействия — все на бэке

| Взаимодействие | Механизм | Фаза |
|---|---|---|
| открыть | COMMAND `showMovements` → generic-ветка → `effect openDialog(tree)` | **Phase 1** |
| закрыть | `effect closeDialog` (фронт уже умеет; кнопка закрытия диалога) | **Phase 1** |
| показать только бух. (ДтКт) | та же форма, дефолтная активная вкладка — бухгалтерия (первая) | **Phase 1** |
| нет движений | `notify("warning", ...)` вместо диалога | **Phase 1** |
| обновить | COMMAND `refreshMovements` → composer пересобирает дерево → `setValue`-патчи строк (или повторный `openDialog`) | **Phase 2** |
| проваливание (клик по `entityRef`-ячейке) | COMMAND `navigate:<domain>:<typeCode>:<id>` → бэк отдаёт `navigate`/`openDialog` карточки записи | **Phase 2** |

**Никакой клиентской логики состава/форматирования/навигации.** Фронт: рендерит присланное дерево, шлёт COMMAND (открыть/обновить/проваливание), применяет эффекты/патчи.

### 3.6 Динамический состав вкладок, порядок, isActive, субконто

1. **Только непустые регистры.** Сервис уже отбрасывает пустые виды (accounting=null, accumulation/information=`List.of()`). Composer добавляет `TAB` только для непустых групп. `KbpRegisterBindings` (объявленные регистры) для состава **не используется** — истина = факт движений (analyst-1c п.1).
2. **Заголовки субконто резолвятся на бэке** (analyst-1c п.2): `AccountingRegisterSubkontoDto` несёт `kindRu` (вид субконто из плана видов характеристик) и `position`. **Число** колонок субконто = максимум субконто на сторону среди строк регистра (или `maxSubkontoCount` типа). **Заголовок колонки субконто `N` — generic «Субконто Дт N» / «Субконто Кт N» по `position`** (challenger minor-3, решено): в одной выборке проводок разные счета имеют **разные виды субконто на одной позиции** (1311 поз.1 = Контрагенты, 1010 поз.1 = Склады) — единого `kindRu` на колонку нет, это **штатный** случай, а не fallback. Поэтому колонка-заголовок всегда generic; конкретный вид (`kindRu`) показывается на уровне **ячейки** (как часть `presentation`/tooltip), не колонки — ровно как окно «Движения документа» в 1С. Композитор может усилить: если у ВСЕХ непустых строк на позиции `N` один `kindRu` — взять его заголовком, иначе generic (детерминированно).
3. **isActive** (analyst-1c п.3): неактивные движения **показываются** (не скрываются), помечаются `_isActive=false` для визуального отличия (Phase 2). Phase 1 — без визуала, но флаг в данных уже есть.
4. **Порядок** (analyst-1c п.4): бух → накопл → сведения; внутри — стабильный порядок сервиса (`LinkedHashMap` по типу).
5. **Ссылочные ячейки навигируемы** (analyst-1c п.5): `entityRef{domain, typeCode, id, presentation}`; проваливание — Phase 2 (§5).

## 4. Колонки — граница метаданные/правила композитора

### 4.1 Каркас бухрегистра — фиксированные правила composer'а (как «костяк» кнопок ADR-0010)

**Решение:** колонки регистра бухгалтерии — **фиксированный детерминированный каркас в `MovementsComposer`**, НЕ строки метаданных. Обоснование (симметрично «костяку» командной панели ADR-0010 §3.4.6 и «каркасу бухрегистра журнала проводок»):

1. **Структура проводки 1С — платформенный инвариант, единый для всех планов счетов:** Период / СчётДт / СубконтоДт1..3 / СчётКт / СубконтоКт1..3 / Сумма / [Количество] / [Вал.сумма] / Содержание / Активность. Это не «UI на тип документа», а **фиксированная семантика регистра бухгалтерии** — она едина для всех ~91 типа, как save/post — единый костяк панели. Заведение её строками метаданных на каждый регистр = дублирование инварианта без вариативности.
2. `AccountingMovementEntryDto` — **структурированный DTO** (не EAV-Map): поля жёсткие (accountDtCode, subkontosDt, summa…). Каркас читает эти поля детерминированно — это функция от данных DTO, не «Java определяет UI на тип».
3. **Условный показ** количественных (`kolichestvo` — если хоть у одной строки `kolichestvoDt/Kt != null`) и валютных (`valyutnayaSumma*`) колонок — **детерминированное правило по данным** (показываем колонку, если в выборке есть ненулевые значения), не per-type хардкод.
4. **Заголовки субконто** резолвятся из `kindRu` (метаданные плана видов характеристик) с fallback «Субконто Дт N» — серверный резолв, не Java-литералы на тип.

**Это НЕ возврат хардкода UI в код** (challenger-вопрос а): каркас бухрегистра — единое правило для всех типов (вариативности по типу документа нет), параметризованное **данными** (какие субконто, есть ли количество/валюта). Аналогия точная: ADR-0010 держит костяк save/post фиксированными правилами composer'а, а не строками реестра, ровно потому что он гомогенен. Бухкаркас гомогенен так же.

### 4.2 Накопление/сведения — колонки из метаданных типа регистра

Колонки регистров накопления/сведений = **атрибуты типа регистра** (измерения + ресурсы), читаемые из метаданных (`AccumulationRegisterType.attributes` / `InformationRegisterType.attributes` — `code`, `nameRu/Kz`, `dataType`, `sortOrder`). Это **данные**, не код:
- composer грузит атрибуты типа регистра (по `registerTypeCode` группы), формирует `TABLE_COLUMN` на каждый (label=`nameRu`, binding=`attr.code`, по `sortOrder`);
- для накопления — добавляется системная колонка `ВидДвижения` (Приход/Расход) из `_movementKind` (RECEIPT/EXPENSE → локализованный label);
- системные колонки: `Период` (`_period`), `Активность` (`_isActive`);
- строки EAV-Map уже keyed по `attr.code` (сервис кладёт `row.put(attr.getCode(), ...)`) — ключ колонки ≡ ключ в строке by construction.

**Почему по-разному (бух = правила, накопл/свед = метаданные):** бухрегистр имеет **единую фиксированную структуру** для всех типов (нет вариативности → правило); накопление/сведения имеют **разный набор измерений/ресурсов на тип регистра** (есть вариативность → метаданные типа). Это honors дисциплину «структура=данные/метаданные, не хардкод per-type»: вариативная часть — данные; инвариантная — правило. Шов совпадает с ADR-0010 (костяк=правила, кастом=реестр) и ADR-0011 (колонки ТЧ=layout-метаданные).

## 5. Проваливание и обновление — фазирование (challenger-вопрос в)

**Phase 1 — read-only без интерактива сверх открыть/закрыть:**
- открыть форму движений (вкладки+таблицы), закрыть, дефолтная вкладка бухгалтерии для ДтКт, notify при отсутствии движений.
- **малая фронт-дельта Phase 1:** нормализация объектной ячейки (`{presentation}`→текст) — иначе ссылочные колонки = `[object Object]` (§1.4, challenger major-2); примитивы/дата/сумма/boolean форматирует сервер.
- ссылочные ячейки **показывают** `presentation` (как обычный текст) — `entityRef{domain,id,presentation}` уже едет в данных (без `typeCode`, §3.1), но фронт его пока не делает кликабельным;
- неактивные строки **показываются** (флаг `_isActive` в данных), без визуального отличия.

**Phase 2 — интерактив (фронт-дельта):**
- **резолв `typeCode` записи** (challenger major-1): слой данных (`AccountingRegisterMapper`/`getObjectAsDto`) дополняется конкретным typeCode по `referenceId`/`valueType` → `entityRef.typeCode`. Это правка слоя данных, явно в Phase 2 (не в «тонком маппинге» composer'а Phase 1);
- **проваливание:** клик по `navigable`-ячейке → COMMAND `navigate:<domain>:<typeCode>:<id>` → бэк отдаёт `navigate` (на карточку EAV-записи) или `openDialog` (карточка во вложенном диалоге). **Логика навигации — на бэке** (резолв «куда вести» по domain/typeCode — серверная); фронт лишь шлёт клик и применяет эффект. Никакой клиентской логики «как открыть карточку»;
- **визуал неактивных строк:** фронт читает `row._isActive===false` → приглушённый/зачёркнутый стиль;
- **обновить:** кнопка «Обновить» в диалоге → COMMAND `refreshMovements` → пересборка дерева.

**Почему проваливание — Phase 2, и почему логика не утечёт на фронт:** проваливание требует фронт-дельты (кликабельная ячейка + dispatch) — нарушать «фронт фиксирован» в Phase 1 не нужно для основной ценности (показать движения). Когда дельта вносится, навигация остаётся server-driven: фронт шлёт `{domain,typeCode,id}` из `entityRef`, бэк решает целевой эффект. `entityRef` кладётся в данные **уже в Phase 1** (forward-compatible) — Phase 2 включает только рендер-поведение, не меняя контракт данных.

## 8. Граница ДАННЫЕ ↔ КОД (сведена; согласована с ADR-0010 §3.4.7 / ADR-0011 §2.5.3)

| Аспект | Источник | Параллель |
|---|---|---|
| **Состав вкладок** (какие регистры) | **факт движений** (`DocumentMovementsByKindDto` — непустые группы) | ADR-0011 строки из EAV |
| **Заголовки вкладок** | **метаданные типа регистра** (`registerTypeNameRu/Kz`) | ADR-0010 label из реестра |

| **Каркас колонок бухрегистра** | **фиксированные правила composer'а** (инвариант структуры проводки 1С, един для всех типов), параметризованные данными (субконто/кол-во/валюта) | ADR-0010 костяк save/post = правила composer'а |
| **Порядок вкладок/колонок** | **детерминированные серверные правила** (бух→накопл→свед; внутри — стаб. порядок сервиса/`sortOrder`) | ADR-0010 orderIndex |
| **Резолв заголовков субконто** | **серверные правила** (`kindRu` + fallback «Субконто Дт N») | — |
| **Форматирование значений ячеек** (дата/сумма/кол-во/boolean/ссылка) | **серверные правила** composer'а (фронт = `String(value)`): `period`→`dd.MM.yyyy HH:mm`, `summa`→дес. строка, `isActive`→текст, ссылка→`{id,presentation,entityRef}` | ADR-0009/0011 (фронт не форматирует) |
| **Навигация проваливания** (Phase 2) | **серверный резолв** (по domain/typeCode → эффект) | ADR-0009 ref.open |
| **Запись движений** (`onPost`) | **код хендлера** — отдельная тема, НЕ здесь | — |

**Двойного стандарта нет:** инвариантная структура (бухкаркас) — правило composer'а (как костяк кнопок); вариативная (колонки накопл/свед) — метаданные (как кастом-реестр/layout-колонки); поведение/навигация — серверный код (как handleCommand). Никакой части композиции не описывает Java конкретного хендлера документа.

**CI-гейт-аналог (опционально, по образцу ADR-0010 «orphan=error» / ADR-0011 round-trip):**
- каждая колонка накопл/свед (`TABLE_COLUMN.binding`) ↔ существующий атрибут типа регистра (`attr.code`) — by construction (composer берёт колонки ИЗ атрибутов), но тест фиксирует, что ключ строки совпадает с binding;
- транслитерация `registerTypeCode`/кодов колонок — по `docs/project/transliteration-1C-java.md` (`TransliterationConsistencyTest`).

## 9. Последствия

**Выигрываем:**
- **По-настоящему server-driven движения:** вкладки/колонки/заголовки = данные движений + метаданные регистров; ни один хендлер не строит дерево в Java.
- **Generic для всех ~91 типа** — ДтКт/движения «бесплатно», без per-type кода; устранён хардкод пилота.
- **Источник данных — существующий `DocumentMovementsByKindService`** (Phase 1 не пишет новый слой данных): composer = тонкий маппинг DTO→узлы. *(Phase 2-резолв `entityRef.typeCode` по `referenceId` — единственное добавление слоя данных, явно отложено, §5.)*
- **Фронт-дельта Phase 1 минимальна** (новых `NodeType` = 0; TABS/TAB/TABLE рендерятся read-only). Честно (challenger major-2): **НЕ ноль** — нужна одна нормализация объектной ячейки (`{presentation}`→текст, ~5 строк), иначе ссылочные колонки = `[object Object]`; примитивы/дата/сумма/boolean форматирует сервер. Интерактив (проваливание/неактивные/typeCode) — изолирован в Phase 2.
- Шов «данные/метаданные/правила/код» согласован с ADR-0010/0011 — единая дисциплина.

**Платим (честно):**
- Каркас бухрегистра — фиксированные правила в Java (не метаданные). Осознанно: структура проводки — платформенный инвариant, вариативности по типу нет (§4.1; challenger-вопрос а закрыт).
- `entityRef`/`_isActive` едут в данных Phase 1, но интерактивятся только Phase 2 — небольшой forward-compatible оверхед в полезной нагрузке.
- Проваливание/обновление откладываются (Phase 2) — Phase 1 даёт просмотр, не навигацию.
- Новые `NodeProps`-константы (`regKind`/`regCode`/`navigable`/`entityRef`/`isActive`) + `MovementsComposer` + generic-ветка + правка костяка панели (ADR-0010).

**Отложено (с обоснованием):**
- проваливание — Phase 2 (требует фронт-дельты; не нужно для основной ценности; контракт данных forward-compatible);
- визуал неактивных — Phase 2 (фронт-дельта; флаг в данных уже есть);
- «Обновить» — Phase 2;
- `presentation=drawer` для движений — опционально (Phase 1 = modal, как текущий ДтКт);
- запись/печать/структура подчинённости — out-of-scope/будущее.

## 10. Альтернативы и контраргументы (challenger round-trip)

Round-trip по трём вопросам PM: (а) каркас бухрегистра правила-vs-метаданные; (б) динамический состав + резолв субконто на бэке; (в) проваливание Phase 1-vs-2.

### Альтернатива 1: каркас бухрегистра строками метаданных (как накопл/свед)
**Описание:** описать колонки бухрегистра строками `command_bar_items`-подобного реестра/layout на каждый регистр бухгалтерии, не правилами composer'а.
**Почему отклонена:** структура проводки 1С (Период/СчётДт/Субконто/СчётКт/Сумма/Содержание/Активность) — **платформенный инвариант, единый для всех ~91 типа** (вариативности по типу документа нет). Описывать инвариант метаданными = дублирование без выгоды; ADR-0010 ровно по этому критерию держит костяк save/post фиксированными правилами composer'а, а не реестром. `AccountingMovementEntryDto` — структурированный DTO (жёсткие поля), а не EAV-Map, поэтому каркас читает его детерминированно. Вариативность (есть ли количество/валюта/сколько субконто) параметризуется **данными выборки**, не кодом-на-тип.

### Альтернатива 2: состав вкладок из KbpRegisterBindings (объявленные регистры)
**Описание:** строить вкладки по `KbpRegisterBindings` (какие регистры документ объявил писать), а не по факту движений.
**Почему отклонена:** analyst-1c п.1 — показывать **только непустые** регистры (COUNT>0 для этого экземпляра). Объявленный, но фактически пустой регистр (документ проведён, но в этот РН ничего не записал) дал бы пустую вкладку — расхождение с 1С. `DocumentMovementsByKindService` уже отдаёт только непустые группы — истина состава = факт. `KbpRegisterBindings` остаётся вспомогательным (диагностика «объявлен, но пуст» — будущее), не источником состава.

### Альтернатива 3: проваливание в Phase 1
**Описание:** сразу сделать ячейки кликабельными с навигацией в карточку.
**Почему отклонена:** требует фронт-дельты (кликабельная ячейка + dispatch + рендер целевой карточки), а фронт фиксирован; основная ценность (просмотр движений) достигается без неё. Отложено в Phase 2 с forward-compatible контрактом: `entityRef{domain,typeCode,id,presentation}` едет в данных уже в Phase 1, навигация остаётся server-driven (бэк резолвит целевой эффект по domain/typeCode), логика не утекает на фронт.

### Контраргументы (challenger round-trip)

1. **Возражение (а)** [возврат хардкода UI]: «фиксированный каркас бухрегистра в Java = ровно то, что ADR устраняет у пилота (`buildDtKtDialog`)». — **Ответ автора:** разница принципиальная. `buildDtKtDialog` — **per-type** дерево в хендлере **одного** типа, показывающее **только** бухгалтерию через прямой `findByRecorder`, с захардкоженными 4 колонками-литералами. Каркас §4.1 — **единое правило composer'а для всех типов**, параметризованное данными (субконто из `kindRu`, кол-во/валюта по наличию значений), читающее структурированный `AccountingMovementEntryDto`. Критерий ADR-0010 («гомогенное → правило composer'а; вариативное → данные») применён буквально: бухструктура гомогенна по типам → правило; колонки накопл/свед вариативны → метаданные. — **Статус: closed.**

2. **Возражение (б)** [резолв субконто на бэке усложняет composer]: «заголовки субконто зависят от счёта (план видов характеристик) — composer должен знать план счетов». — **Ответ автора:** composer **не** резолвит план счетов сам — `DocumentMovementsByKindService`/`AccountingRegisterMapper.resolveSubkontoName` **уже** отдаёт `kindRu`/`kindCode` (вид субконто) и `displayName`/`code` (значение) в `AccountingRegisterSubkontoDto`. Composer лишь использует `kindRu` как заголовок (fallback «Субконто Дт N» по `position`). Резолв — на существующем слое данных, не в composer'е. — **Статус: closed.**

3. **Возражение (в)** [проваливание утечёт на фронт в Phase 2]: «когда добавите клик по ячейке, фронт начнёт сам решать, куда вести». — **Ответ автора:** контракт спроектирован против этого. `entityRef{domain,typeCode,id}` — это **данные** (бэк положил), фронт шлёт их назад в COMMAND `navigate:<domain>:<typeCode>:<id>` без интерпретации; **бэк** резолвит целевой эффект (`navigate`/`openDialog` карточки). Фронт знает только «эта ячейка кликабельна» (`navigable=true`) и «при клике отправить её entityRef» — это рендер-поведение, не логика навигации. Совпадает с ADR-0009 `ref.open` (фронт шлёт, бэк ведёт). — **Статус: closed.**

### Раунд 2 — challenger по трём точкам PM (entityRef, форматирование, заголовки субконто)

Challenger подтвердил по существу: точка (а) каркас бухрегистра — корректен, аналогия с ADR-0010 валидна, двойного стандарта нет; точка (б) состав вкладок (только непустые, отказ от `KbpRegisterBindings`) и механизм резолва субконто — корректны; точка (в) отсрочка проваливания и server-driven-навигация (фронт шлёт `entityRef`, бэк ведёт) — оправданы. Остались 2 major + 2 minor — все закрыты правкой:

4. **major-1 — `entityRef.typeCode` обещан в Phase 1, но источника в DTO нет** [major]: `AccountingRegisterSubkontoDto`/`getObjectAsDto` несут `domain`(=`valueType`)/`referenceId`/`displayName`, но НЕ конкретный typeCode (DICTIONARY ≠ какой справочник). «composer добавит typeCode» — без источника. — **Ответ автора:** принято полностью. **Phase 1 кладёт `entityRef{domain, id, presentation}` без `typeCode`** (§3.1) — этого хватает для forward-compatibility; резолв `typeCode` по `referenceId` = правка слоя данных, явно отложена в Phase 2 (§5). §9 скорректирован: Phase 1 — тонкий маппинг; Phase 2 — единственное добавление слоя данных. — **Статус: closed.**
5. **major-2 — «фронт 0 изменений» (§1.4/§9) vs обязательная дельта рендера ячейки** [major]: `table-node.tsx:113` = `String(row[binding])` → ссылочная ячейка = `[object Object]`, `period` (LocalDateTime) = ISO, `isActive` = «true» (регресс к пилоту, который форматировал `dd.MM.yyyy HH:mm`). Нельзя одновременно «0 фронта» И «entityRef едет Phase 1». — **Ответ автора:** принято. Выбор зафиксирован в ADR (не делегирован фронт-команде): примитивы/дата/сумма/boolean **форматирует сервер** (composer → строка), ссылочная ячейка = **малая фронт-дельта Phase 1** (~5 строк нормализации `{presentation}`). §1.4/§5/§9 переписаны: фронт-дельта Phase 1 минимальна, но НЕ ноль; форматирование period/summa/boolean — серверные правила (§8). — **Статус: closed.**
6. **minor-3 — заголовок субконто-колонки при разных видах субконто на одной позиции** [minor]: 1311 поз.1=Контрагенты, 1010 поз.1=Склады в одной выборке → единого `kindRu` на колонку нет. — **Ответ автора:** принято. §3.6 п.2 уточнён: заголовок колонки субконто `N` — **generic «Субконто Дт N»** (штатно, не fallback); конкретный `kindRu` — на уровне ячейки/tooltip (как в 1С). Опц. усиление: если у всех строк позиции один `kindRu` — взять его. — **Статус: closed.**
7. **minor-4 — форматирование `period`/`isActive`/`summa` не оговорено как серверное** [minor]: §8 покрывал ссылки, не дату/boolean/сумму. — **Ответ автора:** принято; §3.4/§4-row/§8 дополнены — period/summa/kolichestvo/isActive форматирует composer (фронт = чистый `String(value)`). — **Статус: closed.**

**Итог диспута:** 3 альтернативы отклонены с обоснованием; 3 контраргумента раунда 1 + 4 находки раунда 2 (2 major, 2 minor) — все закрыты. Ключевое: каркас бухрегистра — правило (инвариант), не хардкод-на-тип (критерий ADR-0010 применён буквально); состав = факт движений (не объявленные регистры); `entityRef` Phase 1 — `{domain,id,presentation}` без typeCode (typeCode + навигация — Phase 2); фронт-дельта Phase 1 минимальна, но честно НЕ ноль (нормализация ссылочной ячейки); форматирование значений — серверные правила; заголовки субконто-колонок — generic. Для перевода в Accepted (новый composer, граница данные/код, правка костяка ADR-0010) — финальное PM-ревью.
