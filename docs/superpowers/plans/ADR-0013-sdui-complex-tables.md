## 1. Контекст

### 1.1 Что наследуется (НЕ переоткрывается)

ADR-0011 закрепил редактируемую ТЧ: модель строк `[{rowId, colAttrCode:value}]`, ссылочные ячейки `{id,presentation}`, scratch-снимок (`tableCode→List<Map>`), table-level sync (весь массив одним EVENT), save через штатный `saveTableRows`, rowId-remap, **server-driven дисциплину** (структура=метаданные / поведение=код / фронт=рендерер, §2.5), cellWidget-override (§2.5.1), CI-гейт «orphan=error» (§2.5.5). Этот ADR **достраивает** сложные таблицы поверх той же механики — не меняет её.

### 1.2 Проблема — что в каталоге ТЧ требует расширения

Каталог пользователя (`tch_digest.md` + TSV, 77 док с ТЧ, 162 уник.ТЧ, 1721 поле):
- **580 полей в группах колонок, 283 уник.группы**, вложенность **до 2 уровней** — плоская модель `TABLE_COLUMN` (ADR-0011) этого не выражает. ТОП: ЭСФ→Товары (15 групп), КадровоеПеремещение→Сотрудники (10), НачислениеЗарплаты→Начисления (10).
- **79 полей с подвалом-итогом** — серверная строка-итог по колонке.
- **58 скрытых** (`Visible=false`) — явный состав отображаемых колонок.
- **188 read-only**; типы ячеек: Поле ввода (1498), Надпись (151), Флажок (67), Картинка (3) — переиспользуют cellWidget-override ADR-0011 §2.5.1.
- Документы с **двумя связанными ТЧ** (master-detail) — прувпойнт ИПН.

### 1.3 Принцип (тот же server-driven, ADR-0010/0011/0012)

Структура/группы/ориентация/видимость/подвал-флаг/формат/cellWidget = **серверные метаданные** (`layout_nodes`/`layout_node_props`+EAV), не Java per-type. Поведение (пересчёт подвалов, валидация, master-detail-семантика) = **код на бэке**. Фронт = **чистый рендерер**.

## 2. Решение — модель групп колонок

### 2.1 Контракт: новый `NodeType.COLUMN_GROUP` (а не вложенный TABLE_COLUMN)

Группа колонок — новый узел `COLUMN_GROUP` с дочерними `TABLE_COLUMN` (или вложенными `COLUMN_GROUP` для 2-го уровня). Обоснование выбора против «вложенный TABLE_COLUMN+флаг» — §10 Решение 1.

```jsonc
{
  "id": "colgroup.predostavlyatOsnovanie",
  "type": "COLUMN_GROUP",
  "props": {
    "label": "Предоставлять вычет / основание",   // общий заголовок группы (шапка)
    "orientation": "HORIZONTAL",                   // HORIZONTAL | VERTICAL (§2.2)
    "labelKz": "..."
  },
  "children": [
    { "id": "col.osnovanie", "type": "TABLE_COLUMN", "props": { "binding": "Osnovanie", ... } },
    { "id": "col.predostavlyat", "type": "TABLE_COLUMN", "props": { "binding": "PredostavlyatVychet", ... } }
  ]
}
```

- `COLUMN_GROUP` — дитя TABLE-узла, на месте, где раньше были плоские `TABLE_COLUMN`; порядок групп/колонок = порядок children (метаданные `order_index`).
- Вложенность ≤ 2 уровня (`COLUMN_GROUP` внутри `COLUMN_GROUP`) — подтверждена каталогом и developer-1c (практический предел; глубже — error CI-гейта §6).
- Источник — layout-метаданные: `layout_nodes` с `node_type=COLUMN_GROUP`, дочерние `TABLE_COLUMN`; props (`label`/`orientation`) — `layout_node_props`. **Ни один handler не объявляет группы в коде** (server-driven §2.5).

### 2.2 Ориентация группы — `orientation` в метаданных

| `orientation` | Рендер | 1С-аналог |
|---|---|---|
| `HORIZONTAL` (default) | под-колонки **рядом** под общим заголовком → многоуровневая шапка | `ГруппаКолонок` УФ (штатно) |
| `VERTICAL` | поля **друг под другом** в одной составной ячейке-колонке (одна визуальная колонка, в каждой строке — стопка полей) | обычные формы 8.1/8.2; **в УФ штатно НЕ поддерживается** |

**Критическое уточнение (analyst-1c + developer-1c, §9):** вертикальная группа колонок в ячейке ТЧ **в управляемых формах 1С платформой штатно не поддерживается** (была только в обычных формах 8.1/8.2; в УФ `ГруппаКолонок` = только горизонтальная шапка). Значит `orientation:VERTICAL` (составная ячейка «друг под другом») — это **сознательное расширение SDUI поверх 1С-возможностей**, не порт 1С-поведения.

**Решение (после challenger D2): прувпойнт ИПН по умолчанию — HORIZONTAL; VERTICAL — опциональное расширение, не в Phase 1 acceptance критического пути.** Требование «друг под другом» само под флагом «требует подтверждения владельца» (§3.2), а реальная УФ-1С кладёт {Osnovanie, PredostavlyatVychet} **горизонтально рядом** (§9). Вводить непроверенное-в-УФ поведение (VERTICAL, с проблемой commit-единицы D1) на критический путь непроверенного требования — преждевременно. Поэтому: **прувпойнт ИПН реализуется на HORIZONTAL** (две колонки рядом под общим заголовком — паритет с реальной формой, механика ADR-0011 без изменений). VERTICAL остаётся в контракте как опция (§2.3 определяет его commit-семантику), включается **сменой одного prop `orientation`** после подтверждения владельца + выверки analyst-1c — но не блокирует Phase 1.

Ориентация в xlsx явно не закодирована (каталог даёт только дерево групп) — задаётся метаданными группы; для конкретной типовой формы выверяется через analyst-1c.

### 2.3 Составная (вертикальная) ячейка — модель данных + commit-семантика (challenger D1)

Для `orientation:VERTICAL` строка по-прежнему `{rowId, Osnovanie:val, PredostavlyatVychet:val}` (плоские ключи колонок, как ADR-0011 §3.1) — **группировка чисто презентационная**. Фронт рисует обе колонки группы стопкой в одной визуальной ячейке; на провод/в scratch/в save идут **те же плоские ключи** — модель строк ADR-0011 **не меняется**. VERTICAL не вводит вложенности в данные строки, только в рендер.

**Commit-единица VERTICAL = ПОЛЕ (под-колонка), не визуальная ячейка (D1, переопределяет ADR-0011 §2.5.4).** В VERTICAL-ячейке два независимых редактора с **разными** commit-триггерами: `Osnovanie` (текст) — blur/Tab/Enter; `PredostavlyatVychet` (флажок) — клик. ADR-0011 §2.5.4 определял commit-единицу как «ячейку» в расчёте на один редактор; для VERTICAL это явно **переопределяется**:
- каждое под-поле коммитит **по своему триггеру** независимо (blur текста / клик флажка), оба эмитят table-level EVENT (весь массив, ADR-0011 §3.4) — это нормально, поля плоские в строке.
- **coalescing (ADR-0011 §2.5.4а п.3) расширяется на «несколько редакторов в визуальной ячейке»:** «активный редактор сохраняет ввод» применяется **по-полю**, не по визуальной ячейке. Сценарий «печатает Osnovanie, кликает флажок не уводя фокус»: клик флажка коммитит массив, где `Osnovanie` ещё в активном редакторе (uncommitted); при возврате ответа фронт re-apply'ит dirty-снимок (включая **сырой ввод активного `Osnovanie`**) поверх канона (§2.5.4а п.2) → ввод не теряется. Ключ: dirty-снимок ADR-0011 трекает ввод **по полю-колонке**, не по визуальной ячейке — VERTICAL не ломает это, т.к. данные плоские. Граница та же: фронт переносит ввод, сервер даёт производные.
- Чекбокс без естественного blur — commit на клик (intra-cell-переход текст→флажок коммитит текст по его blur, флажок — по клику); два commit'а сериализуются in-flight-правилом (ADR-0011 §2.5.4а) штатно.

Поскольку прувпойнт ИПН — HORIZONTAL (§2.2), VERTICAL-commit-семантика проверяется при первом реальном VERTICAL-кейсе (acceptance Phase 1+, не критический путь ИПН).

### 2.4 Видимость / состав колонок

`TABLE_COLUMN`/`COLUMN_GROUP` props `visible: Boolean` (default `true`; `Visible=false` каталога → `false`). Скрытая колонка:
- **не рендерится** на фронте, но **остаётся в модели строки** (значение есть в `{rowId,...}` — нужно для master-detail-ключа, который скрыт, но участвует в связи, §4).
- Динамика видимости от состояния (показать колонку при условии) — `setProp(colNodeId, "visible", ...)`-патч handler'а (как ADR-0011 §2.5.3, не декларативный `visibleWhen`).

Состав отображаемых колонок = набор `visible:true` колонок в метаданных. Прувпойнт п.2: GrafikVycheta показывает `Razmer` + группу периода {`PeriodDeystviyaNachalo`, `PeriodDeystviyaKonets`}, а `VychetIPN` (ключ связи) — `visible:false`.

### 3.2 ⚠ Допущение PM по требованию п.2 — ФЛАГ ВЛАДЕЛЬЦУ (требует подтверждения)

Требование владельца «в таблице **ВычетыИПН** показывать только Размер/Дата начала/Дата окончания» **противоречит метаданным**: эти 3 колонки принадлежат **GrafikVycheta**, а не VychetyIPN. В VychetyIPN из них есть только `PeriodDeystviyaKonets` (Дата окончания); `Razmer` и `PeriodDeystviyaNachalo` — колонки GrafikVycheta (каталог строки 1416-1419).

**Принятая (PM) метаданно-консистентная трактовка, подтверждённая каталогом:** п.2 относится к **GrafikVycheta** — она показывает только `Razmer` + {`PeriodDeystviyaNachalo`, `PeriodDeystviyaKonets`}, скрывая ключ `VychetIPN`. А VychetyIPN показывает `VychetIPN` + вертикальную группу {`Osnovanie`, `PredostavlyatVychet`} + `PeriodDeystviyaKonets`.

**Статус: требует подтверждения владельца.** Каталог однозначно поддерживает эту трактовку (3 колонки = ровно видимый состав GrafikVycheta). Если владелец имел в виду буквально VychetyIPN — это либо ошибка в требовании, либо нестандартная форма; analyst-1c сверяет с типовой формой 1С (источники §9). **Альтернатива** (если 1С-типовая показывает иначе): применить состав к VychetyIPN, но тогда `Razmer`/`PeriodDeystviyaNachalo` в ней отсутствуют — невыразимо без правки метаданных типа строки (это уже не «скрыть колонку», а «добавить чужой реквизит»).

### 3.3 Ориентация группы {Osnovanie, PredostavlyatVychet}: HORIZONTAL в прувпойнте (challenger D2)

Требование владельца «Osnovanie и PredostavlyatVychet друг под другом» указывает на `orientation:VERTICAL`. Но: (1) само требование п.2 под флагом «требует подтверждения» (§3.2); (2) реальная УФ-1С вертикальную стопку в ячейке ТЧ не поддерживает и кладёт их **горизонтально рядом** (§9); (3) VERTICAL несёт переопределение commit-семантики (§2.3, D1). Решение: **прувпойнт ИПН реализуется на HORIZONTAL** (паритет с реальной формой, нулевой риск механики). VERTICAL — переключаемая опция (смена prop `orientation`), включается после: подтверждения владельца, что нужна именно вертикальная стопка, И выверки analyst-1c, что типовая форма так выглядит. До этого VERTICAL **не на критическом пути** acceptance Phase 1. Это убирает deferred-risk с прувпойнта (challenger D2).

## 4. Решение — master-detail (две связанные ТЧ)

### 4.1 Природа связи (analyst-1c + developer-1c)

VychetyIPN (master) и GrafikVycheta (detail) связаны по реквизиту **`VychetIPN`**: одной строке master соответствуют N строк detail с тем же `VychetIPN` («один master → много detail»). В типовой 1С-форме — **обе таблицы видны одновременно** (одна над другой), detail фильтруется по `ВычетИПН` активной строки master через `ТаблицаФормы.ОтборСтрок`.

**Ключевой технический факт (developer-1c, §9):** `ОтборСтрок` — **клиентский** механизм: detail физически содержит **ВСЕ** строки (для всех master), `ОтборСтрок` лишь **визуально** показывает подмножество по ключу, **без round-trip на сервер**. Фильтрация detail по выбранной master-строке — **презентационный отбор, НЕ бизнес-логика** (данные не меняются, на проводки/расчёты не влияет).

### 4.2 Решение: две отдельные TABLE, master-detail-фильтр — презентационный (на фронте)

| Аспект | Решение |
|---|---|
| **Рендер** | Две отдельные TABLE-узла на форме (master сверху, detail снизу) — обе всегда видны. Не вложенные таблицы. |
| **Данные detail** | Бэк отдаёт **ВСЕ** строки detail сразу (полный `value` TABLE-узла) — как 1С держит все строки в форме. |
| **Связь** | По скрытой колонке `VychetIPN` (visible:false) в detail = ключ связи. Хранится как ссылка `{id,presentation}` (не rowIndex — §9, при reorder/delete master не ломается). |
| **Фильтрация detail по выбранной master-строке** | **Презентационная, на фронте** (как `ОтборСтрок` в 1С): при выборе строки master фронт локально показывает строки detail с `VychetIPN == выбранный`. Round-trip НЕ требуется. |
| **Метаданные связи** | TABLE-узел detail несёт props `masterTable: "table.vychetyIPN"`, `masterKey: "VychetIPN"`, `detailKey: "VychetIPN"` — декларативно, бэк описывает связь данными, не кодом. |

### 4.3 Почему презентационный фильтр на фронте НЕ нарушает «логику на бэке»

Это главный вопрос для challenger. Аргумент (developer-1c): `ОтборСтрок` в 1С — **клиентское UI-свойство**; фильтрация видимости строк по выбранному master — это **презентационное состояние**, не вычисление/валидация/проводка. Принцип «логика на бэке» (ADR-0011 §2.5.2) относится к **бизнес-логике** (пересчёт, валидация, CRUD-эффект), а не к тому, какие из уже-присланных строк показать. Аналогия с ADR-0011: фронт сам пагинирует/листает присланные данные (LIST §2.1.1 ADR-0009 — «грид сам листает уже-описанный источник») — это не утечка логики. Master-detail-фильтр — того же класса: бэк отдал ВСЕ строки + декларативную связь (`masterKey/detailKey`), фронт показывает срез по выбору — **выборка из присланного, не вычисление нового**.

**Граница (явно):** фронт **только фильтрует видимость** уже присланных строк по равенству ключа. Он **НЕ** вычисляет ключ, **НЕ** меняет данные, **НЕ** решает правила связи (их объявляет бэк в `masterKey/detailKey`). Если связь сложнее равенства ключа (напр. диапазон дат, условие) — это уже бизнес-логика → round-trip EVENT на бэк (опция §4.4). Для ИПН — простое равенство `VychetIPN`, презентационный фильтр достаточен.

### 4.4 Опция: server-side master-detail (для сложной связи / огромного detail)

Если detail огромен (тысячи строк — грузить все нецелесообразно) ИЛИ связь сложнее равенства ключа — альтернатива: EVENT `rowActivated` от master → бэк возвращает `setValue(detailTable, отфильтрованные rows)`. Это round-trip, но для типовых документов БГУ (десятки строк) **избыточен** (developer-1c) — Phase 1 использует презентационный фильтр (§4.2). Server-side — отложенная опция, не Phase 1.

## 5. Решение — подвалы-итоги (footer)

### 5.1 Метаданные «у колонки есть подвал»

`TABLE_COLUMN` props `footer: Boolean` (`Подвал=true` каталога; 79 полей) + `footerAgg: "SUM"|"COUNT"|...` (тип агрегата, default SUM). TABLE-узел рендерит строку-подвал, в ячейках колонок с `footer:true` — серверно-вычисленное значение.

### 5.2 Пересчёт подвала — КОД на бэке (как onTableChanged)

Подвал — серверно-вычисляемая агрегатная строка. Значение приходит **патчем** от бэка (не считается фронтом, §2.5.2 ADR-0011):
- бэк при table-EVENT (или master-detail-change) вычисляет агрегаты колонок с `footer:true` (через `TablePartHelper.sumColumn`) и шлёт `setValue(table.<x>.footer.<colCode>, value)` или подвал как часть нормализованного ответа;
- фронт показывает присланные итоги, **локально НЕ суммирует**.

Пересчёт = код handler'а (аналог `onTableChanged` ADR-0011 §3.6); показ = метаданные (`footer:true`). Тот же шов данные/код.

## 6. CI-гейт и метаданные (наследует ADR-0011 §2.5.5)

Усиление до CI-гейта (аналог `CommandBarRegistryConsistencyTest`):
1. **Каждая колонка/группа метаданных ↔ существующий EAV-атрибут** типа строки ТЧ (orphan = error). Для `COLUMN_GROUP` — у группы нет attrCode (она контейнер), но каждый дочерний `TABLE_COLUMN.attributeCode` обязан резолвиться.
2. **`COLUMN_GROUP` вложенность ≤ 2** (3-й уровень = error).
3. **`masterKey`/`detailKey` master-detail ссылаются на существующие колонки** обеих ТЧ (orphan = error).
4. **Транслитерация** attrCode колонок/групп/tableCode — по таблице; `TransliterationConsistencyTest`.
5. **VERTICAL-группа: дочерние колонки существуют** и плоские (не вложенная группа внутри VERTICAL — §10).
Источник гейта = прод-сид (как ADR-0010 §3.4.7 п.1).

## 10. Альтернативы и контраргументы (challenger round-trip)

### Решение 1: `NodeType.COLUMN_GROUP` (новый тип) vs вложенный `TABLE_COLUMN`+флаг

**Альтернатива 1а: вложенные `TABLE_COLUMN` с детьми + флаг `isGroup`.** **Почему отклонена:** `TABLE_COLUMN` семантически = «колонка с binding-атрибутом»; группа binding не имеет (она контейнер с label-заголовком). Перегрузка `TABLE_COLUMN` двумя ролями (лист-колонка vs контейнер-группа) + флаг = неоднозначный контракт, две ветки рендера на одном типе (источник багов, как `LIST` vs `TABLE` в ADR-0009 А3). Отдельный `COLUMN_GROUP` чётко разделяет «контейнер-шапка» (label, orientation, children) от «лист-колонка» (binding, cellWidget, footer). Один новый тип — приемлемая цена ясности.

**Контраргумент к себе:** ещё один NodeType раздувает контракт (как `LIST` в ADR-0009). **Ответ:** группа колонок структурно ОТЛИЧНА от колонки (нет binding, есть orientation/children) — это не косметика, а другой узел; флаг на `TABLE_COLUMN` протёк бы в рендер и сериализацию. Цена — один enum-член + `-am`-сборка.

### Решение 2: master-detail — две отдельные таблицы + презентационный фильтр на фронте (не server-side, не вложенные таблицы)

**Альтернатива 2а: вложенная detail-таблица внутри строки master.** **Почему отклонена:** 1С-форма показывает две **отдельные** таблицы (master сверху, detail снизу), не вложенные; вложенность усложнила бы рендер и save (detail — отдельный TABLE-атрибут документа, не часть строки master). Каталог: GrafikVycheta — отдельная ТЧ документа, не под-ТЧ строки VychetyIPN.

**Альтернатива 2б: server-side фильтр (round-trip на смену master-строки).** **Почему отклонена для Phase 1:** `ОтборСтрок` в 1С — клиентский, detail держит ВСЕ строки (developer-1c); для десятков строк round-trip избыточен и даёт мерцание/задержку. Презентационный фильтр на фронте — паритет с 1С и с пагинацией LIST (ADR-0009). Server-side — опция для огромного detail / сложной связи (§4.4).

#### Контраргументы (challenger round-trip)
1. **Возражение** [ожидается major]: презентационный фильтр detail на фронте = утечка бизнес-логики связи на клиент, противоречит §2.5.2 ADR-0011 «фронт ничего не вычисляет». — **Ответ автора:** фильтр видимости по равенству присланного ключа — **презентационный отбор**, не вычисление (developer-1c: `ОтборСтрок` — клиентское UI-свойство; данные не меняются, на расчёты не влияет). Бэк объявляет связь декларативно (`masterKey/detailKey` в метаданных), фронт показывает срез присланных строк — выборка, не вычисление нового. Аналог — фронт сам листает присланный источник (LIST §2.1.1). Граница: сложнее равенства ключа → round-trip (§4.4). — **Статус: к проверке challenger.**
2. **Возражение** [ожидается]: VERTICAL-составная ячейка — несуществующее в 1С поведение, риск рассогласования с типовой формой. — **Ответ автора:** явно зафиксировано как SDUI-расширение (§2.2), не порт; переключение на горизонталь = смена одного prop `orientation` (server-driven); analyst-1c выверяет реальную форму (§3.3). — **Статус: accepted (осознанное расширение).**

### Решение 3: масштаб metadata-сида на 3 подсистемы — данные, не код

**Альтернатива 3а: per-type Java-описание структуры таблиц.** **Почему отклонена:** прямо противоречит server-driven директиве (ADR-0010/0011); 283 группы × десятки ТЧ в Java = неподдерживаемо. Сид из каталога (`layout_nodes`/`layout_node_props`) — механический маппинг (data-engineer), Java только для пересчётов/валидаций.

**Контраргумент к себе:** 283 группы в данных — большой сид, риск рассинхрона метаданные↔EAV-атрибуты. **Ответ:** CI-гейт «orphan=error» (§6) ловит рассинхрон на прод-сиде; масштаб — аргумент ЗА данные (Java-описание 283 групп хуже), не против.

## 12. Последствия

**Выигрываем:**
- Сложные таблицы (группы, составные ячейки, подвалы, master-detail) — server-driven: структура в метаданных, раскатка на подсистемы = сид (данные), не Java per-type.
- Наследование ADR-0011: модель строк, scratch, save, table-level sync, cellWidget — не переоткрываются.
- master-detail презентационным фильтром — паритет с 1С `ОтборСтрок`, без round-trip; не утечка логики (декларативная связь на бэке).
- VERTICAL-ячейка — SDUI-расширение поверх 1С, переключаемое одним prop.

**Платим:**
- Один новый NodeType (`COLUMN_GROUP`) + props (`-am`-сборка).
- Фронт-дельты: группированная/спанящая шапка, вертикально-составные ячейки, строка-подвал, скрытие колонок, master-detail-фильтр (см. фронт-спеку).
- VERTICAL в УФ-1С нет — риск рассогласования с типовой формой (митигация: выверка analyst-1c + переключаемый prop).
- Допущение п.2 (GrafikVycheta, не VychetyIPN) — **требует подтверждения владельца** (§3.2).

**Отложено:** server-side master-detail, подсистемы Phase 2, вложенность >2.
