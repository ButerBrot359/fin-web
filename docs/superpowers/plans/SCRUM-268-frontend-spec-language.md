# fin-web: язык интерфейса в SDUI-формах (KZ-локализация) — спека для фронта

**Контекст.** Переключатель РУС/ҚАЗ сейчас локализует только «хром» приложения (i18next). SDUI-формы (`POST /api/view`) остаются русскими: фронт не сообщает бэку язык. Бэкенд-поддержка реализована в webbuh (ветка `talgat/SCRUM-268`): сервер принимает язык, фиксирует его в form-session и компонует командную панель, лейблы полей и заголовок документа на нужном языке.

**Связано:** SCRUM-218 (label_kz командной панели), SCRUM-268; предыдущая спека — `SCRUM-268-frontend-spec.md` (композитные виджеты — уже реализовано, спасибо!).

---

## Контракт бэкенда (уже готов)

- `POST /api/view` принимает новое **опциональное** поле body `language: string`.
  - Канонические значения: `"Ru"` / `"Kz"` (регистр как у print `?language=Kz`). Парсинг lenient: `kk`, `kk-KZ`, `Kaz` и т.п. тоже распознаются как казахский.
  - Поле **читается только на `action: "OPEN"`** — язык резолвится один раз и фиксируется в form-session на весь её жизненный цикл. Для `EVENT`/`COMMAND` поле игнорируется (язык берётся из сессии), слать его там не вредно.
  - Fallback-цепочка на сервере: `language` из body → заголовок `Accept-Language` → `Ru`. Отсутствие поля = старое поведение (Ru), обратная совместимость полная.
- **Смена языка = новая сессия.** Мутировать язык живой сессии нельзя — нужно закрыть текущую (CLOSE) и переоткрыть форму (OPEN) с новым `language`.
- Fallback лейблов двунаправленный на сервере: если у атрибута/команды нет KZ-перевода — придёт русский (пустых лейблов не будет). Ничего дополнительно обрабатывать не нужно.
- Локализуются в этом срезе: командная панель (главный ряд + «Ещё»), лейблы полей формы (nameKz атрибутов), заголовок документа (имя типа + « (жасау)»). НЕ локализуются пока: notify-тосты, титулы drawer-ов справочников, «Движения документа»-диалог, связка « от » в заголовке — это бэковые follow-up-ы.

## Что сделать во fin-web

### 1. Слать язык в каждый `/api/view` (одна точка)

`src/features/sdui/api/view-transport.ts` — единственный HTTP-chokepoint SDUI: все 4 билдера запросов (`lib/dispatch.ts` ×2, `lib/relay-selection.ts` ×2) проходят через `viewTransport.post`. Правка в транспорте покрывает всех:

```ts
// types/view.ts — добавить в ViewRequest:
language?: string;

// view-transport.ts — в post(), перед отправкой:
import i18n from 'i18next';
const LANG_MAP: Record<string, string> = { ru: 'Ru', kz: 'Kz' };
req.language = LANG_MAP[i18n.language] ?? 'Ru';
```

Важно: i18next в проекте использует коды **`ru`/`kz`** (`supportedLanguages` в `app/config/i18n/index.ts`), не `kk` — маппинг обязателен (хотя сервер lenient и `kz` тоже поймёт, шлите канонические `Ru`/`Kz`).

Опционально (не обязательно): продублировать заголовком через `instance.interceptors.request.use` (`Accept-Language: kk-KZ` для kz). Body-поле выигрывает у заголовка, так что это только страховка.

Каветка: CLOSE через `navigator.sendBeacon` (`closeBeacon`) не может нести body/заголовок — это ок, на CLOSE язык не нужен.

### 2. Пересоздавать form-session при переключении языка

`src/features/sdui/ui/sdui-screen.tsx`:

1. OPEN-эффект сейчас зависит только от `location.pathname` (deps, ~строка 87) — добавить текущий язык в зависимости, чтобы переключение триггерило re-OPEN.
2. **Главная засада — `useSduiCacheStore`**: restore-ветка (~строки 57-64) при revisit восстанавливает дерево + `formSessionId` из кэша и **пропускает OPEN**. После смены языка кэш вернёт стейл RU-сессию. Нужно: на `i18n.on('languageChanged', …)` инвалидировать SDUI-кэш (все закэшированные роуты — сессии на сервере язычно-фиксированы).
3. Старую сессию перед re-OPEN закрыть (существующий CLOSE/`closeSession`) — иначе на сервере копятся сироты.
4. **UX по dirty-формам**: при re-OPEN несохранённый scratch теряется. Рекомендация: если на открытой SDUI-форме есть несохранённые изменения — confirm-диалог («Переключение языка перезагрузит форму, несохранённые изменения будут потеряны»), по отмене — не переключать язык. Минимально приемлемо для первого среза: просто перезагружать (как F5), но проговорите это решение.

### 3. Критерии приёмки

- [ ] Переключение на ҚАЗ → открытая SDUI-форма (Заявка ГП, ЭСФ) переоткрывается: кнопки «Өткізу және жабу | Жазу | Өткізу | Дт/Кт | Қазынашылыққа жүктеп шығару | Басып шығару | Есептер | Тағы», лейблы полей казахские (там, где у атрибута есть nameKz; иначе русский, пустых нет).
- [ ] Обратно на РУС → всё русское (без стейл-кэша).
- [ ] Каждый `/api/view` в network-трейсе несёт `language`.
- [ ] Навигация по списку/формам после переключения не воскрешает RU-сессии из кэша.
- [ ] `formSessionId` до и после переключения разные; старая сессия закрыта (CLOSE в трейсе).

## Как проверить локально

Бэкенд из ветки `talgat/SCRUM-268`: `docker compose up -d db && ./mvnw spring-boot:run -pl webbuh-api`. Быстрая проверка сервера без фронта:

```bash
curl -s -X POST http://localhost:8080/api/view -H "Content-Type: application/json" -d '{
  "action": {"type": "OPEN"},
  "layoutCode": "ZayavkaNaRegistratsiyuGPSdelki.ФормаОбъекта",
  "route": "/documents/ZayavkaNaRegistratsiyuGPSdelki/new",
  "language": "Kz"
}' | grep -o 'Өткізу[^"]*'
```
