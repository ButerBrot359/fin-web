# SDUI: фронт-правки (fin-web) — дрейф revision на полевых EVENT (ложный STALE_REVISION)

> **Дата:** 2026-06-30
> **Контекст:** при заполнении нового документа `ZayavkaNaRegistratsiyuGPSdelki` повторный
> выбор поля (организация, банковский счёт, описание расхода) падал с
> `409 STALE_REVISION` («clientRevision=1 < currentRevision=2»).

Бэк-часть уже сделана: **EVENT стал revision-толерантным** — сервер применяет дельту поля и
возвращает актуальную revision, не отвечая 409 (`ViewController.handleEvent`). Это снимает
блокирующую ошибку. Но **первопричина дрейфа — на фронте**, и её надо устранить, иначе
останутся лишние запросы и рассинхрон состояния.

---

## Симптом и трассировка

```
1) EVENT field.organizatsiya (client rev 0) → OK   (сервер: rev 0→1, в ответе setValue schetOrg = автосчёт)
2) EVENT field.schetOrg     (client rev 1) → OK, ответ { revision: 2, patches: [] }
3) EVENT field.schetOrg     (client rev 1) → 409 STALE (currentRevision=2)
```
На шаге 3 клиент **снова шлёт rev 1**, хотя шаг 2 вернул revision 2.

---

## Корень (фронт) — две причины

### Причина A — revision не усваивается из ответа с пустыми патчами
Каждое полевое EVENT на сервере делает `session.mutate()` (revision++), **включая no-op
поля** без доменной реакции (`schetOrg`, `opisanieRaskhoda`): они возвращают `patches: []`,
но **`revision` в ответе всё равно новый**. Похоже, фронт обновляет свою revision только когда
есть патчи/`statePatch`, и игнорирует `revision` ответа с пустыми патчами → остаётся на старом
значении → следующий EVENT уходит со стале-revision.

**Фикс:** обновлять локальную revision из **каждого** ответа `POST /api/view` (OPEN/EVENT/COMMAND),
безусловно, до применения патчей и независимо от их наличия:
```ts
// в обработчике ответа dispatch (EVENT/COMMAND), ПЕРВЫМ делом:
treeStore.bumpRevision(res.revision)   // всегда, даже если res.patches пустой
// затем applyPatches / merge statePatch / playEffects
```

### Причина B — серверный `setValue` провоцирует повторный change того же поля
Выбор организации возвращает `setValue` для `field.schetOrg` (автоподстановка основного счёта
организации, `onOrganizatsiyaChanged`). Если компонент поля при программном обновлении значения
(из патча сервера) **сам стреляет change-EVENT**, как при пользовательском вводе — летит лишний
EVENT `field.schetOrg`, поднимающий revision на сервере за спиной пользователя. Это и есть
источник «лишнего» инкремента и рассинхрона.

**Фикс:** различать **серверное** обновление значения и **пользовательский** ввод (тот же
раздел server/user-set, что и для dirty-трекинга в
[frontend-spec-zayavka-gp-features.md](frontend-spec-zayavka-gp-features.md) Фича 2):
- применение `setValue`-патча и `statePatch` от сервера идёт через путь, который **не**
  инициирует `change`-EVENT поля (`setFromServer`);
- `change`-EVENT шлётся **только** на реальный пользовательский ввод (onChange виджета по
  действию пользователя), не на программную установку значения.

---

## Дополнительно (defense in depth) — обработка 409 STALE
Даже после бэк-толерантности на EVENT, прочие команды (`unpost`, `ref.select`, кастомные)
по-прежнему могут вернуть `409 STALE_REVISION` с `currentRevision`. Фронт должен **не показывать
жёсткую ошибку**, а адаптироваться: принять `currentRevision` из тела 409 и (при необходимости)
переотправить действие или подтянуть состояние, как предписывает ADR-0006 §конфликт-резолв.

## Проверка (acceptance)
1. Новый документ → выбрать организацию → счёт подставился автоматически (1 запрос, без лишнего
   EVENT по `field.schetOrg`).
2. Повторно выбрать банковский счёт организации 2–3 раза подряд — **без** `STALE_REVISION`,
   значение каждый раз применяется.
3. Заполнить описание расхода после счёта — без ошибки.
4. После каждого ответа локальная revision = `res.revision`.
