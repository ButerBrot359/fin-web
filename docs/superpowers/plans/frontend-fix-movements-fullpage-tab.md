# SDUI: фронт-правки (fin-web) — «Дт/Кт» полной страницей + вкладкой, единый вход из формы и списка

> **Дата:** 2026-06-30
> **Контекст:** баги 7 и 8 батча по движениям документа (ДтКт).
> - **Баг 7:** из формы документа «Дт/Кт» открывает маленькое модальное окно — должно быть
>   на всю страницу и регистрироваться отдельной вкладкой в панели рабочих столов (история внизу).
> - **Баг 8:** из списка документов «Дт/Кт» открывает другую (легаси) страницу проводок —
>   должно открывать ту же SDUI-страницу движений, что и из формы.

Бэк-часть бага 7 уже сделана. Баг 8 — целиком фронт (роутинг тулбара списка), бэк не меняется.

---

## Баг 7 — полная страница + вкладка рабочего стола

### Что изменил бэк
`MovementsComposer` ([MovementsComposer.java](../../../webbuh-api/src/main/java/kz/asiaservis/view/composer/MovementsComposer.java))
больше не отдаёт `presentation: "modal"`. Корневой PAGE движений теперь несёт:
```jsonc
{
  "id": "dialog.movements",
  "type": "PAGE",
  "props": {
    "title": "Движения документа",
    "presentation": "page",            // ← было "modal"
    "openInWorkspaceTab": true,        // ← НОВОЕ: завести вкладку рабочего стола
    "tabKey": "movements:<documentEntryId>"  // ← НОВОЕ: стабильный ключ вкладки
  },
  "children": [ /* TABS → TAB → TABLE → TABLE_COLUMN, как раньше */ ]
}
```
Поток данных не изменился: строки по-прежнему приходят `setValue`-патчами в view-state по
`binding` каждой таблицы (см. [frontend-spec-movements.md](frontend-spec-movements.md) §1).

### Что сделать (фронт)
Сейчас `effect-handler`/`dispatch` для `openDialog` ветвит только `presentation==='drawer'`
(MUI `Drawer`) иначе `Dialog` ([dialog-host.tsx](../../..) — `frontend-impl-movements.md` §2).
Добавить третью ветку — `presentation==='page'`:

1. **Рендер на всю страницу.** При `presentation==='page'` не открывать `Dialog`/`Drawer`, а
   смонтировать поддерево (`NodeRenderer node={effect.node}`) как полноэкранный контент в
   области рабочего стола (как обычный SDUI-экран), а не оверлеем.
2. **Регистрация вкладки.** Если `props.openInWorkspaceTab===true` — завести/активировать
   рабочий стол (вкладку в `WorkspaceTabBar` внизу) с заголовком `props.title` и идентичностью
   по `props.tabKey`:
   - если вкладка с таким `tabKey` уже есть — **активировать существующую** (не плодить дубль);
   - закрытие вкладки снимает страницу (и, если нужно, шлёт `CLOSE`/чистит view-state движений).
3. **Данные.** Строки уже легли в view-state `setValue`-патчами до `openDialog` — `TableNode`
   читает их по `binding` как и прежде; ничего дополнительно грузить не нужно.

> Минимальный вариант, если полноценная вкладка — большой объём: при `presentation==='page'`
> рендерить контент полноэкранно (full-screen MUI `Dialog` с `fullScreen`), а регистрацию
> вкладки добавить следом. Но цель бага 7 — именно **вкладка в истории**, поэтому
> предпочтителен полноценный рабочий стол.

## Баг 8 — единый вход из списка документов

### Симптом
В списке документов «Дт/Кт» (тулбар списка) ведёт на **легаси-страницу проводок** (отдельный
роут), а не на SDUI-страницу движений, которую открывает кнопка из формы.

### Корень (фронт)
Тулбар списка документов навигирует на старый маршрут проводок вместо того, чтобы запустить
SDUI-поток `showMovements`. Бэк для движений — generic и единый: команда `showMovements`
работает для любого документа по его `documentEntryId` (та же `MovementsComposer`-страница).

### Что сделать (фронт)
«Дт/Кт» в тулбаре списка должна открывать **ту же** SDUI-страницу движений, что и из формы:
- по выделенной строке списка взять `documentEntryId`;
- запустить тот же поток, что кнопка формы: открыть SDUI-движения через `showMovements`
  (та же страница `presentation:'page'` + вкладка из бага 7).

Варианты реализации (выбрать по тому, как устроен список):
- **Через сессию:** открыть/переиспользовать SDUI-сессию документа и сдиспатчить
  `COMMAND {command:'showMovements'}` — бэк ответит `openDialog`(page) + строки.
- **Через прямой вход на страницу движений:** навигировать на SDUI-маршрут движений по
  `documentEntryId`, который инициирует `showMovements` (без открытия полной формы документа).

Главное — **убрать переход на легаси-страницу проводок** и использовать единый generic-поток
движений. Бэк-эндпоинт один и тот же для формы и списка.

> Если для списка нет SDUI-сессии и её заведение нетривиально — согласуйте с бэком тонкий
> вход (напр. `OPEN` лёгкой сессии движений по `route` с `documentEntryId`, сразу
> эмитирующей `showMovements`). По умолчанию переиспользуется существующий generic
> `showMovements` — новых бэк-артефактов не требуется.
