## Изменение 1 — новая запись справочника появляется в списке без перезагрузки

### Проблема
При создании записи справочника из формы выбора («Создать» → сохранить) значение **выбиралось** в поле, но в выпадающем списке опций поля новой записи **не было до перезагрузки страницы**.

### Причина
Ссылочное поле кэширует опции в локальном `useState` (`options`) внутри самого компонента. Кэш сбрасывается только при смене параметров источника (организация/фильтр). Создание записи параметры не меняет → кэш не инвалидируется → следующий `onOpen` видит непустой список и не перезапрашивает (`if (options.length === 0)` = false). Данные на бэке свежие — проблема чисто фронтовая.

### Правка (файл `reference-field-node.tsx`, функция `applySelected`)
Добавлен сброс кэша опций после выбора/создания:

```ts
const applySelected = (opt: SelectOption | null) => {
  const newVal = opt ? fromSelectOption(opt) : null
  if (node.binding) setValue(node.binding, newVal)
  // Сброс локального кэша опций: если запись только что создана из формы выбора,
  // следующий onOpen (options.length===0) перезапросит свежий список и покажет
  // новую запись без перезагрузки страницы.
  setOptions([])
  fireServerEvent('change', newVal)
}
```

Безопасно: при обычном выборе существующего значения максимум один лишний refetch опций при следующем открытии дропдауна. Лечит оба пути создания — server-driven (`applyToParent` → `ref.select`) и клиентский dict-sidebar (`panel.onSelect`).

### Как проверить
1. Открыть документ, у ссылочного поля с включённым «Создать» (напр. поле «Описание расходов», или Контрагент/Договор/Счёт после picker-миграции на бэке) нажать «Создать».
2. Сохранить новую запись — значение выбирается в поле.
3. Снова открыть дропдаун того же поля → **новая запись присутствует в списке без перезагрузки страницы**.
4. Регресс: обычный выбор существующей записи работает как раньше.

## Изменение 2 — двухуровневая шапка (COLUMN_GROUP) в read-only SDUI-таблице

### Зачем
Движения документа (вкладка «Журнал проводок госучреждения / Регистр бухгалтерии», ДтКт) теперь приходят с бэка сгруппированными: колонки дебета обёрнуты в `COLUMN_GROUP` «ДЕБЕТ», кредита — в «КРЕДИТ» (как в старой архитектуре). Раньше `ReadOnlyTable` умел только плоские `TABLE_COLUMN` и **игнорировал** `COLUMN_GROUP` — вложенные колонки Дт/Кт выпадали, шапка была одноуровневой.

Группировка — **решение бэка** (в дереве узлов), фронт лишь рисует присланную структуру. Никакой ДтКт-специфичной логики на фронте нет — рендер полностью generic для любого `COLUMN_GROUP`-дерева.

### Что приходит с бэка (контракт, что рендерим)
Read-only TABLE-узел (`editable=false`) с детьми — смесь плоских `TABLE_COLUMN` и `COLUMN_GROUP`:

```jsonc
{
  "id": "table.acc", "type": "TABLE", "binding": "<...>",
  "props": { "editable": false },
  "value": [ { "rowId": "1", "_accountDtCode": {"presentation":"1315"}, "_fkrDt": {"presentation":"144"}, ... } ],
  "children": [
    { "id": "col.acc.period", "type": "TABLE_COLUMN", "props": { "label": "Период", "binding": "_period" } },
    { "id": "colgroup.debet", "type": "COLUMN_GROUP",
      "props": { "label": "ДЕБЕТ", "orientation": "HORIZONTAL" },
      "children": [
        { "id": "...", "type": "TABLE_COLUMN", "props": { "label": "Счёт", "binding": "_accountDtCode" } },
        { "id": "...", "type": "TABLE_COLUMN", "props": { "label": "Субконто 1", "binding": "_subkontoDt1" } },
        { "id": "...", "type": "TABLE_COLUMN", "props": { "label": "ФКР", "binding": "_fkrDt" } },
        { "id": "...", "type": "TABLE_COLUMN", "props": { "label": "Специфика", "binding": "_spetsifikaDt" } },
        { "id": "...", "type": "TABLE_COLUMN", "props": { "label": "Источник финансирования", "binding": "_istochnikFinansirovaniyaDt" } },
        { "id": "...", "type": "TABLE_COLUMN", "props": { "label": "Подразделение", "binding": "_podrazdelenieDt" } },
        { "id": "...", "type": "TABLE_COLUMN", "props": { "label": "Код платных услуг", "binding": "_kodPlatnykhUslugDt" } }
      ] },
    { "id": "colgroup.kredit", "type": "COLUMN_GROUP", "props": { "label": "КРЕДИТ", ... }, "children": [ /* симметрично _...Kt */ ] },
    { "id": "col.acc.organizatsiya", "type": "TABLE_COLUMN", "props": { "label": "Организация", "binding": "_organizatsiya" } },
    { "id": "col.acc.summa", "type": "TABLE_COLUMN", "props": { "label": "Сумма", "binding": "_summa" } },
    { "id": "col.acc.soderzhanie", "type": "TABLE_COLUMN", "props": { "label": "Содержание", "binding": "_soderzhanie" } },
    { "id": "col.acc.isActive", "type": "TABLE_COLUMN", "props": { "label": "Активность", "binding": "_isActiveLabel" } }
  ]
}
```

Ключевое:
- **Данные строк плоские** — ключи = `binding` листовых колонок (`_accountDtCode`, `_fkrDt`, …). Никаких вложенных `colgroup.*` ключей. Группировка чисто презентационная.
- `binding` листовой колонки может лежать в `node.binding` (server-driven) или `props.binding` (legacy) — фронт читает `node.binding ?? props.binding`.
- Ссылочные ячейки — объект `{id, presentation, entityRef}`; примитивы — строкой. Рендерятся через `renderCellValue` → `presentation`.

### Правка (файл `table-node.tsx`, компонент `ReadOnlyTable` + хелперы)
Добавлено/изменено:
- `readBinding(node)` — `node.binding ?? props.binding` (суперсет прежнего чтения только `props.binding`; обратно совместимо).
- `extractReadOnlyColumns(children)` — теперь **рекурсивно** спускается в `COLUMN_GROUP.children`, собирая листовые колонки в порядке документа (для рендера ячеек строк).
- `collectLeafNodes(node)` — сбор листовых `TABLE_COLUMN` под узлом.
- `buildHeaderModel(children)` → `{ hasGroups, topRow, bottomRow }`:
  - `COLUMN_GROUP` → ячейка верхнего ряда с `colSpan = число листьев`;
  - плоский `TABLE_COLUMN` (вне группы) → ячейка верхнего ряда с `rowSpan = 2` (когда есть группы);
  - листья групп → нижний ряд.
- `ReadOnlyTable` рендерит `<TableHead>` с первым `<TableRow>` всегда и вторым `<TableRow>` **только если** `hasGroups`. `colSpan`/`rowSpan` проставляются на `<TableCell>`; действие удаления (`allowDelete`) — `rowSpan=2` при группах.

### Инвариант обратной совместимости (важно проверить!)
Когда в детях **нет** `COLUMN_GROUP` (обычные read-only таблицы: движения регистров накопления/сведений, диалог `dtkt.rows` и пр.), `hasGroups=false`: второй ряд шапки **не рендерится**, `colSpan`/`rowSpan` = `undefined` (не проставляются) — шапка байт-в-байт как до правки. Ничего не должно измениться визуально для несгруппированных таблиц.

### Как проверить
1. **Движения регистра бухгалтерии (ДтКт)** документа (напр. Заявка ГП / Поступление от контрагента): шапка **двухуровневая** — верхний ряд «ДЕБЕТ» / «КРЕДИТ» (объединённые, по центру), под ними — под-колонки (Счёт, Субконто, ФКР, Специфика, Источник финансирования, Подразделение, Код платных услуг). Плоские колонки (Период, Организация, Сумма, Содержание, Активность) спанят обе строки шапки. Значения ячеек резолвятся (presentation), пусто → пустая ячейка.
2. **Регресс (несгруппированные):** движения регистра накопления/сведений и любые прочие read-only таблицы — одноуровневая шапка как раньше, без пустого второго ряда.
3. Сверить с эталоном старой архитектуры (богатая таблица «Журнал проводок»): состав колонок и группы совпадают.

> Предусловие: соответствующий бэкенд (`webbuh-api`) с обновлённым `MovementsComposer` (эмитит `COLUMN_GROUP`) должен быть развёрнут — иначе движения придут плоскими и двухуровневая шапка просто не активируется (что тоже корректно).

## Принцип (чтобы не искать «логику» на фронте)
Обе правки — рендер/UX, без бизнес-логики:
- Инвалидация кэша опций — это UX-механика показа (не вычисление).
- Группировка ДЕБЕТ/КРЕДИТ **решается на бэке** (`COLUMN_GROUP` в дереве); фронт лишь превращает присланное дерево в DOM (двухуровневый `<th colspan>`). `colSpan`/`rowSpan` = механический подсчёт листьев, а не логика.
- Никакого ДтКт-специфичного кода на фронте нет — рендер generic для любого `COLUMN_GROUP`-дерева (тот же примитив используют комплексные таблицы ИПН, ADR-0013).
