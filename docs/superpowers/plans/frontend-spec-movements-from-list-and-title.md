# Frontend-спека: движения документа из формы списка + заголовок с названием документа

Статус: спека для фронтендера (fin-web). Бэкенд-часть реализуется параллельно в webbuh-api.
Дата: 2026-07-09. Контекст: три бага по «Движениям документа» (ДтКт).

Связанные документы: `frontend-spec-movements.md`, `frontend-spec-movements-1c-block-parity.md`,
`frontend-impl-movements.md`, ADR-0012.

---

## Резюме багов

1. **Проводки пустые при вызове ДтКт из формы списка.** Из открытой формы (SDUI) проводки
   заполнены, из списка — пусто.
2. **На SDUI-странице движений нет заголовка с названием документа и нет стрелок назад/вперёд**,
   как у обычной формы. Должно быть: «Движения документа: {название документа}».
3. **В нижних рабочих вкладках (история) вкладка движений называется просто «Движения документа»** —
   должна содержать название документа: «Движения документа: Приходный кассовый ордер AAY00-00034 …».

Решение по #1 согласовано с владельцем: **полный SDUI-паритет** — список должен рендерить движения
тем же путём, что и форма (SDUI `openDialog`), а не legacy-страницей.

---

## Первопричины (по текущему коду fin-web)

### Баг #1
- Кнопка ДтКт в тулбаре списка `src/widgets/document-list-toolbar/ui/document-list-toolbar.tsx`
  (кнопка `152-165`, обработчик `handleMovements` `121-130`) делает **router-navigate** на
  `/modules/{pageCode}/document/{moduleCode}/{entryId}/movements?title=…&from=list`.
- Роут ведёт на **legacy-страницу** `src/pages/documents/document-movements/ui/document-movements-page.tsx`
  (`App.tsx:128-131`), которая дёргает **устаревший** эндпоинт
  `GET /api/document-entries/id/{entryId}/movements`
  (`src/pages/documents/document-movements/api/document-movements-api.ts:30`).
- Этот `/id/`-эндпоинт **не содержит регистра бухгалтерии** (он `@Deprecated`, отдаёт только
  накопление/сведения) → колонка/таблица проводок пустая, хотя `AccountingPostingsTable` во фронте есть.
- Из формы же идёт SDUI `POST /api/view` c `command:"showDtKt"` → бэк собирает движения через
  `DocumentMovementsByKindService` (регистр бухгалтерии включён) → проводки заполнены.

### Баги #2 / #3
- SDUI-страница движений собирается на бэке (`MovementsComposer`) с
  `PAGE.props.title = "Движения документа"` — без названия документа. Отсюда и заголовок страницы,
  и (при рендере как workspace-вкладки) имя вкладки — без названия документа.
- Стрелки назад/вперёд — это chrome обычной формы, на странице движений сейчас отсутствуют.

## Что делает бэкенд (уже реализуется — фронту на вход)

### 2.1. Новый session-less эндпоинт движений (для #1)

```
GET /api/view/movements/{documentEntryId}
```

Возвращает **тот же `ViewResponseDto`**, что и `POST /api/view`, но без сессии. Тело —
эффект `openDialog` с полностью собранным SDUI-деревом движений (те же узлы `PAGE → TABS → TAB →
TABLE → COLUMN_GROUP/TABLE_COLUMN`, что и из формы), плюс строки в `childState`.

Форма ответа:

```jsonc
{
  "formSessionId": null,
  "revision": 0,
  "tree": null,
  "state": null,
  "patches": null,
  "effects": [
    {
      "type": "openDialog",
      "node": {
        "id": "dialog.movements",
        "type": "PAGE",
        "props": {
          "title": "Движения документа: Приходный кассовый ордер AAY00-00034 от 09.07.2026 14:46:32",
          "presentation": "page",
          "openInWorkspaceTab": true,
          "tabKey": "movements:27856464"
        },
        "children": [ /* TABS → TAB → TABLE(readOnly, COLUMN_GROUP ДЕБЕТ/КРЕДИТ …) */ ]
      },
      "childState": {
        "movements.acc.ZhurnalProvodokGosUchrezhdeniya": [ /* строки проводок */ ],
        "movements.accum.VzaimoraschetyPoZayavkam": [ /* строки регистра накопления */ ]
      }
    }
  ]
}
```

Особенности:
- **`formSessionId: null`** — сессии нет. Эффект самодостаточен: строки лежат в `childState`
  (по тем же binding, что и `TABLE.binding` в дереве), setValue-патчей в этом ответе нет.
- Структура `node` **идентична** тому, что приходит из формы на `showDtKt` (можно переиспользовать
  весь существующий рендер `openDialog` presentation:"page").
- Нет движений (документ не проведён) → в `effects` придёт `notify` уровня `warning`
  вместо `openDialog`. Документ не найден → тоже `notify` `warning`.

### 2.2. Заголовок с названием документа (для #2, #3)

`PAGE.props.title` теперь = `"Движения документа: {presentation документа}"` (например
«Движения документа: Приходный кассовый ордер AAY00-00034 от 09.07.2026 14:46:32»).
Это касается **обоих** путей — и `showDtKt` из формы, и нового `GET /api/view/movements/{id}`.
Tab-level заголовки вкладок регистров (props.title у узлов `TAB`) остаются именами регистров.

## Что нужно сделать на фронте

### Задача A — движения из списка через SDUI (баг #1)

Заменить legacy-путь на SDUI. Целевое поведение: клик ДтКт в списке открывает ту же
самую страницу движений, что и из формы.

Рекомендуемый вариант (минимальная связность): **сделать роут `.../{entryId}/movements`
SDUI-потребителем нового эндпоинта** вместо legacy `DocumentMovementsPage`.

Шаги:
1. В обработчике/загрузчике движений вызвать `GET /api/view/movements/{entryId}` вместо
   `fetchDocumentMovements` (`document-movements-api.ts:30`). Старый `/id/`-эндпоинт больше не
   использовать.
2. Взять `response.effects[0]`:
   - если `type === "openDialog"` → отрендерить `effect.node` (presentation:"page") тем же
     рендером, что и SDUI-панель движений из формы;
   - если `type === "notify"` → показать нотификацию (нет движений / не найден), не открывать пустую страницу.
3. Строки таблиц брать из `effect.childState` по `TABLE.binding` — **ровно как это уже сделано**
   для панели движений, открытой из формы (панель читает `childState`, см.
   `frontend-spec-movements-1c-block-parity.md §5.4` и `dialog-host.tsx` ветка presentation:"page").
4. Рендер таблиц движений — это уже готовые SDUI-узлы: `TABLE(readOnly) + COLUMN_GROUP(ДЕБЕТ/КРЕДИТ) +
   TABLE_COLUMN`, компоненты в `features/sdui/ui/nodes/composite/*`. Никакого `MovementTable`/
   `AccountingPostingsTable` — рендерит generic node-renderer. Это и даёт «как из формы»
   (двухуровневая шапка, серверное форматирование, ссылочные ячейки).

Практический момент: у списка сейчас нет `SduiSessionProvider`/`DialogHost`
(`sdui-screen.tsx:142` монтирует их только внутри SDUI-экрана). Для этого роута нужен
лёгкий SDUI-контекст рендера дерева движений: достаточно отрендерить `effect.node.children`
через `node-renderer` с источником значений из `childState` (сессия/dispatch не нужны — страница
движений read-only, полей ввода/событий в ней нет). Переиспользуй существующий presentation:"page"
рендер из `dialog-host.tsx:92-121`, вынеся его или вызвав в контексте нового роута.

Legacy-файлы после перевода можно удалять (по согласованию): `document-movements-page.tsx`,
`document-movements-api.ts`, `MovementTable`/`AccountingPostingsTable`, `resolve-page-type`-ветка
`document-movements` — но это отдельный cleanup, не блокер фикса.

### Задача B — заголовок страницы движений (баг #2)

1. **Заголовок**: показывать `node.props.title` (бэк уже включает название документа —
   «Движения документа: {название}»). Не хардкодить «Движения документа» на фронте.
2. **Стрелки назад/вперёд**: страница движений должна иметь тот же chrome навигации, что и обычная
   форма документа (кнопки назад/вперёд в шапке). Переиспользуй компонент шапки/навигации обычной
   SDUI-формы (`PageHeader` / тот же механизм, что рендерит стрелки на форме документа). Бэк это не
   драйвит — это чисто фронтовый chrome рабочей вкладки/страницы.

### Задача C — имя нижней вкладки в истории (баг #3)

Вкладка движений в нижнем `workspace-tab-bar` должна называться так же, как заголовок —
«Движения документа: {название документа}».

- Бэк присылает `PAGE.props.openInWorkspaceTab = true`, `PAGE.props.tabKey = "movements:{id}"`,
  `PAGE.props.title = "Движения документа: {название}"`.
- Имя вкладки брать из `props.title`. `tabKey` использовать как идентификатор вкладки (повторный
  показ движений того же документа переиспользует вкладку, а не плодит дубли).
- Важно: `openInWorkspaceTab`/`tabKey` в текущем фронте ещё **не поддержаны** (`dispatch.ts:46-64`
  их не читает; presentation:"page" рендерится fullScreen-модалкой в `dialog-host.tsx:92-121`).
  Нужно реализовать мост `openDialog(openInWorkspaceTab:true)` → создание рабочей вкладки в
  `features/workspace-tabs` (см. `workspace-tab-sync.tsx`, `use-tab-meta.ts`), с заголовком из
  `props.title` и ключом `tabKey`. См. также `frontend-fix-movements-fullpage-tab.md` (планируемое поведение).

## Приёмка

- **#1**: ДтКт из формы списка на проведённом документе открывает страницу движений с
  **непустыми проводками** (регистр бухгалтерии, вкладка «Журнал проводок…»), визуально идентичную
  той, что открывается из формы (группы ДЕБЕТ/КРЕДИТ, форматирование, ссылочные ячейки).
- **#2**: заголовок страницы движений = «Движения документа: {название документа}»; в шапке есть
  стрелки назад/вперёд как у обычной формы.
- **#3**: нижняя вкладка в истории называется «Движения документа: {название документа}»; повторное
  открытие движений того же документа переиспользует ту же вкладку.
