# SCRUM-263 — Динамическая видимость полей/колонок формы (`formConfig.visibility`)

**Тип:** спецификация для фронтенд-разработчика (fin-web).
**Связанный backend:** ветка `talgat/SCRUM-263` (webbuh), коммит `c52ff357` — контракт `formConfig.visibility`.
**Статус кода:** backend закоммичен; fin-web — **не закоммичен**, реализуется по этой спеке.
**Референс-реализация:** в рабочем дереве Talgat'а есть рабочий (e2e-проверенный) черновик из 5 файлов —
можно свериться, но канон — эта спека (в ней чище/масштабируемее, чем в черновике).

---

## 1. Зачем это (контекст SCRUM-263)

Аналитик по SCRUM-263 (документ «План финансирования организации», QA-bounce) требует, чтобы на форме
документа **видимость части реквизитов и колонок ТЧ зависела от значения шапочного поля «Источник финансирования»**:

| Источник финансирования | Что видно в ТЧ «План финансирования» |
|---|---|
| «Деньги от реализации … ТРУ» (`DengiOtRealizatsii`) | ФКР + Код платных услуг + Специфика |
| Бюджетные средства / софинансирование / местное самоуправление | ФКР + Специфика (Код платных услуг скрыт) |
| Благотворительность / возвратные / Особые расходы / гранты | **только Специфика** (ФКР и Код платных услуг скрыты) |

Плюс реквизиты `СтатьяДДС / СчётКассовыхРасходов / СчётФинансирования / ДвижениеФинансирования` видны только при
`ВидПлана = «По обязательствам»`, а `ТекстОснования` — только при `ВидОперации = «Корректировка»`.

**Ядро проблемы:** backend **уже вычисляет** эту видимость (метод `buildFormConfig`, пересчёт на каждый
`handle-event`) и **уже возвращает** её фронту в ответе `handle-event`. Но fin-web этот кусок ответа
(`response.data.data.formConfig`) **полностью игнорирует** — поэтому динамические правила аналитика не работают.
Эта спека закрывает потребление контракта на фронте. **Это не PFO-специфичная фича** — механизм общий для любого
документа, чей `FormHandler` эмитит `visibility`.

---

## 2. Что уже сделано на backend (не трогать — просто потреблять)

Коммит `c52ff357`. В ответе `POST /api/document-entries/{typeCode}/handle-event`:

```jsonc
{
  "data": {
    "attributes": { /* новые значения полей — УЖЕ обрабатывается фронтом */ },
    "formConfig": {
      "visibility": {                         // ← НОВОЕ, сейчас игнорируется фронтом
        "PlanFinansirovaniyaKodPlatnykhUslug": false,  // колонка ТЧ
        "PlanFinansirovaniyaFKR": true,                // колонка ТЧ
        "PlanFinansirovaniyaStatyaDDS": true,          // колонка ТЧ
        "DvizhenieFinanisrovaniya": true,              // поле шапки
        "TekstOsnovaniya": false                       // UI-элемент шапки
        // ... остальные ключи
      }
    }
  }
}
```

### Контракт ключа (важно — совпадает с существующей конвенцией `fieldFilters`)

- **Поле шапки** → ключ = **код реквизита** (`node.code`), напр. `DvizhenieFinanisrovaniya`.
- **Колонка ТЧ** → ключ = **`<КодТЧ><КодКолонки>`** (конкатенация, БЕЗ разделителя), напр.
  ТЧ `PlanFinansirovaniya` + колонка `KodPlatnykhUslug` → `PlanFinansirovaniyaKodPlatnykhUslug`.

Это **та же** конвенция «пути поля», что уже используется в `FieldFilter` (см.
`src/entities/form-config/types/form-config.ts`, где ключ `fieldFilters` для строки ТЧ = `<КодТЧ><КодКолонки>`,
напр. `OsnovnyeSredstvaMOL`). Backend строит ключи из тех же констант, которыми названы атрибуты, → совпадение
с `${attribute.code}${col.code}` на фронте гарантировано.

### Семантика (обязательно соблюсти)

- ключ **отсутствует** в `visibility` → элемент **видим** (fallback на статический `showInForm`/`showInList`);
- ключ = **`false`** → элемент **скрыт**;
- ключ = **`true`** → видим.

Fallback «нет ключа → видим» критичен: документы, чей `FormHandler` не эмитит `visibility`, ведут себя как
раньше (карта пустая → всё видно). **Никакой регрессии для других форм.**

## 3. Связь с изменениями БД (важно — разделение static vs dynamic)

Аналитик также просил **убрать навсегда** 6 субконто-колонок ТЧ (`СчётКассовыхРасходовСубконто1..3`,
`СчётФинансированияСубконто1..3`) и лишние колонки регистра РН. Это сделано **миграциями БД** (webbuh, коммит
`1fe60483`): `show_in_form=false` / `show_in_list=false` на этих атрибутах.

**Фронт эти статически-скрытые колонки уже НЕ показывает** — `use-table-columns.ts` фильтрует по `a.showInForm`,
списки движений — по `showInList`. **Правок фронта для статического удаления колонок НЕ нужно.**

| Правило видимости | Механизм | Нужна ли правка fin-web |
|---|---|---|
| Убрать 6 субконто-колонок ТЧ навсегда | миграция `show_in_form=false` | нет (уже фильтруется) |
| Убрать лишние колонки движений РН | миграция `show_in_list=false` | нет |
| «Код платных услуг» видим по Источнику (динамически) | `formConfig.visibility` | **да — эта спека** |
| «только Специфика» для 4 источников | `formConfig.visibility` | **да — эта спека** |
| Видимость по ВидПлана/ВидОперации | `formConfig.visibility` | **да — эта спека** |

**Почему разделение:** статическая видимость (одинакова всегда) — это метаданные атрибута в БД; динамическая
(зависит от значений других полей во время редактирования) — не выразима в статических метаданных, поэтому
приходит рантайм-контрактом `formConfig`. Не смешивать: не дублировать статическую видимость в `visibility`.

## 4. Что реализовать (FSD-слайс `features/form-renderer`)

Все правки в одном слайсе `src/features/form-renderer` (+ его типы). Следовать FSD: импорт напрямую из файлов
сегментов (barrel только на уровне слайса), тексты — через `useTranslation` (здесь новых строк нет), никаких
хардкод-строк ключей — вынести конвенцию пути в утилиту (см. ниже).

### 4.1 Утилита пути поля (новое — для чистоты/DRY)

**Зачем:** конвенция ключа (`код поля` для шапки, `<КодТЧ><КодКолонки>` для ТЧ) сейчас продублирована в двух
местах (`fieldFilters` в `field-node`/`table-field`). Чтобы `visibility` не дублировал её в третий-четвёртый раз,
вынести в одну тестируемую функцию — тогда при будущем упрочнении контракта (см. §7) правка в одном месте.

`src/features/form-renderer/lib/utils/field-path.ts`:
```ts
/** Путь поля формы (конвенция FieldFilter/visibility): шапка = код поля, колонка ТЧ = `<КодТЧ><КодКолонки>`. */
export const headerFieldPath = (fieldCode: string): string => fieldCode
export const tableColumnPath = (tableCode: string, columnCode: string): string => `${tableCode}${columnCode}`
```
(Опционально отрефакторить существующие `fieldFilters`-обращения на эту же утилиту — устраняет магические
конкатенации по коду. Не обязательно в этом тикете, но желательно.)

### 4.2 Тип контекста — `visibilityMap`

`src/features/form-renderer/types/renderer-context.ts` — добавить в `FormRendererContextValue`:
```ts
/**
 * Динамическая видимость элементов из formConfig.visibility (ответ handle-event).
 * Ключ — путь поля (см. lib/utils/field-path). false → скрыт; отсутствие ключа → видим (fallback showInForm).
 */
visibilityMap: Record<string, boolean>
```
**Зачем:** видимость нужна и в `field-node` (шапка), и в `table-field` (колонки ТЧ) — обе читают контекст, как уже
читают `fieldFilters`. Контекст — консистентная точка (не проп-дриллинг).

### 4.3 Захват `visibility` из ответа события

`src/features/form-renderer/lib/hooks/use-form-events.ts` — сейчас `onSuccess` берёт только `attributes` и
**игнорирует** `formConfig`. Добавить необязательный колбэк и вызвать его:
```ts
// в параметрах хука:
onVisibility?: (visibility: Record<string, boolean>) => void
// в onSuccess (до/после обработки attributes — независимо):
const fc = response.data.data.formConfig as { visibility?: Record<string, boolean> } | undefined
if (fc?.visibility && onVisibility) onVisibility(fc.visibility)
```
**Зачем:** это единственная точка, где приходит рантайм-`formConfig`. Не трогать существующую обработку
`attributes` (табличные replacer'ы и `form.setValue`) — только добавить ветку видимости.

### 4.4 Стейт видимости в `FormRenderer` + проброс в контекст

`src/features/form-renderer/ui/form-renderer.tsx`:
```ts
const [visibilityMap, setVisibilityMap] = useState<Record<string, boolean>>({})
// передать onVisibility: setVisibilityMap в useFormEvents(...)
// добавить visibilityMap в contextValue + в deps useMemo
```
**Зачем:** `FormRenderer` уже владеет `contextValue` (там же живёт `fieldFilters`, `onFieldChange`). Стейт
видимости логически там же. Обновление реактивно: сменил «Источник» → `handle-event` → `setVisibilityMap` →
ре-рендер полей/колонок.

### 4.5 Применение к полю шапки

`src/features/form-renderer/ui/field-node.tsx` — прочитать `visibilityMap` из контекста и скрыть поле:
```ts
// ВАЖНО: разместить проверку ПОСЛЕ всех хуков (rules-of-hooks), рядом с существующим `if (!attribute) return null`.
if (node.code in visibilityMap && !visibilityMap[node.code]) return null
```
`in` + отрицание (а не `=== false`) — чтобы «ключ отсутствует» ≠ «false», и чтобы пройти eslint
`no-unnecessary-boolean-literal-compare` (тип индексации `Record<string,boolean>` — `boolean`, не `boolean|undefined`).

### 4.6 Применение к колонкам ТЧ

`src/features/form-renderer/ui/table-field.tsx` — прочитать `visibilityMap`, отфильтровать колонки перед
построением `tableColumns` (react-table), добавить `visibilityMap` в deps `useMemo`:
```ts
const visibleColumns = columns.filter((col) => {
  const key = tableColumnPath(attribute.code, col.code)
  return !(key in visibilityMap) || visibilityMap[key]
})
// строить dataCols из visibleColumns
```
**Зачем не в `use-table-columns`:** тот хук делает react-query по типу строки и фильтрует по СТАТИЧНОМУ
`showInForm` (кэшируется). Динамическую видимость накладывать поверх, в компоненте (реактивно к смене шапки),
не смешивая со статическим слоем.

## 5. Начальное состояние (обязательно учесть — иначе баг на существующих документах)

`handle-event` присылает `visibility` **только при изменении поля**. При **открытии уже сохранённого** документа
(где «Источник финансирования» уже заполнен) события ещё не было → `visibilityMap` пуст → колонки, которые
должны быть скрыты, покажутся до первого касания поля.

**Требование:** засеять начальную видимость при загрузке формы. Рекомендуемый чистый вариант — при монтировании
формы, если поле-триггер («Источник финансирования», а также «Вид плана»/«Вид операции») уже имеет значение,
один раз инициировать его форм-событие (тот же путь, что при ручном изменении), чтобы backend вернул актуальный
`formConfig.visibility`. Не хардкодить имена полей ПФО в общий `form-renderer` — брать из атрибутов те, у которых
есть `formEvent` (в `use-form-events` уже строится `eventFieldMap`). Согласовать конкретную стратегию сидинга
(mount-триггер vs отдельный ответ на загрузке документа) с бэкендом, если mount-триггер даёт лишние запросы.

## 6. Критерии приёмки (e2e, как проверял аналитик/бэкенд)

Форма нового/существующего ПФО, ТЧ «План финансирования» непустая:
1. Источник = **«Деньги от реализации … ТРУ»** → в ТЧ видимы **Код платных услуг** и **ФКР**.
2. Источник = **«Особые расходы»** (или благотворительность/возвратные/гранты) → в ТЧ видима **только Специфика**
   (Код платных услуг и ФКР скрыты).
3. Смена Источника **на лету** перестраивает колонки без перезагрузки страницы.
4. Существующий проведённый ПФО при **открытии** сразу показывает корректный набор колонок (см. §5).
5. 6 субконто-колонок ТЧ отсутствуют всегда (это уже даёт миграция — регресс-проверка, что фронт их не вернул).
6. Другие типы документов (без `visibility` в ответе) рендерятся как раньше — регрессии нет.

Тесты: unit на `field-path` и на фильтрацию колонок (Vitest + Testing Library); e2e — Playwright по п.1-4.
Бэкенд для локального e2e: webbuh на порту, `VITE_API_BASE_URL` на него (webbuh без Spring Security, CORS `*`).

## 7. Известное ограничение (не блокер, follow-up)

Ключ ТЧ = `<КодТЧ><КодКолонки>` **без разделителя** — теоретически коллизия (`"AB"+"CD"` = `"ABC"+"D"`).
Это **унаследовано** от существующей конвенции `fieldFilters` (тот же формат), поэтому здесь оставляем как есть
ради консистентности. Упрочнение (разделитель `ТЧ.Колонка` или структурный ключ `{tableCode, columnCode}` +
контракт-тест «каждый ключ `visibility` резолвится в реальный атрибут») — отдельным тикетом на обе стороны
(backend+fin-web), т.к. меняет контракт. Пока — держать конвенцию единой через `lib/utils/field-path.ts` (§4.1),
чтобы будущая правка была в одном месте.
