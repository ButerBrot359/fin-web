# Действия бэкенда по замечаниям QA — ОС/первичные документы

> **Дата:** 2026-06-19
> **Контекст:** Замечания аналитиков по тестированию ОС-документов (Поступление от контрагента, Перемещение ОС, Инвентаризация ВНА). Фронт проанализировал все 5 пунктов — ниже то, что требует действий от бэкенда.

---

## F1 (крит.) — Смена Организации очищает табличные части

### Проблема

При смене поля «Организация» в документе «Поступление от контрагента» очищаются табличные части (ОС/НМА/ТМЗ). Пользователь теряет введённые строки.

### Что проверил фронт

Фронтовый механизм (`use-form-events.ts`) заменяет таблицы **только если** сервер **явно вернул** массив для этого поля в ответе `handle-event`. Если в ответе таблицы не упоминаются — они не трогаются.

### Действие бэкенда

**Проверить ответ `POST /api/document-entries/{typeCode}/handle-event` при eventName = `OnOrganizatsiyaChanged`:**

1. Если ответ содержит пустые массивы для табличных частей (например `"OsnovnyeSredstva": []`) — **убрать их**. Не возвращать ТЧ в ответе вообще, если не нужна их очистка.
2. Если нужно показать warning о пустом префиксе организации — возвращать `clientHint: { action: "showWarning", message: "..." }`. Фронт пока не обрабатывает `clientHint` — **согласовать формат**, фронт добавит обработку (тост/баннер).

### Критерий приёмки

`handle-event` при смене Организации НЕ содержит в `attributes` ключей табличных частей → строки на месте.

## F4 — Автоподсчёт СуммаДокумента

### Проблема

Поле «СуммаДокумента» приходится вводить вручную. При первом сохранении — ошибка «сумма пустая». В 1С сумма считается автоматически: `Σ(ОС.ТекущаяСтоимость) + Σ(НМА.ТекущаяСтоимость) + Σ(ТМЗ.Сумма) + Σ(Услуги.Сумма)`.

### Что проверил фронт

На фронте **нет** клиентского механизма вычислимых полей. Пересчёт возможен только через form-events (server-driven): при изменении строки ТЧ сервер пересчитывает сумму и возвращает новое значение.

### Действие бэкенда

**Два варианта (выбрать один):**

**(A) Form-events (рекомендуется):**
- Зарегистрировать form-event на изменение строк ТЧ (ОС, НМА, ТМЗ, Услуги)
- При каждом `handle-event` возвращать пересчитанное значение `SummaDocumenta` в `attributes`
- Фронт автоматически подхватит (механизм `use-form-events.ts` → `form.setValue`)

**(B) Пересчёт только при сохранении:**
- Бэкенд уже считает сумму при сохранении — убрать валидацию «сумма пустая» для первого сохранения
- Минус: пользователь не видит сумму до сохранения

### Критерий приёмки

Добавил ОС со стоимостью → сумма документа обновилась сама (вариант A) или сохранение прошло без ошибки (вариант B).

## F5 — Фильтрация МОЛ/Физлица по Организации

### Проблема

При выборе МОЛ (физлица) в строке ОС список физлиц выводится без отбора по организации документа.

### Что проверил фронт

Фронтовый механизм `useCellDependency` **уже поддерживает** фильтрацию table-cell пикеров по header-полям через `dependsOn` в метаданных типа. Когда тип физлица будет иметь `dependsOn: [{ sourceAttributeCode: "Organizatsiya", targetAttributeCode: "..." }]`, фронт автоматически передаст `af=targetAttr:orgId` в запрос пикера.

### Действие бэкенда

**Добавить `dependsOn` в метаданные типа физлица (МОЛ):**

```jsonc
// GET /api/universaldomain-types/DICTIONARY/FizicheskieLitsa
{
  "dependsOn": [
    {
      "sourceAttributeCode": "Organizatsiya",   // поле в header документа
      "targetAttributeCode": "Vladelets"         // фильтр на стороне справочника
    }
  ]
}
```

Точные имена атрибутов (`sourceAttributeCode`, `targetAttributeCode`) — согласовать с моделью данных.

### Критерий приёмки

Фронт при открытии пикера МОЛ передаёт `af=Vladelets:<orgId>` → бэк возвращает только физлиц этой организации.

## F3 — Ввод кода с клавиатуры в reference-полях

### Проблема

Аналитик сообщает: «Счёт и аналогичные ссылочные поля выбираются только через выпадающий список; ввести код руками нельзя».

### Что проверил фронт

Фронтовый `DictField` (Autocomplete) **поддерживает** ввод с клавиатуры: при вводе текста срабатывает debounced серверный поиск (`/search?q=...`).

### Действие бэкенда

**Проверить что endpoint `/search` для конкретного типа (Счёт, ACCOUNT_PLAN) ищет по коду:**

1. Убедиться что `GET /api/universaldomain-entries/ACCOUNT_PLAN/{typeCode}/search?q=1210` возвращает счета с кодом содержащим `1210`
2. Если `/search` ищет только по `nameRu` — добавить поиск по `code`
3. Если для типа нет endpoint'а `/search` — добавить

### Критерий приёмки

Набрал код счёта `1210` в поле → сервер вернул подходящие счета → пользователь выбрал без открытия списка.

Потеря данных формы при навигации в дочерний picker — **исправлено на фронте** (коммит `fa86ed4`). Все панели стека теперь остаются mounted, form state сохраняется.
