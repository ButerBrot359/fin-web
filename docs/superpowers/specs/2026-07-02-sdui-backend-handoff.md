# SDUI: спека для бэкенда — контрактные фиксы и целевая архитектура

Дата: 2026-07-02. Автономный документ для бэк-команды: не требует чтения фронтового кода.
Основание: ревизия `docs/superpowers/specs/2026-07-02-sdui-course-audit.md`.

## 0. Принцип

Фронт SDUI — максимально «тупой» рендерер. Он:

- ходит только в `POST /api/view` (+ `DELETE /api/view/{id}`); все прочие URL получает из ответов бэка (effects, `optionsSource.url`, actions);
- не знает имён команд и событий — все имена присылает бэк в actions;
- не принимает решений — «когда кнопка активна», «что сбрасывает dirty», «какой формой открыть» описывает бэк декларативными метаданными;
- допускает только generic-флаги рендера (`editable`, `collapsible`, `orientation`); бизнес-именованные флаги (`showDefaultDictionary` и т.п.) запрещены.

Ниже — часть A: точечные фиксы текущего контракта (можно делать независимо, каждый пункт самодостаточен) и часть B: целевая архитектура (чтобы фиксы части A не пришлось переделывать).

## A. Контрактные фиксы

### A1. Метаданные действий вместо знания имён команд (приоритет: высокий)

**Сейчас.** Фронт держит список `['save','saveAndClose','post','postAndClose']` и по имени команды решает: (а) слить несохранённые строки таблиц в state перед отправкой, (б) сбросить признак dirty после успеха. При закрытии вкладки фронт сам шлёт `command: 'save'`.

**Проблема.** Любая новая «сохраняющая» команда (например `approve`) требует правки фронта. Это бизнес-логика на фронте.

**Требуется.** Каждое действие (кнопка, пункт меню), присылаемое в дереве, несёт декларативные метаданные:

```json
{
  "type": "BUTTON",
  "label": "Провести и закрыть",
  "action": {
    "kind": "COMMAND",
    "name": "postAndClose",
    "flushPendingTables": true,
    "resetsDirty": true,
    "closeAfter": true
  }
}
```

- `flushPendingTables: boolean` — перед отправкой включить в state несохранённые строки редактируемых таблиц;
- `resetsDirty: boolean` — после успешного ответа снять признак «есть изменения»;
- `closeAfter: boolean` — после успеха закрыть вкладку/диалог.

Отсутствие поля = `false`. Имя команды (`name`) для фронта — непрозрачная строка.

Дополнительно: сценарий «закрыть вкладку с несохранёнными изменениями» — бэк присылает в ответе OPEN отдельный дескриптор `onDirtyClose` (какой action выполнить при подтверждении сохранения), фронт не хардкодит `'save'`.

### A2. Имена событий целиком от бэка (приоритет: высокий)

**Сейчас.** Фронт конструирует имена: `addRow:<binding>`, `deleteRow:<binding>:<rowId>`, `ref.select:<targetNodeId>`.

**Требуется.** Кнопки «добавить строку», «удалить строку», «выбрать», «копировать» приходят как ноды с готовым action:

```json
{ "action": { "kind": "EVENT", "name": "addRow:PlanFinansirovaniya" } }
```

Для действий уровня строки (удаление) — шаблон подстановки, объявленный бэком:

```json
{ "action": { "kind": "EVENT", "name": "deleteRow:PlanFinansirovaniya:{rowId}" } }
```

Фронт подставляет `{rowId}` механически, не зная семантики. (Editable-table уже частично работает по EVENT-модели — привести table-node к тому же виду.)

### A3. Активность кнопок — от бэка (приоритет: высокий)

**Сейчас.** В форме выбора из справочника фронт определяет «кнопке нужна выбранная строка» по префиксу имени события (`ref.select:` / `ref.copy:`).

**Требуется.** Декларативное условие в метаданных действия:

```json
{ "action": { "kind": "EVENT", "name": "ref.select", "requiresSelectedRow": true } }
```

Фронт активирует/деактивирует кнопку по флагу. Для остальных случаев бэк управляет `enabled: false` через патчи.

### A4. `optionsSource.url` всегда присылается (приоритет: высокий)

**Сейчас.** Если `optionsSource.url` не пришёл, фронт сам строит URL из карты доменов: `DICTIONARY → /api/dictionary-entries/...`, `DOCUMENT → /api/document-entries/...`, `ACCOUNT_PLAN → /api/account-plan/...`.

**Требуется.** Для каждого ссылочного поля (REFERENCE) бэк обязан присылать `optionsSource.url` (готовый URL поиска/автодополнения). Фронтовая карта доменов удаляется. Новый домен на бэке перестаёт требовать релиза фронта.

### A5. Единый формат пагинации (приоритет: средний)

**Сейчас.** Ответы списков приходят то как `{ content: [...] }`, то как `{ items: [...] }` — фронт пробует оба.

**Требуется.** Один формат страницы для всех списочных ответов SDUI (предлагается `{ items, total, page, size }` — финальный вид на усмотрение бэка, но один).

### A6. `presentation` объекта обязателен (приоритет: средний)

**Сейчас.** Для отображения ссылочного значения фронт перебирает `presentation ?? displayName ?? nameRu ?? name ?? String(id)`.

**Требуется.** Каждый объект-значение (в state, в options, в строках списков) несёт готовую строку `presentation`. Фронт рендерит её и ничего не выводит сам.

### A7. Обязательные пропсы вместо разрешающих дефолтов (приоритет: средний, согласованное изменение)

**Сейчас.** При отсутствии пропа фронт разрешает: `editable → true`, `allowAdd/allowDelete/allowReorder → true`, `domain → 'DICTIONARY'`, `presentation диалога → 'modal'`.

**Решение (2026-07-02).** Дефолт всегда запрещающий: отсутствие пропа = `false` / поле обязательно. Забытый проп проявляется как «кнопки нет» (заметно и безопасно), а не как «пользователь может удалять» (дыра).

**Требуется от бэка.** Явно проставить во всех существующих layout:

- `editable`, `allowAdd`, `allowDelete`, `allowReorder` на таблицах;
- `domain` на REFERENCE-полях;
- `presentation` (`modal` | `page`) на openDialog-effects.

Переключение дефолтов на фронте выполняется синхронно с этим — иначе сломаются текущие экраны.

### A8. Печать и файлы — только через effects (приоритет: средний)

**Сейчас.** Документация фиксирует печатные команды (`print:Forma77`) как известные фронту исключения.

**Требуется.** Печать = обычная COMMAND с непрозрачным именем; в ответе — effect `download` с готовым URL. Фронт не знает про печать ничего. Тексты ошибок формирования файлов — тоже от бэка (сейчас захардкожены на фронте).

### A9. Имена иконок из согласованного реестра (приоритет: низкий)

**Решение (2026-07-02).** SVG-ассеты живут на фронте. Бэк присылает имя иконки строкой (`"icon": "folder"`); фронт рендерит по имени, неизвестное имя → fallback-иконка. Реестр имён ведётся совместно (предлагается стартовый список в отдельном приложении к этой спеке при реализации shell).

## B. Целевая архитектура

Цель части B — зафиксировать направление, чтобы фиксы части A ложились в конечную картину.

### B1. Один роут на фронте

`App.tsx` сводится к одному catch-all маршруту (+ возможный `/auth`). Всё, что рендерится, определяется ответом `POST /api/view` на текущий путь/параметры. Следствия для бэка:

- любые переходы — effect `navigate` с готовым `route`;
- `layoutCode` не конструируется фронтом (сейчас фронт склеивает `<moduleCode>.ФормаОбъекта`) — код layout приходит в navigate-effect или пункте меню;
- «куда после закрытия/сохранения» — тоже route от бэка (сейчас фронт хардкодит переход на `/` и строит путь списка сам).

### B2. Shell как SDUI (включая левое боковое меню)

Весь каркас экрана описывается бэком:

- **Sidebar (левое меню)** — SDUI-нода: состав пунктов, порядок, вложенность, имена иконок (см. A9), action навигации на каждом пункте (route от бэка). Никаких `GET /api/settings/modules` + построения путей на фронте. Меню полностью настраивается бэком — в т.ч. per-role/per-user, если потребуется, без участия фронта.
- **TopBar** — SDUI-нода (сейчас статические заглушки).
- **Вкладки рабочей области** — механика вкладок (открыть/закрыть/переключить, ограничение количества) остаётся generic-фронтовой, но заголовок, иконка и поведение вкладки приходят в ответе view; фронтовая таблица из 14 URL-регексов для определения типа страницы удаляется.

Предлагаемая форма: ответ на OPEN корневого экрана содержит дерево `SHELL { sidebar, topbar, workspace }`; последующие OPEN документов рендерятся в workspace.

### B3. Списки: LIST_FORM

Сейчас списки документов/справочников — легаси-страницы с прямыми API. Целевое состояние: список — такой же SDUI-экран (`LIST_FORM`), открываемый через `POST /api/view` с layout списка; колонки, фильтры, тулбар и действия строк описывает бэк. Отчёты (ОСВ и др.) — тот же механизм.

### B4. Выбор из справочника: CHOICE_FORM

Сейчас выбор значений ссылочных полей идёт через легаси-панель с 7 прямыми эндпоинтами (изолирована за фронтовым gateway). Целевое состояние: кнопки «показать все» / «создать» у REFERENCE-поля — серверные actions, открывающие `CHOICE_FORM` через тот же `/api/view` (openDialog-effect). После этого легаси-панель удаляется; фронтовый gateway остаётся как точка подмены реализации.

### B5. Авторизация

Единственный допустимый второй роут — `/auth`. После логина бэк присылает route первого экрана (не фронтовая карта редиректов). Детали — отдельная спека при постановке задачи.

### B6. Роадмап (предлагаемый порядок)

1. **A1–A4** — устраняют бизнес-логику фронта в текущем контуре (независимые, можно параллельно).
2. **A5–A8** — гигиена контракта; A7 — синхронный релиз фронт+бэк.
3. **B2 sidebar + B1** — shell server-driven, один роут; удаляются URL-регексы.
4. **B3 LIST_FORM** — миграция списков, начинают удаляться легаси-страницы.
5. **B4 CHOICE_FORM** — удаление легаси-панели выбора.
6. **B5 auth** — по готовности.

### Зафиксированное исключение

Master-detail фильтрация связанных таблиц выполняется на фронте (решение 2026-07-02, подтверждает ADR-0013 §4.2): данные обеих таблиц уже находятся в state, фронт лишь показывает срез по выбранной строке мастера. Условие: ключи связи (`masterKey`/`detailKey`) присылает бэк в пропсах ноды. Это презентационная логика, не бизнес-логика.
