## 1. Целевая модель (канон)

Зафиксирована с владельцем продукта 2026-07-02:

1. **Один роут на фронте.** В `src/app/App.tsx` рендерится один catch-all маршрут (плюс, возможно, отдельный роут авторизации в будущем). Никаких `/documents/:id`, `/dictionaries/...` и т.д.
2. **Мало эндпоинтов бэка.** Пара общих основных (`POST /api/view`, `DELETE /api/view/{id}`). Дополнительные, связанные с действиями (скачивание файла, печать), присылаются бэком внутри actions/effects — фронт их не знает заранее.
3. **Весь экран server-driven, включая shell.** Левое боковое меню (sidebar), вкладки рабочей области, topbar — всё настраивается и описывается бэком как SDUI-ноды. Фронт не строит пути, не хранит регексы URL, не решает какие пункты меню показать.
4. **Никакой бизнес-логики на фронте.** Фронт не знает имён команд, не конструирует имена событий, не решает когда кнопка disabled, не фильтрует данные по бизнес-правилам.
5. **Флаги допустимы только для рендера базовых вещей** (табы, кнопки, тулбары): generic-пропсы вида `editable`, `collapsible`, `orientation`. Строго запрещены бизнес-именованные флаги вида `showDefaultDictionary`.
6. **Конфиги описываемые, а не булевы намёки.** Вместо «фронт знает, что команда `save` сбрасывает dirty» — бэк присылает метаданные действия (`resetsDirty: true`).
7. **Легаси не трогаем сейчас**, но фиксируем карту будущего удаления (раздел 6).

## 2. Вердикт

**Ядро движка держит курс.** Транспорт единый (`POST /api/view`), patch-first с formSessionId/revision, обработка 409, маршруты в effects приходят с бэка, пропсы нод в основном generic, zod-валидация на границе. Архитектурный фундамент правильный.

**Отклонения кластеризуются по краям** и делятся на три класса (раздел 3), плюс shell полностью фронтовый (раздел 4), плюс сами канонические документы легализуют часть отклонений (раздел 5).

## 7. Риски будущего («не проебёмся ли»)

1. **resolve-page-type регексы** — любое изменение структуры URL молча ломает вкладки. Устраняется одним catch-all роутом.
2. **Дрейф протокола через имена** — пока фронт знает `save`/`ref.select:`, каждое новое бизнес-действие требует правки фронта. Метаданные действий устраняют класс проблем целиком.
3. **DOMAIN_PATH_MAP** — новый домен на бэке = правка фронта. `optionsSource.url` всегда — устраняет.
4. **Разрешающие дефолты** — забытый проп бэка молча даёт пользователю право редактировать/удалять. Запрещающие дефолты переводят ошибку из «дыра в правах» в «кнопка не появилась» (заметно и безопасно).
5. **Двойная пагинация** — тихое несоответствие форматов даст пустые списки без ошибки.
6. **Легализованные исключения в доках** — без явного роадмапа «этап 2» отклонения становятся постоянными. Handoff-спека фиксирует направление.
7. **Auth** — при появлении auth-роута он должен остаться единственным вторым роутом; логика «куда редиректить после логина» — от бэка (route в ответе), не фронтовая карта.

## 8. Принятые решения (2026-07-02)

1. Master-detail фильтрация — легальное презентационное исключение (ключи связи от бэка).
2. Дефолты пропсов — всегда запрещающие; `domain`, `presentation` обязательны. Согласованное изменение фронт+бэк.
3. Иконки — ассеты на фронте, бэк присылает имя из согласованного реестра, fallback на неизвестное.
4. Shell целиком server-driven, **включая левое боковое меню**: состав, порядок, иконки и навигация пунктов настраиваются бэком.
