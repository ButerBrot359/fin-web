# Карточка ОС и «Поступление от контрагента»: вид операции, группа ОС, вид ОС

Дизайн-спека по трём замечаниям аналитиков ([исходная постановка](../plans/68-os-frontend-handoff.md)).
Все задачи касаются **легаси-контура** (FormRenderer + метаданные), не SDUI:
у `PostuplenieOtKontragenta` `newView: false`, карточка справочника `OsnovnyeSredstva`
рисуется fallback-конфигом из метаданных (form config для неё не существует).

## Результаты ревизии (проверено на dev-стенде 2026-07-03)

| # | Задача аналитиков | Факт |
|---|---|---|
| 1 | «Вид операции» не подставляется | **Уже работает end-to-end.** Диалог → `new?VidOperatsii={code}` → `GET /document-entries/{type}/new?VidOperatsii=...` → поле заполнено. Кода не требуется |
| 2 | «Группа ОС» показывает весь плоский список | Подтверждено: пикер ходит в `GET /universaldomain-entries/DICTIONARY/VidyDolgosrochnykhAktivov/search` (64 позиции). Параметр `groupsOnly` этот эндпоинт **игнорирует** |
| 3 | «Вид ОС» лишнее на форме | Поле рендерится, потому что у атрибута `showInForm: true`. Fallback-конфиг фронта уже фильтрует по `showInForm` |

Ключевая находка: в DTO атрибута уже есть поле `referenceSelectionMode`
(сейчас у `GruppaOS` и `VidOS` — `GROUP_AND_ELEMENT`) — готовый маппинг
1С-настройки «выбор групп и элементов». Новый флаг в контракте не нужен.

## Решение

Настройки полей живут в метаданных бэкенда (вариант «как 1С»), фронт реагирует универсально.
Хардкоды по кодам атрибутов и form config для карточки ОС не заводим.

### Контракт бэк → фронт

1. **Метаданные:** у атрибута `OsnovnyeSredstva.GruppaOS` бэк выставляет
   `referenceSelectionMode: "GROUP"` (выбор только групп).
2. **Данные:** эндпоинты `GET /api/universaldomain-entries/{domain}/{typeCode}/search`
   и `/paged` принимают query-параметр `groupsOnly=true` → возвращают только записи
   с `isGroup = true`.
3. **Скрытие поля:** у атрибута `OsnovnyeSredstva.VidOS` бэк выставляет `showInForm: false`.

Детальная постановка для бэкенда: [backend-handoff-os-card-fields.md](../plans/backend-handoff-os-card-fields.md).

### Изменения фронта

Одно место — `src/features/form-renderer/ui/field-node.tsx`:

- в `useMemo` с `searchParams` домешивать `{ groupsOnly: 'true' }`, если
  `attribute.referenceSelectionMode === 'GROUP'`;
- параметр автоматически попадает во все потребители: дропдаун пикера (`DictField`,
  который шлёт `searchParams` в `/search`) и сайдбар «Показать все»
  (`push({ searchParams })` → `/paged`).

Тип `DocumentAttribute.referenceSelectionMode` уже объявлен как `string`
(`src/entities/document-type/types/document-type.ts:59`) — изменений типов не требуется.

Поведение до готовности бэка: `referenceSelectionMode` у `GruppaOS` остаётся
`GROUP_AND_ELEMENT` → фронт ничего не домешивает → текущее поведение не меняется.
Регресса нет, выкладывать можно независимо.

### Что фронт НЕ делает

- Задача 1 — без изменений кода: проверить на стенде аналитиков после выкладки.
- Задача 3 — без изменений кода: скрытие сработает от `showInForm: false` с бэка.

## Тестирование

1. **До бэка:** открыть карточку ОС → пикер «Группа ОС» работает как раньше
   (полный список), «Вид ОС» виден. Никаких изменений.
2. **После бэка:** пикер «Группа ОС» показывает 13 групп (Земля, Здания, …,
   Прочие долгосрочные активы); «Показать все» — тоже только группы;
   «Вид ОС» отсутствует на форме; выбранная группа сохраняется.
3. **Задача 1:** создать «Поступление от контрагента» через диалог → поле
   «Вид операции» заполнено выбранной операцией.

## Отклонённые альтернативы

- **Form config карточки ОС + расширение `FieldFilter`** — пришлось бы описать все
  ~50 полей формы и завязаться на инфраструктуру form-configs-server.
- **Хардкод по кодам атрибутов на фронте** — не переиспользуется, копит легаси.
- **Переключение пикера на `/universaldomain-directories` (дерево)** — другой формат
  ответа, «Показать все» остался бы нефильтрованным; search-эндпоинт единообразнее.
