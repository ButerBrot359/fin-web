# SCRUM-264 — Замечания по документу «Заявка на регистрацию ГПС» (SDUI V2)

**Дата:** 2026-07-20
**Ветка:** `feature/SCRUM-264-zamechaniya-gps-v2`
**Зона:** SDUI (новая архитектура документа)

## Контекст

Задача содержит список замечаний по документу «Заявка на регистрацию гражданско-правовой
сделки» (`ZayavkaNaRegistratsiyuGPSdelki`, `newView=true`). Большинство пунктов уже
закрыто в предыдущих итерациях. Разбираются 4 оставшихся открытых пункта.

Разведка (чтение кода + проверка рантайма бэка `https://dev-api.qazyna.ai`) показала:
только один пункт — чистый фронт; остальные упираются в бэкенд либо уже им закрыты.

## Разбор пунктов

### ① Мелькающий диалог «Выбор вида операции» — фронт-фикс

**Симптом:** при «Создать» в списке ГПС на мгновение появляется и сам закрывается диалог
выбора вида операции. Для документа без видов операции диалога быть не должно.

**Причина:** `src/widgets/document-list-toolbar/ui/document-list-toolbar.tsx`, `handleCreate` —
диалог открывается оптимистично (`setDialogOpen(true)`) до ответа `on-get-form`. Для ГПС
бэк возвращает пустой список видов → срабатывает ветка `else`, диалог закрывается → «мелькание».

**Решение (по спеке бэкендера, 1 файл):**
- убрать оптимистичный `setDialogOpen(true)` из начала `handleCreate`;
- открывать `setDialogOpen(true)` только внутри ветки «виды есть», после `setOperations(...)`;
- убрать два ненужных `setDialogOpen(false)` из веток `else` и `catch`.

Прочие функции (`handleSelectOperation`, `handleCloseDialog`) и разметку не трогаем.

**Критерии приёмки:**
1. «Создать» на ГПС → диалог не появляется даже мельком, сразу форма `/new`.
2. «Создать» на документе с видами операции (ПКО/РКО) → диалог показывается как прежде.
3. Ошибка `on-get-form` → диалог не появляется, переход на `/new`.

### ② Нет статуса исполнения заявки — спека бэку

SDUI-форма полностью server-driven: фронт рендерит только дерево нод из `POST /api/view`
по `layoutCode="ZayavkaNaRegistratsiyuGPSdelki.ФормаОбъекта"`. В текущем дереве ноды
статуса исполнения нет, в контракте `ViewResponse`/`ViewNode` поля статуса нет.

Примитив для отрисовки на фронте готов (`src/features/sdui/ui/nodes/display/badge-node.tsx`,
рендерит MUI `Chip` с цветом success/warning/error/info). Фронту делать нечего, пока бэк
не положит ноду статуса в дескриптор формы.

**Действие:** запрос бэку — включить в дескриптор формы ГПС ноду `BADGE`/`LABEL` со
статусом исполнения. Уточнить у заказчика семантику «статуса исполнения».

### ③ Проводки отдельной вкладкой на весь экран — уже закрыто бэком, только QA

Требование: окно проводок (ДтКт) — отдельная полноэкранная вкладка с заголовком (включая
название документа), кнопками назад/закрытие; для обоих путей — из формы документа и из списка.

**Проверка рантайма (оба пути), документ 27856662:**
- Из списка — `GET /api/view/movements/27856662`;
- Из формы — `COMMAND showDtKt` в сессии формы.

Оба возвращают идентичный эффект `openDialog` с:
```
props.openInWorkspaceTab = true
props.tabKey             = "movements:27856662"
props.title              = "Движения документа: Заявка на регистрацию ... AAY00-00005 ..."
props.presentation       = "page"
```

Фронт-инфраструктура это уже отрисовывает как полноэкранную workspace-вкладку:
`src/features/sdui/lib/open-dialog-panel.ts` → `openPanelTab` → `src/app/layout/layout.tsx`
(активная вкладка `sdui-panel` на весь main, обёрнута в `PageHeader` с title, «назад»,
крестиком). Оба пути сходятся в `openDialogAsPanel` (list — `open-movements.ts`, форма —
`dispatch.ts` через `effect-handler`).

**Действие:** кода не требуется. QA-подтвердить визуально оба пути в приложении, отметить
пункт закрытым.

### ④ Фильтр «договоры контрагентов» по Организации и Контрагенту — спека бэку

Требование: список договоров контрагента в поле документа должен быть отфильтрован по уже
заполненным Организации и Контрагенту документа.

Механизм на фронте есть: `src/features/sdui/ui/nodes/fields/reference-field-node.tsx`
читает фильтр из `node.props.filter` / `optionsSource.params` и перезапрашивает список при
смене фильтра. Но значения фильтра должен присылать бэк — по спеке через
`setOptions(field.dogovorKontragenta, {filter:{Organizatsiya, Vladelets}})` в ответ на
EVENT смены Организации/Контрагента (см. `docs/superpowers/plans/events-and-commands-quickstart.md`).

**Known-issue фронта (follow-up, не в этой ветке):** `src/features/sdui/lib/patch-applier.ts`
(~стр. 79) кладёт `patch.options` в `props.options`, тогда как поле читает фильтр из
`props.filter`/`optionsSource.params`. Даже если бэк пришлёт `setOptions({filter})`, до
запроса он не дойдёт. Маппинг чинится на фронте, когда бэк начнёт слать фильтр.

**Действие:** запрос бэку — слать `setOptions` с `filter:{Organizatsiya, Vladelets}` при
смене Организации/Контрагента. Фронт-маппинг `setOptions→props.filter` — отдельный тикет.
